# W4.2 web-rendering capability 设计

> WebView 429 行只是冰山一角。整个 `src/views/web/` 域当前持有 ~1500 行代码,
> 其中 70% 是 driver-级业务编排(SyncDriver / TranslateDriver / slotBus / sync-inject)
> 错置在 view 层。本文档设计 capability 化路线,实施完成后 WebView 预计 429 → ~80 行,
> view 域只剩 React 渲染 + 命令注册 + 状态订阅。

### 1.2 charter 对齐评估

charter § 1.3 规则 C(颗粒度):
> 即便当前只一个视图消费,只要符合"有状态封装"特征,也建立独立 Capability,
> 为未来跨视图复用预留接口。

`SyncDriver` / `TranslateDriver` / `slotBus` 全部符合"有状态 / 有生命周期"特征——所以**必须独立成 capability,不论是否有未来其他消费者**。这是规则 C 的字面要求,Wave 1~3 已经把"换底层零成本"承诺立住,W4.2 是把这个承诺真正应用到 web 域。

charter § 1.3 规则 A:
> `electron WebContentsView` | capability.web-rendering(主进程封装)
> `electron session.webRequest / webContents` | capability.browser-capability

charter 已经预留了 `web-rendering` 名字。本设计跟 charter 列表对齐。

### 1.3 driver vs capability 的二选一

V2 当前架构(eslint.config.js:46 注释)把 driver 单独拎出 capability 概念,作为"必经的业务驱动层"。`text-editing-driver` 是先例:它封装 PM(npm 包),被 view 通过 `<textEditingDriver.Host />` 使用,自身又有 `setLinkClickHandler` 等 view 注入接口。

web 域的 SyncDriver / TranslateDriver 已经命名为 driver——它们应当跟 text-editing-driver 同模式,做成:
- `src/drivers/web-sync-driver/` — 双 webview 事件同步引擎(SyncDriver + sync-inject)
- `src/drivers/web-translate-driver/` — Google Translate 注入引擎(TranslateDriver + google-translate-inject)

而**整个 webview tag 的生命周期编排 / partition 守门 / dom-ready 时序**——这部分应该是 `capability.web-rendering` 的内核(Host 组件 + 状态 + commands)。

**所以:1 个 capability(web-rendering) + 2 个 driver(web-sync-driver / web-translate-driver)**。

这跟 text-editing 域的现状不一致(text-editing 是 1 driver 没 capability),但 text-editing 是历史遗留——V2 走到现在,web 域是第一个真按 charter § 1.3 规则 C 严格归位的领域,以后 text-editing 重整也参考这个三层结构。

## 2. 设计目标

按 charter § 1.2 + § 1.3 + § 1.4:

1. **WebView / TranslateWebView 作为纯 view 声明**,只做组合 + 状态订阅 + 命令注册,LOC ≤ 100
2. **web-rendering capability 作为 web 域唯一对外面孔**,view 通过 `install: ['web-rendering']` 声明依赖
3. **SyncDriver / TranslateDriver 作为 driver**,web-rendering capability 内部编排它们,view 看不到
4. **`<webview>` tag 的 DOM 操作** 全部封装在 capability 内的 `<Host />` 组件
5. **slot-bus 作为内存 pub/sub 原语下沉到 shared/event-bus**(类似 Wave 3.3 ChannelHub),或归 capability 内部
6. **不破坏现有功能**:webview 浏览、双栏翻译、ContextMenu、URL bar 全部保持

非目标:
- ❌ 重写翻译协议 / 同步协议 — 字节级保持原样
- ❌ 优化 SyncDriver 的 80ms poll(留 follow-up)
- ❌ 改 V2 web view 数据模型(per-ws state schema 不动)
- ❌ **view 通过 capabilityRegistry.get(id).api 间接拿 capability 实现**——
  charter line 88 的字面要求,但 V2 现状所有 capability(selection / clipboard /
  undo-redo / drag-and-drop / insertion / media-storage)都走"模块级 export +
  view 直 import"路径,W4.2 跟随现状(WebView 直 `import { webRendering } from
  '@capabilities/web-rendering'`)。**承担 follow-up:** 等所有 view+capability
  统一整理时再切到 registry.get 间接路径(charter v0.5+ 范围)。
- ❌ **install 列表里出现 driver ID**——见 § 4.2,W4.2 起 install 严格只列
  capability ID,driver 是 capability 内部实现,view 不可见(audit review P1-A 修订)

### 3.2 责任边界

#### capabilities/web-rendering/slot-bus.ts(从 views 迁,但留 capability 内)
- 跨 slot 模块级 event bus(left/right 双向通信)
- 仅为 web 双栏翻译用,**不暴露给 capability 外部**(不在 capability index.ts re-export)
- 内部 `<Host />` / `<TranslateHost />` 通过相对路径 `./slot-bus` import
- driver(web-sync-driver)需要 emit/subscribe 时**通过 capability 注入回调**,不直接 import slot-bus
  → 见 § 4.1 Host props 的 onSyncMessage / sendSyncMessage 注入设计(实施时细化)

#### drivers/web-sync-driver/
- SyncDriver 类(双 webview 事件同步引擎)
- sync-inject.js(注入到 webview 内的脚本)
- sync-protocol.ts(action 常量)
- **接口**:`createSyncDriver(side, options) → { bind(wv), start(), destroy(), takeControl(), yield(), reinject() }`
- driver 协议铁律 5:不 import 其他 driver(本身不依赖)

#### drivers/web-translate-driver/
- TranslateDriver 类(Google Translate 注入引擎)
- google-translate-inject.js
- **接口**:`createTranslateDriver(options) → { bind(wv), inject(targetLang), destroy() }`
- 不依赖 web-sync-driver(两个 driver 各自独立)

#### capabilities/web-rendering/
**职责**:封装整个 `<webview>` tag 的生命周期 + 跟两个 driver 的协调编排

**对外 API**:
```ts
export const webRendering = {
  /** 主 Host(普通 webview)*/
  Host: ({ workspaceId, ...props }: HostProps) => ReactElement,
  /** 翻译模式 Host(disablewebsecurity + 翻译注入)*/
  TranslateHost: ({ workspaceId, ...props }: TranslateHostProps) => ReactElement,
};
```

`<Host />` 内部封装:
- webview ref 绑定 + dom-ready 守门(WebView.tsx 107-211 这一大块)
- did-navigate / did-start-loading / did-stop-loading 事件转发
- context-menu 转发(view 内仍接 callback,具体显示由 view 注册)
- SyncDriver lifecycle(WebView.tsx 213-289 这块)
- 跨平台 partition 配置(WEBVIEW_PARTITION / WEBVIEW_TRANSLATE_PARTITION)

**view 通过 props 注入:**
- `currentUrl: string`(per-ws state)
- `partition: string`(默认走 capability 自带,可 override)
- `translateMode: boolean`(决定是否启动 SyncDriver)
- `onContextMenu(payload)`(view 自己的 context menu 注册)
- `onUrlChanged(url)`(setWebUrl 写入 per-ws state)

#### views/web/WebView.tsx(目标 ≤ 100 行)
```tsx
export function WebView({ workspaceId }: WebViewProps) {
  // 1. 订阅 per-ws state(currentUrl, targetLang)
  // 2. 订阅 slotBinding(判断 isTranslateMode)

```

view 不再:
- 直接 import SyncDriver / slot-bus / WEBVIEW_PARTITION
- 写 webview ref handler / dom-ready / did-navigate 编排
- 处理 SyncDriver lifecycle

view 仍然:
- React 渲染 / hooks 订阅
- Toolbar 操作处理(命令路由)
- 命令注册 / 右键菜单注册

## 4. 接口契约细则

### 4.1 web-rendering capability 接口

```ts
// src/capabilities/web-rendering/index.ts

import { capabilityRegistry } from '@slot/capability-registry/capability-registry';
import { Host } from './Host';
import { TranslateHost } from './translate-host';

export interface WebRenderingHostProps {
  /** 当前 ws id(用于 SyncDriver side / partition 等)*/
  workspaceId: string;
  /** 当前显示 URL(view 从 per-ws state 取,传入)*/
  currentUrl: string;
  /** 是否处于双栏翻译模式(决定 SyncDriver 是否启动)*/
  translateMode: boolean;
  /** webview tag partition(默认走 capability 配,可 override)*/
  partition?: string;
  /** 右键菜单事件 — view 自己处理 contextMenuController */
  onContextMenu?: (payload: WebContextMenuPayload) => void;
  /** URL 变化(view 写入 per-ws state)*/
  onUrlChanged?: (url: string) => void;
  /** loading 变化(view 决定是否显示 spinner)*/
  onLoadingChanged?: (loading: boolean) => void;
  /** 导航能力变化(canGoBack / canGoForward)*/
  onNavStateChanged?: (state: { canGoBack: boolean; canGoForward: boolean }) => void;
}

export interface WebRenderingApi {
  /** 主区域 Host 组件(普通 webview)*/
  Host: React.ComponentType<WebRenderingHostProps>;
  /** 翻译模式 Host 组件(disablewebsecurity)*/
  TranslateHost: React.ComponentType<TranslateHostProps>;
  /** 命令式 API:让 view 通过 ref 调 webview 操作 */
  loadURL: (workspaceId: string, url: string) => void;
  goBack: (workspaceId: string) => void;
  goForward: (workspaceId: string) => void;
  reload: (workspaceId: string) => void;
  stop: (workspaceId: string) => void;
}

export const webRendering: WebRenderingApi = { Host, TranslateHost, loadURL, goBack, goForward, reload, stop };

// Wave 1 模式
capabilityRegistry.register({ id: 'web-rendering' });
```

`loadURL` / `goBack` 等命令式 API 走"按 workspaceId 查 webview 实例"路径:capability 内维护 `Map<workspaceId, WebviewElement>`,view 通过 webRendering.loadURL(wsId, url) 调用,不需要传 ref。这个模式参考 text-editing-driver `instanceRegistry`。

### 4.2 web-rendering install 关系(audit review P1-A 修订)

WebView 在 [src/views/web/index.ts](../../../../src/views/web/index.ts) 注册时,**install 严格只列 capability ID**:

```ts
registerView({
  id: 'web-view',
  install: ['web-rendering'],   // ← 只 1 个 capability,driver 不可见
  // ...
});
```

#### 为什么 install 不列 driver ID

audit review P1-A 指出:把 driver ID 写进 install 列表会"弱化能力注册语义",让 view 组装实现细节,违反 charter § 1.2 注册原则的精神。

修订后:
- web-sync-driver / web-translate-driver 是 **web-rendering capability 的内部实现**,view 不知道也不需要知道它们存在
- capability 的 `index.ts` 内部 `import { SyncDriver } from '@drivers/web-sync-driver'`(capability → driver 是 charter § 1.3 表格里 capability 封装 driver 的合规向下调用,driver 协议铁律 5 只禁 driver 之间互拉)
- view 只声明依赖 `web-rendering` 这一个 capability id

#### KNOWN_DRIVER_IDS 后续

[known-driver-ids.ts](../../../../src/slot/view-type-registry/known-driver-ids.ts) 当前 1 条 `text-editing-driver`(NoteView install 仍引用)。

W4.2 实施 **不为 web 加新 driver ID** 到白名单。`text-editing-driver` 这条留作 charter v0.5+ 整理范围(届时 text-editing 也按 web-rendering 模式改成 capability + 多 driver,install 改为 `['text-editing']`,白名单整体淘汰)。

**度量目标**:W4.2 完工后 KNOWN_DRIVER_IDS 仍 1 条,install-coverage 表 web-view 行 drivers 列为 `—`,capability 列含 `web-rendering`。

## 7. 风险与决策点(请 user review)

### 风险 1(✅ resolved 2026-05-08):slot-bus 留 capability 内部

audit review P2-B 反馈:left/right 是 web 双栏业务语义,直接进 shared/ 会让共享层携带业务语义,违反 shared 中性原则。

**已采纳**:slot-bus 留 `capabilities/web-rendering/slot-bus.ts`,**不进 shared**。

附带规则:
- slot-bus 不在 capability index.ts re-export(对外不可见),只供 capability 内 `<Host />` / `<TranslateHost />` 用
- driver 层(SyncDriver)需要 emit/subscribe 时通过**接口注入**(构造函数参数),不直接 import slot-bus(driver 协议铁律:driver 不依赖具体 view/capability 模块)
- 未来若有第二个 view 需要类似 slot 双向通信,**仍不进 shared**——抽到中性命名的 capability(slot-bus 是有状态实例,违反 shared "无状态原语"原则)

### 风险 2(✅ resolved 2026-05-08):capability + 2 driver 三层架构

**已采纳选项 A**:`web-rendering` capability + `web-sync-driver` + `web-translate-driver` 三层。

audit review 中你确认这条"明显更符合分层原则"(P1-A 反馈),所以维持本方案。

跟 text-editing 域(当前 1 driver 含 Host)不一致,但 text-editing 是历史遗留——**charter v0.5+ 整理时按本方案重构**(text-editing 拆 1 capability + ≥1 driver,install 改 `['text-editing']`,KNOWN_DRIVER_IDS 整体淘汰)。

附带:
- `<Host />` / `<TranslateHost />` 在 capability 层(持 React 状态 + lifecycle 编排)
- SyncDriver / TranslateDriver 在 driver 层(纯逻辑无 React)
- capability → driver 是合规向下调用(charter § 1.3 表格语义,Wave 2 lint 现有规则覆盖)
- driver 之间 0 代码 import(铁律 5,SyncDriver / TranslateDriver 互不依赖)

### 风险 3:C5 是巨型 commit 还是再拆

C5 把 WebView / TranslateWebView 大改 + 删旧文件,改动跨 view + driver + capability。可以再拆:
- C5a:WebView 改用 webRendering.Host(老旧 sync 兜底文件还在)
- C5b:TranslateWebView 改用 webRendering.TranslateHost(老旧 translate 兜底文件还在)
- C5c:删除所有 re-export 兜底 + 加 install 字段

这样每步独立 typecheck,但 C5a/b 之间互独立,可并行验证。**3 个 commit 还是 1 个?**

### 风险 4:实施时间预估

W4.2 体量按 commit 估:
- C1 slot-bus:30 min
- C2 sync-driver:1 h(381 行 inject.js 的路径 + ?raw 注入需仔细)
- C3 translate-driver:30 min
- C4 web-rendering 新建:**3-4 h**(从 view 复制 + 改写 props/callbacks + Host.tsx 跟 TranslateHost.tsx 共享逻辑提取)
- C5 view 迁移:**2 h** + 完整功能回归(本身 1 h 测试)

**总计 7-9 小时连续工作**,不能一坐到底,建议分 2-3 个 session,每 session 完成 1-2 个 commit + 你一轮验证。
