# W4.1 Keymap Registrar 设计

> NoteView 165 行的根因不是"代码多",是 80~134 行 `window.addEventListener('keydown')`
> 内嵌了 3 条命令路由的硬编码业务编排,违反 charter § 1.4 "view 0 UI 实现代码"。
> 本文档设计一层 L4 keymap registrar,让 view 通过声明式 keymap 字段注册快捷键,
> 实施完成后 NoteView 预计 165 → ~95 行。

## 2. 设计目标

按 charter § 1.2 注册原则 + § 1.4 view 归属:

1. **view 通过声明式字段注册快捷键**,不写任何 `window.addEventListener`
2. **快捷键触发统一走 commandRegistry.execute**,不直接 import handler
3. **Cmd+K 这种"前置条件 + DOM 触发"的复杂逻辑**也能表达(否则 view 还是要写代码)
4. **不影响 driver 内的局部 keydown**(不接管 DOM 节点级别的快捷键,只管全局)
5. **新增基础设施 ≤ 100 行代码**(只服务一个客户,过设计比缺设计更糟)

非目标:
- ❌ 跨平台 key 别名表(Cmd vs Ctrl 等)— 暂只接 `mod` 通配
- ❌ keymap 冲突检测 / 优先级 — 只有一个 view 用,先不管
- ❌ IME composition 守门 — view 全局 keymap 触发条件是 `Cmd+<key>`,IME 不会冲突

## 3. 接口设计

### 3.1 ViewDefinition 加 `keymap?` 字段

[src/slot/view-type-registry/view-definition.ts](../../../../src/slot/view-type-registry/view-definition.ts) 加:

```ts
/** 快捷键绑定(可选;view 全局快捷键,只在该 view 是当前活跃 view 时生效)*/
export interface KeymapBinding {
  /**
   * key 表达式 — 简化版:
   * - 'mod+k'      → Cmd+K (mac) / Ctrl+K (win/linux)
   * - 'mod+['      → Cmd+[ / Ctrl+[
   * - 'mod+shift+k' → Cmd+Shift+K
   *
   * 修饰键固定顺序:mod, shift, alt(其他不支持)
   * key 部分 case-insensitive
   */
  key: string;
  /** commandRegistry 命令 ID(string 引用,charter § 1.2)*/
  command: string;
  /**
   * 触发前置条件(可选,数组语义 AND)— 全部满足才触发,否则放行让事件继续冒泡
   *
   * - 'has-text-selection':window.getSelection 非空且非 collapsed
   * - 'in-view-area':事件 target 在 [data-view-id="<viewId>"] 内
   * - 'not-in-input':事件 target 不是 <input> / <textarea> / [contenteditable]
   *
   * 复杂条件加新枚举值(扩展 listener),**不允许 command handler 内再做声明式条件检查**——
   * 否则注册原则退化为"注册 + 命令里写硬编码"。
   *
   * 示例(NoteView Cmd+K):
   *   enabledWhen: ['has-text-selection', 'in-view-area']
   */
  enabledWhen?: Array<'has-text-selection' | 'in-view-area' | 'not-in-input'>;
}
```

### 3.2 ViewDefinition.keymap

```ts
export interface ViewDefinition {
  // ...既有字段
  /** view 全局快捷键(可选)*/
  keymap?: KeymapBinding[];
}
```

### 3.3 NoteView 注册示例(目标形态)

```ts
// src/views/note/index.ts
registerView({
  id: 'note-view',
  install: [...],
  component: NoteView,
  navSideTab: { label: 'Note', icon: '📝', order: 1 },
  keymap: [
    { key: 'mod+k', command: 'note-view.popup-link',
      enabledWhen: ['has-text-selection', 'in-view-area'] },
    { key: 'mod+[', command: 'note-view.go-back',
      enabledWhen: ['in-view-area', 'not-in-input'] },
    { key: 'mod+]', command: 'note-view.go-forward',
      enabledWhen: ['in-view-area', 'not-in-input'] },
  ],
});
```

3 条 keymap 都指 `note-view.<action>` 命名空间命令(对齐 [note-commands.ts](../../../../src/views/note/note-commands.ts) 既有命名)。

**注意**:Cmd+[/] 显式带 `not-in-input` 条件,因为 driver 内 file-block / image / etc. 的 url input 也用 Cmd+[/](浏览器默认行为或局部绑定)——keymap registrar 不能抢这些事件。

## 4. 实施层架构

### 4.1 新建 `src/slot/keymap-registry/`

```
src/slot/keymap-registry/
├── README.md                    (~20 行)
├── keymap-types.ts              (KeymapBinding 类型,~25 行)
├── keymap-registry.ts           (Map<viewId, KeymapBinding[]> + register/unregister/getActive,~40 行)
└── keymap-listener.ts           (全局 keydown 监听 + 路由,~40 行)
```

**总计 ~125 行新基础设施**,略超目标 100 行,但每文件职责单一,可接受。

### 4.2 keymap-registry.ts 接口

```ts
class KeymapRegistry {
  private bindings: Map<string /* viewId */, KeymapBinding[]> = new Map();

  /** 由 view-type-registry.distributeToRegistries 调用 */
  register(viewId: string, bindings: KeymapBinding[]): void;
  unregisterByView(viewId: string): void;

  /**
   * 给定按下的 key 表达式 + 当前激活的 view id,查找匹配的 binding。
   * 返回 null 表示无匹配。
   */
  match(viewId: string, keyExpr: string): KeymapBinding | null;
}
export const keymapRegistry = new KeymapRegistry();
```

### 4.3 keymap-listener.ts 接口

```ts
/**
 * 启动全局 keymap 监听(renderer 启动时调一次,见 platform/renderer/index.tsx)
 *
 * 路径:
 * 1. 全局 keydown → 解析为 'mod+k' 这种 key 表达式
 * 2. 查当前活跃 view(workspaceManager.getActiveId().slotBinding 拿 view id)
 *    — 同时存在 left 和 right 两个 view,优先看事件 target 所在 view(data-view-id)
 * 3. keymapRegistry.match(viewId, keyExpr) 拿 binding
 * 4. enabledWhen 数组逐项校验(AND 语义),全部通过才触发——
 *    listener **不内置任何 baseline 条件**(如不再隐式忽略 input 元素),
 *    所有过滤都由 view 在 keymap 字段显式声明,保持注册原则纯净
 * 5. 通过 → e.preventDefault() + commandRegistry.execute(binding.command)
 */
export function startKeymapListener(): () => void;
```

### 4.4 view-type-registry 集成

`distributeToRegistries` 加一行:
```ts
if (def.keymap) {
  keymapRegistry.register(def.id, def.keymap);
}
```

`unregisterRegistries` 加:
```ts
keymapRegistry.unregisterByView(id);
```

### 4.5 启动时序

[src/platform/renderer/index.tsx](../../../../src/platform/renderer/index.tsx) 启动末尾追加:
```ts
import { startKeymapListener } from '@slot/keymap-registry/keymap-listener';
// ... 既有 reportL5Alive 等
startKeymapListener();  // 全局 keydown 路由
```

### 4.6 "活跃 view"判定细节

key 落在 left 还是 right view?设计:
- 用事件 `target` 节点向上找最近的 `[data-view-id="..."]`
- 找不到 → 退化为 workspaceManager 当前活跃 ws 的 `slotBinding.left`(主 view)
- 这要求 view 主容器 div 加 `data-view-id={viewId}` 属性

NoteView 的 `<div className="krig-note-view" data-view-id="note-view">` —— 一行属性,无副作用。

## 7. 风险与决策点(请 user review)

### 风险 1:活跃 view 判定退化路径会不会出错

`data-view-id` 找不到时退化为 `workspaceManager.slotBinding.left`,这意味着如果用户 focus 在 right slot(如 web view)按 Cmd+K,会被误判为左栏 NoteView 的快捷键。

**缓解**:`enabledWhen: 'has-text-selection'` 给 NoteView 的 Cmd+K 兜底——如果是 web view 内选文字,window.getSelection 在 webview tag 外不可见,Cmd+K 会被 enabledWhen 拒。Cmd+[/] 没这个保护,但目前只有 NoteView 注册了这两条,无冲突。**未来如果其他 view 也注册 mod+[/]**,这个判定要重新考虑。

### 风险 2:keymap 跟 driver 内部 keydown 抢事件

driver 内部 `urlInput.addEventListener('keydown')` 用的是 capture: false 默认,事件先在 input 上跑,再冒泡到 window。如果用户在 file-block 的 url input 里按 Cmd+K,会先被 input 的 handler 处理(driver 内部行为),冒泡到 window 时如果 input 没 stopPropagation,keymap registrar 还会收到。

**缓解(已在 § 3.1 接口里固化)**:加 `enabledWhen` 枚举值 `'not-in-input'`,view 显式声明哪些 keymap 需要规避 input 元素。listener 不内置任何 baseline 行为——所有过滤声明式表达,注册原则纯净。

**代价**:view 写每条 keymap 都要思考是否需要 `not-in-input`(NoteView 的 Cmd+[/] 已显式带,见 § 3.3 示例)。少一条声明就是少一层防护,这是声明式的固有 trade-off。

### 决策点 1(resolved 待你确认):keymap 字段需要 ID 吗?

我设计的 KeymapBinding 没有 `id` 字段。其他 Registry(contextMenu / toolbar 等)都有 id,用于 `unregisterByView` 等管理。

但 keymap 不需要单独 unregister(view unregister 时整批删),所以省了 id。**OK 吗?**

如果你要对齐既有 Registry 模式加 id,我加(对实现影响很小)。

### 决策点 2(✅ resolved 2026-05-08):enabledWhen 改数组(AND)+ 加 not-in-input

review 反馈 P2-A:单值枚举会让组合条件被推到 command handler,违反"声明式注册"原则。

**已采纳**:`enabledWhen: Array<'has-text-selection' | 'in-view-area' | 'not-in-input'>`,数组语义 AND。

变化对照:
| 之前 | 现在 |
|---|---|
| `enabledWhen?: 'has-text-selection' \| 'in-view-area'` | `enabledWhen?: Array<...>`,3 枚举值 |
| Cmd+K 的"选区在 NoteView 内"由 handler 内部判断 | `enabledWhen: ['has-text-selection', 'in-view-area']` 直接表达 |
| listener 内置"input 元素静默放行"baseline | `not-in-input` 枚举值显式声明,view 自负责 |

NoteView 3 条 keymap 全部用上 `enabledWhen` 数组(见 § 3.3 更新后的示例)。

### 决策点 3:keymap-registry 是否走 lint 白名单

Wave 2 的 capability 块禁了 `@slot/<视图层 9 子目录>`。新增的 `keymap-registry/` 算视图层基础设施,**capability 不应反向 import**(因为 keymap 是 view 概念,capability 跨 view 复用不依赖具体 view 的 keymap)——所以**不需要白名单,Wave 2 现有规则自然覆盖**。

view 层(views/**)用 keymap-registry 不会触发 lint(views 块没禁 @slot/*)。

✅ 不动 lint config。
