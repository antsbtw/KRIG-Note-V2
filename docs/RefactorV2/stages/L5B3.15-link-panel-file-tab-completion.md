## 1. 实际改动 vs 设计 — 偏差点

### 1.1 link-click plugin 顺手修 file:// URL 解码 bug

设计文档没明示 — 实施时发现 build-link-click-plugin 原 file:// 处理用
`href.replace(/^file:\/\//, '')` 直接传 `openPath`,但 FileTab 输出的 href 是
`encodeURIComponent` 过的(`%20` 等)。openPath 需要原始 OS 路径,会导致**含空格 /
中文文件名打不开**。

改成 `decodeURIComponent(new URL(href).pathname)` + try/catch fallback 兼容老 file://
href(无 encoding)。**这是 pre-existing bug,B3.15 顺手修**。

### 1.2 FileTab 加了路径预览

设计中 placeholder 阶段无路径预览,实施时加了 `.krig-link-panel__file-path` 显路径(等同于
"看到要链接的文件路径"),帮用户判断 link 模式是否可用(Blob 来源时显"无路径")。

### 1.3 没做 file Tab 的内部 keyboard 路由

设计 § 2.1 假设 FileTab 容器 onKeyDown 接收 Enter/Esc。实施加了 `tabIndex={-1}` +
useEffect focus 容器,实际是否生效靠用户测(有可能 PopupBinding 全局 Esc 已经接管)。

## 3. 用户验证清单

`npm run dev`,Note view,以下逐项测:

### LinkPanel 三 Tab(judge #2 - #4)
1. 打开任一现有笔记 — 不白屏
2. 选中一段文字 → Cmd+K — 弹 LinkPanel,**看到三个 Tab**:📄 笔记 / 📎 文件 / 🔗 网页
3. 点击"📎 文件" Tab — 显示"📂 选择文件..." 按钮 + 提示文字

### import 模式(#5 - #6)
4. 文件 Tab 点 "📂 选择文件..." → 选一个 PDF
5. 切到双模式 UI:**默认选中"导入到媒体库"** + 显示文件名 + 路径 + [重选] [确认]
6. 直接点 [确认] → 面板关闭,选中文字应该带下划线(link mark)
7. F12 看 DOM 的 `<a class="link" href="...">` — href 是 `media://...`
8. 点击该 link 文字 → **系统 PDF 阅读器打开该文件**(media:// 路由生效)

### link 模式(#7 - #8 + URL 解码 fix)
9. 再选一段文字 → Cmd+K → 文件 Tab → 选 PDF → **改选"链接到原文件"** → 确认
10. F12 看 link mark href — 是 `file:///<encoded-path>`(有空格的话是 `%20`)
11. 点击该 link → 系统默认 app 打开
12. **关键测试**:选一个**含空格或中文**文件名的 PDF(如 `测试 文档.pdf`)→ link 模式
    → 点击 → 应该能打开(B3.15 修了 URL 解码 bug — V2 之前直接用 encoded path 打不开)

### 自动选 Tab(#9 - #10)
13. 选中"已有 file:// link"的文字(刚做过的)→ Cmd+K → **LinkPanel 默认打开"文件" Tab**
14. 同上,看到底部"移除链接"按钮 — 点击 → link mark 消失

### 失败提示(#11)

## 4. 已知约束 & 后续优化

### 4.1 link mark 显示无协议图标

V2 link mark 默认是文字加下划线,**用户看到 link 不知道是 file/media/web**。
Phase B+ 可考虑给不同协议的 link 加协议图标(本阶段不做)。

### 4.2 Windows 路径未支持

POSIX `/` 分隔符,Windows `C:\...` 不支持(跟 B3.14 决策一致)。

### 4.3 大文件 import 阻塞

FileReader.readAsDataURL 对 50MB+ 大文件可能卡 1-3 秒,UI 显 "导入中..." 状态(对齐 V1)。

### 4.4 file://  URL decode 修复影响范围

build-link-click-plugin 修了 file:// 解码 — **理论上是兼容修复**(老 href 无 encoding 也能 decode 成自己),但若有用户已存的 file:// link,验证时一并测(用户验证清单 #12)。

## 4.5 补丁:右键"移除链接"(用户反馈)

实施末用户提议:既然 LinkPanel 加链接,右键应配套移除链接。**追加补丁**:

| 文件 | 改动 |
|---|---|
| `slot/interaction-registries/context-menu-registry/context-menu-types.ts` | EnabledWhen 加 `'has-link'` + ContextInfo 加 hasLink 字段 |
| `slot/interaction-registries/context-menu-registry/context-menu-registry.ts` | 过滤分支加 has-link 条件 |
| `slot/triggers/use-context-menu-trigger.ts` | 通过 selection capability `getCurrent().activeMarks` 算 hasLink |
| `slot/triggers/context-menu-controller.ts` | 默认 ContextInfo 补 hasLink:false |
| `views/note/context-menu-content.ts` | 加 `🔗 移除链接`(group=marks, order=31, enabledWhen='has-link') |
| `views/note/note-commands.ts` | 注册 `note-view.cm-remove-link` → textEditingDriverApi.removeLink |
| `views/web/context-menu-integration.ts` | 补 hasLink:false(web view 不消费此条件) |

设计要点:
- 走 selection capability(slot 层依赖,合规);driver emit selection 时已含 activeMarks,
  capability 缓存 last value,右键时同步读
- enabledWhen='has-link' 不满足时**整项不渲染**(对齐 V1 "条件显示" 行为,不显示 disabled 项)
- 对应 LinkPanel 底部"移除链接"按钮,UX 对称(创建链接 vs 删除链接闭环)

新增 ~30 行 + 几处类型扩展,typecheck 全过。
