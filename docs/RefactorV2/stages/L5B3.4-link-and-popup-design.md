#### 2.1.2 popup 基础设施(driver 之外,在 slot/ 内)

新建 5 个文件:

```
src/slot/
├── interaction-registries/popup-registry/
│   ├── popup-types.ts            ← PopupItem 接口
│   ├── popup-registry.ts         ← register / unregister / get(id)
├── triggers/popup-controller.ts  ← state(activeId / anchorEl)+ show/hide
└── frame-bindings/PopupBinding.tsx ← portal 渲染当前 popup
```

PopupItem 形态:

```ts
export interface PopupItem {
  id: string;                              // 'note-view.popup.link' / 'note-view.popup.color'
  view?: string;
  /** popup 内容 React 组件(driver/view 提供)*/
  Component: React.ComponentType<PopupCloseProps>;
  /** anchorRef + estimated size 用作初始定位(防溢出)*/
  estimatedSize?: { width: number; height: number };
}

export interface PopupCloseProps {
  onClose: () => void;
}
```

trigger 方式:命令 `note-view.open-popup-link` 调 `popupController.show('note-view.popup.link', anchorEl)`。
floating-toolbar 按钮 click 触发命令(沿用现有命令机制,不破坏)。

#### 2.1.3 FloatingToolbarItem 扩 popup-trigger 类型

```ts
interface FloatingToolbarItem {
  ...
  kind?: 'button' | 'popup-trigger';   // 新加,缺省 'button'(向后兼容)
  popupId?: string;                     // kind='popup-trigger' 时必填
}
```

`FloatingToolbarBinding.tsx` 渲染时按 kind 分支(button click → command,popup-trigger → 调 popupController.show)。

## 3. 决策点(请用户拍板)

### Q1:popup 基础设施挂在 slot 还是 view?

- **A. slot 维度**(我推荐):popup-controller / popup-registry / PopupBinding 进 `src/slot/`
  - 理由:跟 SlashMenu / HandleMenu / ContextMenu 同维度;view 只往 popup-registry 注册自己的 popup,基础设施跨 view 共享
  - 后续:graph view / web view 也能用 popup
- **B. view 维度**:NoteView 内部自己做 popup,不进 slot/
  - 理由:简单
  - 风险:每个 view 重复造,违反 V2 "slot 跨 view 复用" 原则

**推荐 A**。

### Q2:FloatingToolbarItem 的扩展兼容性

加 `kind?: 'button' | 'popup-trigger'`(缺省 'button',现有 8 个按钮零改动)。✅ 完全向后兼容,无争议。

### Q3:LinkPanel 是否做 ColorPicker 同款 popup,还是各做各的?

- **A. 共享 popup 基础设施 + 各自 Component**(推荐):popup-registry 管 portal + 位置 + 关闭;具体 LinkPanel / ColorPickerPanel 各自实现 React 组件
- **B. LinkPanel 走 popup,ColorPicker 仍 cycle 按钮**:省掉 ColorPicker 改造
  - 缺点:L5-B3.3 留下的 Plan C-1 缩水成永久状态

**推荐 A**(本阶段顺手把 ColorPicker 升级,L5-B3.3 缩水转正)

### Q4:跨 ws "krig://note/{id}" 路由 — 当前 ws 右栏 还是 用户当前 ws?

V1 行为:右栏(right slot)打开,不影响左栏当前编辑。
V2 候选:
- **A. 同 V1**(推荐):点 link → 当前 ws 的 slotBinding.right = 'note-view' + setActiveNote(target),不改左栏
- **B. 直接切左栏**:左栏 setActiveNote(target),不开右栏
  - V2 简化路径,但用户失去原笔记位置

**推荐 A**(对齐 V1 心智模型 — 链接是"开新栏看",不打断当前编辑)

### Q5:同文档 anchor 滚动 — 走右栏还是左栏当场?

V1 复杂分支:同文档 krig://block 链接走当场 PM 滚动(不开右栏);跨文档才开右栏。
V2 候选:
- **A. 1:1 V1**(推荐):同文档当场滚动,跨文档开右栏
- **B. 一律右栏**:简化,但同文档体验差(为啥要开右栏看自己?)

**推荐 A**。

### Q6:笔记导航历史栈 — 全局还是 per-ws?

V1:全局单一历史栈
V2 候选:
- **A. 全局**(推荐):跟 V1 一致
- **B. per-workspace**:每个 ws 独立历史
  - 优点:多 ws 工作不串
  - 缺点:复杂,且跨 ws 跳转难

**推荐 A**(简单 + 对齐 V1)

### Q7:Cmd+K 快捷键

V1:Cmd+K 触发 LinkPanel(选中文字时 = 加链接;光标无选区时 = 在当前位置加链接,但需要先选词)
V2 候选:
- **A. 走选中文字才触发**(推荐):Cmd+K 时 selection 非空 → 弹 LinkPanel;否则什么都不做
- **B. 复制 V1**:Cmd+K 任何时候触发,空 selection 时智能选当前词
  - 实现复杂

**推荐 A**(简单 + 90% 用户路径)

## 6. 风险与开放问题

### 6.1 popup 跟 floating-toolbar 重叠时谁先关?

floating-toolbar 在选区上方,LinkPanel 也在选区附近 — 两者位置可能冲突。

**预案**:LinkPanel 弹出时 floating-toolbar 不关(用户期待"我从浮条点的按钮,弹出的 popup 跟浮条同框"),用 z-index 让 LinkPanel 盖在 floating-toolbar 上。

### 6.2 选区在 LinkPanel 输入框时 PM selection 失焦,setLink 失败

V1 用 onMouseDown 阻止选区清空(`onMouseDown={(e) => e.preventDefault()}`)+ Apply 时 view.focus(),保持 selection。V2 同款实现。

### 6.3 跨 ws 跳转 link?

当前 ws 的 link 点 `krig://note/{id}` 把目标笔记开到当前 ws 的右栏。如果用户当前在 ws-A,但 target 笔记在 ws-B 的左栏 — 是否要跨 ws 跳?

**预案**:本阶段不跨 ws(就在当前 ws 右栏开)— 跨 ws 跳转留更后阶段。

### 6.4 L5-B3.3 cycle 命令删不删?

cycle 命令可作 keyboard shortcut 备份(用户 Cmd+Shift+某键 直接循环不开 popup)。

**预案**:**保留命令**(成本低),只删 floating-toolbar / 顶部 toolbar 的 button 注册项,改成 popup-trigger。
