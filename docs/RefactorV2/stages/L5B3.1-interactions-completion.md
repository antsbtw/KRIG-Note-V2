## 4. 阶段中遇到 / 解决的问题

**这是 L5-B3.1 实施过程中坦诚记录** — 8 次迭代修复 handle 一个 bug,过程曲折。

### 4.1 handle widget decoration 模式失败(初版误判)

**初版**:每个 block 一个 widget Decoration(`Decoration.widget(pos, ...)`)
**问题**:widget 在 doc 顶层位置,实际渲染在 block 内联首位,跟"block 左侧浮动手柄"语义不符 — 鼠标 hover 探测/CSS 选择器全错位
**修法**:换成"单个浮动 DOM + mousemove 跟踪"(对齐 V1)
**对应 commit**:`9ecc79d`

### 4.2 floating-toolbar 浮条挡住选区文字

**现象**:浮条 top = 选区 top - 8,但浮条高度 ~38px 向下绘 → 盖住选区
**修复**:driver 计算 y 时减掉浮条高度;binding 不用 useCollisionPosition(它假设向右下展开,对 floating-toolbar 不适用)— 自管 viewport 边界
**对应 commit**:`1990d41`

### 4.3 handle 看不见(8 次迭代)

最大坑。从首版"看不见"到最后"对齐 V1 可工作",经历:
- v1:挂在 view.dom 父容器 + position:absolute → 父容器 maxWidth 居中导致定位偏移到屏幕中部
- v2:改 position:fixed + body → handle 视觉位置正确,但鼠标移到 handle 上无法操作
- v3:用 visibility:hidden 控显隐 → 隐藏元素**不接收鼠标事件**,handle mouseenter 永远不触发,cancelHide 永不调,handle 永久消失
- 最终:**对齐 V1 — opacity:0/1 控显隐**(opacity:0 元素仍接事件)+ isHovered 标志位 + 100ms/300ms 延迟 hide

**关键教训**:
1. `visibility: hidden` ≠ `opacity: 0` — 前者不接事件,后者接;UI 控件想保持可点击,用 opacity
2. **优先读 V1 实施再设计**(我跳过了 V1 调研直接写 widget,绕了 8 圈才回到 V1 同款模式)

**对应 commits**:`9ecc79d` / `600b885` / `7714b79` / `ca66348` / `7b15b75` / `b9e0141` / `f584d20` / `ba14936` / `a015062`

### 4.4 dropCursor 加上后拖动变成复制

**现象**:加 prosemirror-dropcursor 后,拖拽 block 释放时 block "复制" 一份(不带格式)而不是移动
**根因**:我用 `view.dom.addEventListener('drop', ...)` 是**冒泡阶段**监听,PM 默认 drop handler 在更早阶段已处理 dataTransfer.text/plain → 当文字插入
**修法**:改用 `plugin.props.handleDrop`(PM 推荐 — 在 PM 默认处理之前调用,返回 true 截获);移除 dragstart 的 text/plain MIME(干扰源)
**对应 commit**:`c4b76b3`

### 4.5 拖动彻底失效(handleDrop 调了但 dataTransfer 是空)

**现象**:`handleDrop` 被 PM 调用,但 `dataTransfer.types` 是 `Array(0)`,所有 getData 返回空字符串
**根因**:**浏览器 drag-and-drop "protected mode"安全策略** — 自定义 MIME 在 dragstart 可写,但 dragover/drop 阶段 dataTransfer 被清空(防 XSS)。业务数据不能跨 dragstart→drop 通过 dataTransfer 传
**修法**:模块级 `activeDrag` 变量(同进程同窗口,不需跨边界)— dragstart 存,handleDrop 读,dragend 清
**对应 commit**:`3d59181`

### 4.6 拖动到 block 间隙时失败

**现象**:posAtCoords 返回 `pos: 163, depth: 0`,代码 `$pos.depth === 0 → return true`,不 dispatch
**根因**:鼠标精确落在 block 之间间隙(doc 顶层),resolved depth=0 — 但这是合法插入位置(用 result.pos 自身即可)
**修法**:depth ≥ 1 用 block 上半/下半判定;depth === 0 直接用 result.pos
**对应 commit**:`8b49f17`

## 9. 遗留问题 / 待优化项

### 9.1 handle 跟编辑区强耦合
**状态**:跟 V1 一样,鼠标离开 view.dom 时 handle 隐藏(可用但有时 100ms 延迟感知偏长)。
**未来候选**:`mouse-tracker` capability(框架级单点鼠标位置中心)— 见 [docs/RefactorV2/notes/mouse-tracker-capability-future.md](../notes/mouse-tracker-capability-future.md)
**触发条件**:出现第 2 个高频鼠标交互时(例如 link mark hover tooltip)

### 9.2 handle 视觉抖动(切笔记)
**现象**:切笔记时 handle 短暂出现在旧位置然后消失。
**应对**:可加 view destroy 时立即 hide handle(L5-B3.x 微调)

### 9.3 dropCursor 的视觉粗细
**状态**:用了 PM 默认 + color #4a90e2 + width 2;视觉不强烈。
**应对**:L5-C+ 真有用户反馈再深化(如 V1 的拖拽预览 ghost / 自定义 dropCursor)

### 9.4 跨 instance 拖拽
**状态**:目前 plugin.handleDrop 检查 `instanceId === activeDrag.instanceId`,跨 NoteView 实例拖拽不支持。
**应对**:L5-B3.3 multi-envelope clipboard 时一并设计跨实例数据传递

### 9.5 handle menu Turn Into 扁平 vs 子菜单
**状态**:Q3=A 拍板扁平 4 项;V1 是子菜单(更紧凑)。如视觉不满意,加子菜单支持(ContextMenuItem 加 children)。

### 9.6 slash menu 的 IME 兼容
**状态**:依赖 PM 标准 inputrules 行为(composition 期间不触发);未深测中文 / 日文 IME 长 query 场景。

### 9.7 capability dnd 业务层 onDrop 是 noop
**状态**:Q5=A 协议形态完整,但实际 drop 业务在 plugin.handleDrop。如果未来跨 view 拖拽,业务层需挪到 capability.onDrop。

### 9.8 selection capability getText() 仍 noop
**状态**:L5-B2 注释的 noop 依然没实施(L5-B3.1 没用到);L5-B3.3 跨 view 复制选区文字时再做。
