## 1. 修复清单(按发生顺序)

### Bug #1 — NodeView 被 PM 销毁,⬇ 按钮失效

**症状**:点 ⬇ 后 console.trace 显示 NodeView destroy → buildFramework,download 流程被打断。

**根因**:
1. PM `selectable=true` 节点的 mousedown 触发 NodeSelection 更新 → NodeView 重建
2. NodeView 内部 DOM 变化(progress bar、文本切换)被 PM 的 MutationObserver 视作外部 DOM 改动 → 重建

**修复**:
| commit | 修复点 |
|---|---|
| `3382ad5` | download-button mousedown 改 click,mousedown 仅 preventDefault+stopPropagation |
| `15cd417` | onTranscript / onPhaseChange 用本地 framework 引用绕过 framework null race |
| `10a4bb2` | NodeView 加 `ignoreMutation` 让 PM 忽略内部 DOM 变化 |

### Bug #2 — yt-dlp 首次执行 1-3 分钟卡死

**症状**:install 完成后 checkStatus 跑 `yt-dlp --version` 卡住,看似 hang。

**根因**:yt-dlp 是 PyInstaller bundle,未签名时 macOS amfi 验证极慢(首次 1-3 分钟)。

**修复**:
| commit | 修复点 |
|---|---|
| `f1d8600` | xattr / checkStatus 都加 timeout 防卡死 |
| `506717c` | install 完成后做 ad-hoc codesign(`codesign --force --deep --sign -`),签后启动 < 2s + checkStatus timeout 提到 30s |

### Bug #3 — YouTube 列表 URL 下载几百个视频

**症状**:贴 `https://www.youtube.com/watch?v=xxx&list=PLxxx` 形式 URL,yt-dlp 下整个 playlist。

**修复**:`c0c729e` 加 `--no-playlist`。

### Bug #4 — 360p 太糊

**症状**:默认 format 只有 360p,用户嫌画质差。

**根因**:
1. yt-dlp 默认拿单文件 progressive(YouTube 上限 360p)
2. 720p+ 是 DASH(video-only + audio-only 分文件),需要 ffmpeg 合并

**修复**:
| commit | 修复点 |
|---|---|
| `df0714e` | format 升 720p,无 ffmpeg 时 fallback 360p |
| `516672d` | 集成 ffmpeg-static,DASH 自动合并 mp4 |
| `dad7d4b` | Vite 把 `__dirname` 重写成 `.vite/build/`,ffmpeg-static 路径错;改用 `app.getAppPath()` 显式构造 + `fs.existsSync` 验证 |

### Bug #5 — 装 yt-dlp 要点两次

**症状**:第一次点 ⬇ 触发 install,装完后要再点一次才下载。

**修复**:`537bc65` runFlow 单次点击完成 install + download。

### Bug #6 — 文件外部删了 UI 仍显示 phase=done

**症状**:Finder 删了视频文件,⬇ 按钮还是 📁(已下载),点击报错。

**修复**:
| commit | 修复点 |
|---|---|
| `6d850f4` | main 端 showItemInFolder 加 `fs.existsSync` 检测,文件不存在返 false |
| `baaf266` | 渲染端按钮 phase=done 时检测文件;不存在自动 reset 到 idle |

### Bug #7 — YouTube 反爬

**症状**:下载报 `Sign in to confirm you're not a bot`。

**迭代过程**(多次反复):
| commit | 方案 | 用户反馈 |
|---|---|---|
| `4945a89` | yt-dlp `--cookies-from-browser chrome` | "不应该去 Chrome 拿用户的 token" — 拒 |
| `f906274` | 改用 webview partition `persist:webview` 的 cookies,导出 Netscape 文件传 yt-dlp | 用户认同 |
| `e91b15e` | 检到没登录弹 modal 引导 | "弹 modal 多此一举" — 拒 |
| `414b018` | 删 modal,直接调 `web-view.open-url` 跳右屏 google 登录 | 用户认同 |

### Bug #8 — 流程顺序错(白装 36MB 才发现要登录)

**症状**:用户没登录 YouTube 时,先装了 36MB yt-dlp,装完才发现下载需要登录,白装。

**修复**:
| commit | 修复点 |
|---|---|
| `12bf158` | `checkYoutubeCookies` 检"真登录"cookie(SID/HSID/SSID/APISID/SAPISID/`__Secure-1PSID`/`__Secure-3PSID`/`__Secure-1PAPISID`/`__Secure-3PAPISID`/LOGIN_INFO 任一存在)而非仅 cookie 数 > 0 — 访客 PREF/VISITOR_INFO 不算登录 |
| `e1322ee` | runFlow 顺序改:**先检登录** → 没登录 → 跳右屏 web view → return;已登录 → 检 ytdlp status → 没装就 install → download |

## 4. 经验教训(沉淀)

### 4.1 PM NodeView 与 DOM 事件
- selectable=true 节点内部 DOM 控件**必须**用 mousedown(preventDefault + stopPropagation) + click(实际逻辑)双层模式,否则 mousedown 触发 PM NodeSelection 更新 → NodeView 销毁重建。
- NodeView 内部异步改 DOM(progress bar、phase 切换)**必须**在 NodeView 类实现 `ignoreMutation` 返 true,否则 PM MutationObserver 看见外部 DOM 改动会重建 NodeView。

### 4.2 macOS 二进制分发
- PyInstaller bundle / Electron 之外的可执行二进制,装完**必须**做 `xattr -dr com.apple.quarantine` + `codesign --force --deep --sign -` ad-hoc 自签,否则 amfi 首次验证 1-3 分钟,看似 hang。
- xattr / codesign / 二进制 `--version` 探测**都加 timeout**,防 macOS Gatekeeper 卡死整个 main 进程。

### 4.3 Vite 打包
- Vite main bundle 会把 `__dirname` 重写成 build dir(`.vite/build/`),依赖 `path.join(__dirname, ...)` 的 npm 包(ffmpeg-static / yt-dlp-wrap 之类)路径会错。修复:用 `app.getAppPath()` 显式构造 + `fs.existsSync` 多候选探测。
- electron-builder 的 `asarUnpack` 配置原生二进制路径会从 `.../app.asar/` 变成 `.../app.asar.unpacked/`,代码两边都要试。

### 4.4 YouTube 反爬 + 真登录检测
- "有 cookies"≠"已登录":访客访问 youtube.com 也会有 PREF/VISITOR_INFO 等。**真登录**要看 Google session cookies(SID/HSID/SSID/APISID/SAPISID/`__Secure-1PSID`/`__Secure-3PSID` 等)任一存在。
- 不读用户系统 Chrome cookies,用 app 自己的 webview partition(`persist:webview`)隔离;导出 Netscape 格式给 yt-dlp `--cookies <file>`。

### 4.5 流程顺序设计
- 装大依赖之前**先做廉价的前置检查**(本例:登录态检测)。重组合,不要先装才发现要登录。

### 4.6 诊断 log 工作流
- 修 bug 加 log 两段式:`diag: ...` commit 加 log,`fix: ...` commit 修代码;最后一个 `chore: 清理 ...` commit 一次性删干净。便于事后回溯每个修复对应的诊断证据,也便于一次性清理。
