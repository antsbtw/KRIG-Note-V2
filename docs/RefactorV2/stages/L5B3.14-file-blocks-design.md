## 0. 背景

V1 三件套各司其职(`v1-note-migration-audit.md` Phase B4):

| 节点 | V1 文件 | 行数 | 角色 |
|---|---|---|---|
| **fileBlock** | `blocks/file-block.ts` | 297 | block 级附件卡片;字节进 mediaStore,自包含跟着 note 走 |
| **fileLink** | `blocks/file-link.ts` | 148 | inline atom 文件 chip 📎,段落中引用附件 |
| **externalRef** | `blocks/external-ref.ts` | 254 | block 级"引用卡片",只存 file:// / https:// URL,**不拷字节** |

**V1 总规模 ~700 行**,V2 直迁估 ~750 行(加 mediaResolve IPC + 补 main 侧)。

### 三者区别(对齐 V1 设计哲学,不要混淆)

| 维度 | fileBlock | fileLink | externalRef |
|---|---|---|---|
| PM 类型 | block | **inline atom** | block |
| 字节存储 | mediaStore(拷进 KRIG) | 同 fileBlock | 仅存 URL,不拷 |
| 文件移动 / 删除 | 不影响(自包含) | 同 fileBlock | **断链**(原文件丢) |
| KRIG Graph 价值 | 内部资产 | 内部资产 | 外部知识引用关系 |
| 典型场景 | AI 生成 PDF / Code Interpreter 文件 / 用户拖入 | 段落中提到附件 | 引用本机 PDF / 网页书签 |

### V1 → V2 改造点

1. **viewAPI(IPC)→ V2 直接调 electronAPI**(单 React tree,无 viewAPI 层)
2. **render-block-base.ts** V1 共享基类 → V2 各 block 独立 NodeView(L5-B3.5 已建立模式,
   不复用 V1 的 createPlaceholder 抽象)
3. **node 名 camelCase**(fileBlock / fileLink / externalRef)— V2 既有约定:
   - block id 必须无短横线(否则 PM content 表达式 SyntaxError,见 feedback memory)
   - V2 已有 mathBlock / mathInline / noteLink 走 camelCase
4. **render-block 全局 hover 装饰**(V1 `dom.classList.add('render-block')` 配
   global block-handle plugin)— V2 当前不依赖此模式,统一走 V2 block-handle plugin
   现有路径(光标在该 block 时显手柄)
5. **electronAPI 缺口**:补 3 个 IPC(本阶段一并加)
   - `mediaResolvePath(media://) → 本地 path`
   - `showItemInFolder(path)`
   - `getFilePath(File)`(Electron 32+ webUtils.getPathForFile 包装)

## 1. 主进程 IPC 补齐(前置)

### 1.1 mediaResolvePath

`src/platform/main/media/media-store-impl.ts` 已知 `MEDIA_DIR`,
`media://<path>` → `path.join(MEDIA_DIR, urlPath)`。

新加 export:
```ts
// media-store-impl.ts
export function resolveMediaPath(mediaUrl: string): string | null {
  if (!mediaUrl.startsWith('media://')) return null;
  const urlPath = mediaUrl.replace('media://', '');
  // 防御:校验解析后路径仍在 MEDIA_DIR 内(防 ../ 越界)
  const candidate = path.resolve(MEDIA_DIR, urlPath);
  if (!candidate.startsWith(MEDIA_DIR)) return null;
  return candidate;
}
```

ipc handler 包装:
```ts
// src/platform/main/ipc/media-handler.ts(扩展)
ipcMain.handle('media:resolvePath', (_e, url: string) => {
  const p = resolveMediaPath(url);
  return p ? { success: true, path: p } : { success: false };
});
```

### 1.2 showItemInFolder

`src/platform/main/ipc/shell-handler.ts` 加:
```ts
ipcMain.handle('shell:showItemInFolder', async (_e, filePath: string) => {
  if (!path.isAbsolute(filePath)) return { ok: false, reason: 'not absolute path' };
  shell.showItemInFolder(filePath);
  return { ok: true };
});
```

### 1.3 getFilePath(Electron 32+)

`webUtils.getPathForFile(File)` 是新 API,**只能在 renderer 侧调**(File 对象在 main 不能传)。
preload 直接暴露:
```ts
// preload
import { webUtils } from 'electron';
contextBridge.exposeInMainWorld('electronAPI', {
  // ...
  getFilePath(file: File): string {
    try { return webUtils.getPathForFile(file); } catch { return ''; }
  },
});
```

### 1.4 electron-api.d.ts 加 3 个声明

```ts
mediaResolvePath(mediaUrl: string): Promise<{ success: boolean; path?: string }>;
showItemInFolder(filePath: string): Promise<{ ok: boolean; reason?: string }>;
getFilePath(file: File): string;  // 同步
```

### 1.5 storage 层封装

```ts
// src/storage/media-store.ts 加
export async function mediaResolvePath(url: string): Promise<string | null> {
  if (!window.electronAPI?.mediaResolvePath) return null;
  const r = await window.electronAPI.mediaResolvePath(url);
  return r.success ? r.path ?? null : null;
}
```

`showItemInFolder` / `getFilePath` 直接走 electronAPI,view 层用。

## 2. fileBlock(block 级附件卡片)

### 2.1 NodeSpec

```ts
// src/drivers/text-editing-driver/blocks/file-block/spec.ts
const fileBlockNodeSpec: NodeSpec = {
  group: 'block',
  // V1 经验:NOT atom: true — PM 不给 atom 派 mousemove,block-handle hover 不工作
  // V1 用 content: 'text*' 让 block 可"进入"但 NodeView 不暴露 contentDOM 保持只读
  content: 'text*',
  draggable: true,
  selectable: true,
  attrs: {
    src:      { default: '' },     // media:// URL
    mediaId:  { default: '' },
    filename: { default: '' },
    mimeType: { default: '' },
    size:     { default: null },   // bytes
    source:   { default: null },   // 'user-uploaded' / 'ai-generated' / 等(可选 metadata)
  },
  parseDOM: [{ tag: 'div.krig-file-block', getAttrs(dom) { ... } }],
  toDOM(node) { return ['div', { class: 'krig-file-block', ...attrs }]; },
};
```

### 2.2 NodeView(两态)

**placeholder 态**(无 src):上传按钮 + URL embed
**card 态**(有 src):图标 + 文件名 + MIME · 大小 + [打开] [在 Finder 显示]

```ts
// src/drivers/text-editing-driver/blocks/file-block/node-view.ts
const fileBlockNodeView: NodeViewConstructor = (initialNode, view, getPos) => {
  const dom = document.createElement('div');
  dom.classList.add('krig-file-block');
  dom.setAttribute('contenteditable', 'false');

  function renderCard(n: Node): void { /* icon + meta + [打开] [Finder 显示] */ }
  function renderPlaceholder(): void { /* file picker + url embed */ }

  function paint(n: Node) {
    if (n.attrs.src) renderCard(n);
    else renderPlaceholder();
  }

  paint(initialNode);

  return {
    dom,
    update(updated) { /* 切换态;src 变就 paint */ },
    stopEvent() { return true; },
    ignoreMutation() { return true; },
  };
};
```

**关键差异 vs V1**:
- V2 不复用 `createPlaceholder` 共享组件 — 内联简单 UI(file picker `<input type=file>` + 一个 URL 输入框)
- V1 `dom.classList.add('render-block')` 全局 hover 装饰 — V2 直接由 block-handle plugin 处理,不挂额外 class

### 2.3 上传路径(对齐 paste-media / image NodeView)

```ts
input.addEventListener('change', async () => {
  const file = input.files?.[0];
  if (!file) return;
  const reader = new FileReader();
  reader.onload = async () => {
    const dataUrl = reader.result as string;
    const r = await mediaPutBase64(dataUrl, file.type, file.name);
    if (r.success && r.mediaUrl) {
      updateAttrs({
        src: r.mediaUrl,
        mediaId: r.mediaId || '',
        filename: file.name,
        mimeType: file.type,
        size: file.size ?? null,
        source: 'user-uploaded',
      });
    }
  };
  reader.readAsDataURL(file);
});
```

### 2.4 打开 / 显示 in Finder

```ts
async function openFile(src: string): Promise<void> {
  if (src.startsWith('media://')) {
    const path = await mediaResolvePath(src);
    if (path) await window.electronAPI?.openPath?.(path);
  } else if (src.startsWith('file://')) {
    const path = decodeURIComponent(new URL(src).pathname);
    await window.electronAPI?.openPath?.(path);
  } else {
    await window.electronAPI?.openExternal?.(src);
  }
}

async function showInFinder(src: string): Promise<void> {
  let path: string | null = null;
  if (src.startsWith('media://')) path = await mediaResolvePath(src);
  else if (src.startsWith('file://')) path = decodeURIComponent(new URL(src).pathname);
  else if (src.startsWith('/')) path = src;
  if (path) await window.electronAPI?.showItemInFolder?.(path);
}
```

## 3. fileLink(inline atom 文件 chip)

### 3.1 NodeSpec(对齐 noteLink 模式)

```ts
const fileLinkNodeSpec: NodeSpec = {
  inline: true,
  group: 'inline',
  atom: true,
  attrs: {
    src:      { default: '' },     // media:// / file:// / 绝对路径
    filename: { default: '' },
  },
  parseDOM: [{ tag: 'span.krig-file-link', getAttrs(dom) { ... } }],
  toDOM(node) {
    return ['span', { class: 'krig-file-link', 'data-src': node.attrs.src },
      ['span', { class: 'krig-file-link__icon' }, '📎'],
      ['span', { class: 'krig-file-link__name' }, node.attrs.filename || 'file'],
    ];
  },
  // 复制还原源码(对齐 noteLink leafText 模式)
  leafText: (node) => `📎${(node.attrs.filename as string) || 'file'}`,
};
```

### 3.2 NodeView(简单)

- 渲染 `📎 <filename>`,contenteditable=false
- 单击 → 打开文件(走 fileBlock 同样的 openFile)
- 右键 → 内置小菜单(打开 / 在 Finder 显示 / 复制路径)
  - V1 自绘 contextmenu 元素 — V2 走同样的内联做法(跟 V1 对齐,简单)
  - 不接 V2 contextMenuRegistry(那个是 view 级 PM 区域 contextmenu,inline atom 内嵌
    单独菜单更直观)

### 3.3 Slash menu

V1 `slashMenu: null`(只通过其他流程创建,如 paste / drag)。
V2 也保持 null — fileLink 是从 fileBlock 派生 / paste 流程产生,**不在 slash 主菜单**。

如果用户希望在段落中插入文件 chip,先创建 fileBlock,然后 "drag handle 转 inline" 是
未来的可能路径。本阶段不做。

### 4.3 file picker → file:// URI

```ts
input.addEventListener('change', async () => {
  const file = input.files?.[0];
  if (!file) return;
  const filePath = window.electronAPI?.getFilePath?.(file);
  if (!filePath) {
    console.warn('[externalRef] getFilePath returned empty');
    return;
  }
  const href = `file://${filePath.split('/').map(s => s ? encodeURIComponent(s) : '').join('/')}`;
  updateAttrs({
    kind: 'file',
    href,
    title: file.name,
    mimeType: file.type || '',
    size: file.size ?? null,
    modifiedAt: file.lastModified ?? null,
  });
});
```

### 4.4 打开

function openRef(href: string): void {

    try {
      const path = decodeURIComponent(new URL(href).pathname);
      window.electronAPI?.openPath?.(path);
    } catch { /* ignore */ }
  } else {
    window.electronAPI?.openExternal?.(href);
  }
}

## 6. 决策点

### Q1:render-block 共享基类 vs 各自独立

V1 用 `createPlaceholder` 共享 placeholder UI(image/audio/video/fileBlock/externalRef 复用)。

- **A. V2 各自独立**(对齐 L5-B3.5 image 已建立模式):简单直接,3 个 NodeView 各写各的
- **B. 抽 V2 共享 placeholder helper**:DRY,但 audio/video 还没做,过早抽象

**推荐 A** — V2 image 已经是独立 NodeView 模式;audio/video 做完后再回头看是否抽
(三件不够,五件后再定)— 跟 V2 "不过早抽象" 原则一致。

### Q2:fileBlock content 是 text* 还是 atom

V1 选 `content: 'text*'`(非 atom)— 注释说"PM 不给 atom 派 mousemove,block-handle
hover 不工作";NodeView 不暴露 contentDOM 保持只读。

V2 看 image block 选了什么:
- L5-B3.5 image `content: 'block'`(caption 是 block 子)— image 有 caption 需要可编辑
- mathBlock 用 `atom: true`?待查

V2 fileBlock 没有 caption,**最干净的是 atom**;但若 V2 block-handle plugin 也跟 V1
一样依赖 mousemove,需 V1 同款 `content: 'text*'` 兜底。

**推荐 A. atom: true**(简单,V2 block-handle 实现可能跟 V1 不同 — 如果 hover 真的
不工作再退回 'text*'),实施时先验证 V2 block-handle 行为。

### Q3:NodeView 自绘 contextmenu vs 接 V2 contextMenuRegistry

V1 fileLink 自绘小菜单(打开/Finder 显示/复制路径)。

- **A. 自绘**(V1 行为,简单内嵌):对齐 V1
- **B. 接 V2 contextMenuRegistry**:统一菜单系统,但 inline atom 在 PM 区域内,需要
  `e.target` 路由判断 — 复杂

**推荐 A** — 内嵌小菜单 ~30 行,V2 contextMenuRegistry 是 view 区域级菜单,挂 inline
atom 节点不直观;V1 已验证模式直迁。

### Q4:externalRef file picker 失败处理

Electron 32+ `webUtils.getPathForFile` 对非 disk 来源(Blob URL / 拖拽 web 图片)返回空。

- **A. 同 V1**:console.warn 不做事,placeholder 留显
- **B. 用户提示**:placeholder 显红字"无法解析文件路径,请用 URL embed"

**推荐 B** — 用户层面失败信息更友好,实现仅几行(在 placeholder 区显 error)。

### Q5:fileBlock 上传完成后 caret 行为

上传完成 → 节点从 placeholder 切到 card 态。光标应该:
- **A. 留在节点上**(NodeSelection):用户可以连按 Enter 加段落
- **B. 移到节点之后**(下一行):自动续写

**推荐 A** — 对齐 V1 行为,用户可显式按 Enter 推进。

## 9. 风险

### 9.1 mediaResolvePath 路径越界

`media://../../etc/passwd` 类路径攻击。
**预案**:`resolveMediaPath` 内 `path.resolve(MEDIA_DIR, urlPath)` 后必须 startsWith(MEDIA_DIR)
否则返回 null(白名单兜底)。

### 9.2 webUtils.getPathForFile 在 Blob URL 失败

Electron 32+ 不在 web blob / Drag from browser 暴露 path。
**预案**:placeholder 显 error 提示(决策 Q4 = B),用户改用 URL embed。

### 9.3 fileBlock content: 'text*' 副作用

V1 注释提到非 atom + content: 'text*' 是为了 block-handle hover 工作;但 PM 仍可能
让用户在内部输文字(虽然 NodeView 不暴露 contentDOM)。
**预案**:实施时用 atom: true 优先,若 hover 不工作再退回。

### 9.4 file:// URL 编码

V1 用 `path.split('/').map(s => s ? encodeURIComponent(s) : '').join('/')` 编码。
**注意**:Windows 路径(`C:\...`)需要单独处理(本阶段 macOS-only,假设斜杠路径)。
**预案**:本阶段不支持 Windows;留 issue 文档。
