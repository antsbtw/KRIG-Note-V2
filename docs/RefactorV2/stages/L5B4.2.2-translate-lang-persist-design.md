## 0. 背景

L5-B4.2(基线)的双屏同步翻译用 Google Translate widget 注入实现,体验丝滑(页面无重载、链接自然、无反爬虫风险),但**切语言**(zh-CN → ja 等)不可靠 — widget 一旦初始化用 cookie 决定语言,运行时改 cookie / dispatchEvent 都不刷新页面。

L5-B4.2.1 尝试用 Google translate.goog URL 路径替代 widget 注入,虽然解决了切语言问题,但其他维度全面降级:reCAPTCHA 频繁触发、加载慢、左栏被 widget 中转 URL 污染、Google 自家服务直接不工作。**已确认放弃 merge,分支归档**。

本阶段(L5-B4.2.2)选择第三条路:**接受 widget 注入"切语言不可靠"的天然限制,改用产品 UX 兜底** —

- 用户切语言 → 持久化到 per-ws state + 弹 banner "需重启 app 才生效"
- 重启后语言偏好恢复,新会话用新语言
- 第一次启动用 `navigator.language` 自动选默认语言(避免硬编码 zh-CN 不友好)

切语言低频(用户大概率长期看一种语言,只是默认值需要符合自己的母语),"重启换语言"在用户工程上完全可接受。

## 1. 实施范围

### 1.1 In-Scope

#### 1.1.1 targetLang 持久化

写入 per-ws state 的 web pluginState:

```ts
interface WebPluginState {
  currentUrl: string;
  targetLang?: string;  // 新增,可选(老 ws 没此字段时用 navigator.language fallback)
}
```

读写 helper:`getWebTargetLang(ws)` / `setWebTargetLang(workspaceId, lang)`,沿用 `currentUrl` 同款 pattern。

#### 1.1.2 第一次启动默认语言 — `navigator.language`

```ts
function getDefaultTargetLang(): string {
  const lang = navigator.language || 'en';
  // 简单规则:中文系统统一 zh-CN,日韩英直接对应,其他用 en
  if (lang.startsWith('zh')) return 'zh-CN';
  if (lang.startsWith('ja')) return 'ja';
  if (lang.startsWith('ko')) return 'ko';
  return 'en';
}
```

仅在 per-ws state 没有 `targetLang` 时用作默认,后续从持久化读。

#### 1.1.3 翻译入口统一到 WebToolbar

**目标:** 把"开关翻译"+"配置语言"两个动作合并到 WebToolbar 那个翻译按钮,**右栏顶部 select 删掉**(避免双入口)。

**新交互(WebToolbar 翻译区域):**

```
[🌐] [▾]
 │    └── 旁边小箭头(展开语言菜单)
 └── 翻译按钮(单击 = 开关翻译,目标语 = 持久化的 targetLang)
```

点小箭头 ▾ 展开下拉菜单:

```
✓ 中文      ← 当前
  日本語
  한국어
  English
```

- 点列表里某个语言:
  - 翻译**已开** → 写入持久化 targetLang + 弹 banner"重启后生效,立即重启?"
  - 翻译**未开** → 写入持久化 targetLang(下次开翻译用),不弹 banner

#### 1.1.4 重启 banner

在 WebView 内顶部插入(只在切了 lang 但没重启时显示):

```
┌─────────────────────────────────────────────────────────┐
│ ⚠ 已选 日本語,需重启 app 才生效  [立即重启] [稍后]  │
└─────────────────────────────────────────────────────────┘
```

- 立即重启 → 调 `electronAPI.restartApp()` → main 进程 `app.relaunch() + app.exit(0)`
- 稍后 → 关闭 banner;持久化的 lang 在下次 app 启动时生效

#### 1.1.5 main 进程 + preload `restartApp` IPC

- `IPC_CHANNELS.SHELL_RESTART_APP` 新通道
- main 进程 handler:`app.relaunch(); app.exit(0);`
- preload 暴露 `electronAPI.restartApp(): void`
- 类型加到 `electron-api.d.ts`

#### 1.1.6 删除 TranslateWebView 顶部 select

- 移除 LANG_OPTIONS 和 select 的 JSX
- targetLang 改成从 per-ws state 读取 + initial mount 时传给 TranslateDriver
- 切 lang useEffect 整个删除(因为 targetLang 在持久化路径下只有 mount 时读一次,运行时不变)

### 1.2 Out-of-Scope

| 项 | 留 |
|---|---|
| Widget 注入翻译核心 | 保留(L5-B4.2 baseline 路径不变) |
| sync driver / slot-bus | 保留 |
| disablewebsecurity / CSP bypass | 保留 |
| 切语言运行时刷新 | **不做** — 走重启路径 |

## 2. 决策点

### Q1:翻译按钮配置入口形式?

**A. 翻译按钮 + 旁边小箭头 ▾(展开下拉)** — 两个独立点击区
**B. 翻译按钮(单击开关,右键展开菜单)** — 单按钮双行为
**C. 设置面板里加"翻译目标语言"全局项** — 入口深,但语言是稳定偏好,改的少

**推荐 A** — 视觉上"翻译开关 + 语言配置"是一组,用户看小箭头就懂"展开看语言选项"

### Q2:重启提示形式?

**A. WebView 内顶部 banner**(可关、可"立即重启") — 非阻塞,推荐
**B. 浏览器原生 confirm** — 立即阻塞,体验粗糙
**C. 啥都不弹,静默持久化** — 用户切了发现没变会困惑

**推荐 A**

### Q3:per-ws targetLang 还是全局 targetLang?

**A. per-ws** — 每个 workspace 独立(不同 ws 看不同语种新闻)
**B. 全局** — app 级偏好,所有 workspace 共享

**推荐 A** — 与 currentUrl 一致,更灵活;切 ws 时各自加载各自的语言

### Q4:第一次启动默认语言?

**A. `navigator.language` 自动**(中文系统 → zh-CN,日 → ja...)
**B. 第一次启动弹"选你的母语"框** — 显式但打扰
**C. 默认 English + banner 提示设置** — 国际化中性但中文用户不友好

**推荐 A**

### Q5:重启时如何不丢失用户当前 ws 状态?

V2 已经有 ws 持久化机制(workspaceManager),app.relaunch 后会自动恢复 — 不需要额外做。

✅ 已验证:`workspace-state` 持久化覆盖 currentUrl / activeNoteId / etc.,重启后状态自动恢复。targetLang 加进去后也走同一机制。

## 5. 风险

### 5.1 navigator.language 在 webview 里的可靠性

renderer 主进程的 navigator.language 是 Electron 进程语言,通常跟系统一致。webview 子进程不需要(我们只在 renderer 用)。**低风险**。

### 5.2 banner 与 WebToolbar 高度交互

WebView 内顶部 banner 会挤压 webview 高度;切 ws 时 banner 状态需要重置(否则 ws-A 切 lang 后 banner 出现,跳到 ws-B banner 仍存)。**用 transient state(per-ws)管理,不持久化**。

### 5.3 重启不平滑

`app.exit(0)` 突然 — 但 V2 ws 持久化覆盖 + 用户主动触发,可接受。可考虑 `app.quit()` 走正常退出流程(触发 before-quit 钩子);需要看 ws 持久化是否在 before-quit 时落盘。**实施时确认**。
