## 0. 背景

L5-B3.5 image / L5-B3.6 math 完成后,Phase A 还差最后一个:**table**。

为什么 table 是 Phase A 收官:
- AI 提取(尤其 ChatGPT / Claude 对比表 / API 文档总结)大量含表格
- markdown `| col | col |` 是基础语法,L5-B4.3.1 md-to-pm 已有解析逻辑
  (现在走 unknown 占位)
- table 是 NoteEditor 最复杂的 block(嵌套 tableRow > tableCell/tableHeader > block),
  完成后 Phase A 闭环 → 解锁 L5-B4.3.2/3 AI 闭环

**关键效果(预期)**:跟 image/math 同款"反向驱动 schema 补齐"路线 — md-to-pm 主体不动,
markdown 转出来的 `table` / `tableRow` / `tableHeader` / `tableCell` 节点自动从 unknown
占位升级为真实 table。

## 2. NodeSpec 设计(4 个节点)

### 2.1 schema 关系

```
table          content='tableRow+'           tableRole='table'    isolating
  └── tableRow    content='(tableCell|tableHeader)+'  tableRole='row'
        ├── tableCell    content='block+'   tableRole='cell'        isolating
        └── tableHeader  content='block+'   tableRole='header_cell' isolating
```

**关键约束(命中 V2 短横线 schema 限制)**:
- `tableRow` content 引用 `tableCell` / `tableHeader` 名 — V2 PM content 表达式不能含 `-`,
  但这几个名字都驼峰,**OK**
- `tableCell.content = 'block+'` — 用 group 引用,**OK**
- 不能直接 ref `text-block`(短横线问题),用 group 'block'(已是 V2 既有约定)

### 2.2 attrs

V1 cell / header 都有:
- `colspan`(default 1)
- `rowspan`(default 1)
- `colwidth`(default null,数组 `number[]`)
- `align`('left' | 'center' | 'right' | 'justify' | null)

V2 直迁,parseDOM / toDOM 跟 V1 一致。

## 3. NodeView 设计(table)

V1 `tableNodeView`(422 行)实现:

### 3.1 外层布局

```
.krig-table-block-wrapper
  ├── .krig-table-block__col-indicators   (顶部 hover 列指示条)
  ├── .krig-table-block__row-indicators   (左侧 hover 行指示条)
  ├── .krig-table-block__scroll
  │     └── <table class="pm-table">
  │           ├── <colgroup> (动态 col widths)
  │           └── <tbody> = contentDOM   ← PM 接管
  ├── button.krig-table-block__add-col-btn  (右侧 +col)
  └── button.krig-table-block__add-row-btn  (底部 +row)
```

### 3.2 列/行指示器交互

- hover table → 显示指示器条(顶部 col / 左侧 row)
- 点指示器 → 弹原生 DOM 菜单(`contenteditable="false"`)
  - 列菜单:Add column before/after / Duplicate column / Delete column / Align(分隔)
  - 行菜单:Add row before/after / Duplicate row / Delete row
- 点 +col/+row 按钮 → 直接在末尾加列/行

### 3.3 colwidth 同步

V1 用 `prosemirror-tables` 的 `updateColumnsOnResize` 同步 colgroup width
(用户拖拽 cell 边界改宽)。V2 直迁。

```ts
plugin: () => [

  tableToolbarPlugin(), // CellSelection 时弹 toolbar(merge/split/align)

```

V2 直迁这 3 个 plugin。

## 5. 决策点

### Q1:prosemirror-tables 版本

- V1:`^1.8.5`
- 最新:`^1.x`(估计仍 1.x,API 稳定)

**A. 跟 V1 一致 ^1.8.5**(直迁安全)
**B. 装最新**(可能有改进,但要测兼容)

**推荐 A** — 直迁 = 直迁

### Q2:CellSelection toolbar UI 是否纳入本阶段(B+ 修订)

V1 `tableToolbarPlugin`(~200 行):用户多选 cell(CellSelection)时屏幕弹浮动 toolbar,
提供:merge cells / split cell / align(left/center/right/justify)。

**v0.1 草案推荐 A(直迁全套)** → 用户反馈担心未来扩展性 → **改 B+**:

- **B+. 业务 commands 全保留(`mergeCells` / `splitCell` / `setCellAlign` 等都 export),
  但本阶段不写 toolbar UI plugin** — 后续接 V2 floating-toolbar registry / context-menu
  registry 暴露这些命令(更一致,且 V2 升级 popup 系统时 table 自动跟着升级,不会成
  孤岛)

具体:
- ✅ commands.ts 完整直迁(~260 行 — duplicateRow / duplicateColumn / setCellAlign 等)
- ✅ prosemirror-tables 自带的 mergeCells / splitCell / addColumnAfter 等 export 出去
- ❌ tableToolbarPlugin(`~200 行 V1 浮动 DOM toolbar`)不迁
- 🔜 后续 sub-stage(L5-B3.7.1)接 V2 floating-toolbar / context-menu registry,把
  上述命令暴露成 toolbar 项 / 菜单项 — 跟 bold/italic/link 等同一体系

### Q3:列/行指示器 + 菜单 UI(B+ 修订)

V1 view.ts 422 行里 ~250 行是这块装饰性 UX。

**v0.1 草案推荐 A(直迁全套)** → 用户反馈担心扩展性 → **改 B+**:

- **B+. 砍指示器 + 自建 DOM 菜单**(~250 行 V1 装饰),只留必要的:
  - ✅ `<table><tbody>` contentDOM
  - ✅ outer `.krig-table-block__scroll` wrapper(防超宽溢出)
  - ✅ +col / +row 按钮(右侧 / 底部,简单按钮)
  - ✅ colwidth resize(走 prosemirror-tables `columnResizing()` 插件 — 库内置)
  - ❌ hover 列/行指示器条(留后续)
  - ❌ 自建 DOM 菜单(改用 V2 context-menu registry)
- 用户操作列/行通过:**右键菜单**(V2 context-menu registry,本阶段也不做菜单注册,
  留 L5-B3.7.1)/ **Tab 导航**(自带)/ **+col/+row 按钮**(本阶段做)

业务命令 commands.ts 仍然完整直迁(addColumnAfter / deleteColumn / duplicateRow 等),
未来 sub-stage 接 V2 context-menu registry 暴露,跟 image / math 风格统一。

NodeView 工作量从 ~420 行缩到 ~150 行。

### Q4:slash menu 入口

加 `/table` 项。但 V1 默认插入 `2x2` 表格,V2 应该:

- **A. 默认 3x3 + 第一行是 header**(常见办公表格)
- **B. 默认 2x3 + 第一行 header**(对齐 V1)
- **C. 弹"行数 / 列数"输入**:简单弹窗

**推荐 A** — 3x3 是 markdown 表格常见尺寸;后续可加 `/table 5x4` 这种参数化

### Q5:CSS 命名

V1 用 `.table-block*` / `.pm-table` / `.table-block__menu` 等。V2 加 `krig-` 前缀:
- `.krig-table-block` / `.krig-table-block__wrapper` / `.krig-table-block__scroll`
- `.krig-pm-table` / `.krig-pm-table th` / `.krig-pm-table td`
- `.krig-table-block__menu` / `.krig-table-block__menu-item`
- `.krig-table-block__col-indicator` / `.krig-table-block__row-indicator`

**推荐 A**:加 `krig-` 前缀(跟 V2 现有 block CSS 风格一致)

## 8. 风险

### 8.1 prosemirror-tables 跟 V2 现有 keymap 冲突

V2 已有 buildListKeymap / buildCodeBlockKeymap / buildHardBreakKeymap 等,table keymap
里 Tab 行为可能跟 V2 现有 tab handling 冲突(比如 list Tab 缩进)。

**预案**:tableKeymap 装在 table BlockSpec.plugin 里,优先级高于全局 keymap;table 内
按 Tab 由 prosemirror-tables 优先处理。如果冲突,在 plugin 顺序里把 table 放前面。

### 8.2 isolating + cascadeBoundary

V1 table / tableCell / tableHeader 都设 `isolating: true`(选区 / 删除 / undo 不跨边界)。
V2 BlockSpec 也有 `cascadeBoundary` 概念,V1 设 `cascadeBoundary: true`。两者语义可能不
完全等价。

**预案**:实施时按 V1 设;如果出现选区跨边界 bug 再调

### 8.3 CSS 复杂度

V1 table CSS 散在 note.css 多处,本阶段需要从 V1 摘出来 + 加 `krig-` 前缀。
工作量稍多但风险低。
