## 1. 实际改动 vs 设计 — 偏差点

### 1.1 update() transcript 路径改 recomputeCues(不 paint)

设计 § 2.3 文案"transcript 变化触发 paint";**实际**改成只 recomputeCues
+ 不重 paint(只在 src/embedType 切换时 paint):

**原因**:dispatch attr → update → paint → 重建 textarea → 折叠状态丢失 + 光标 jump。
textarea 输入触发 dispatch 时,**textarea 已经持有最新文本**,再重建反而破坏 UX。

代码注释里完整说明了这个判断(node-view.ts update() 函数内 ~10 行注释链)。**正确的 cues 缓存**
仍能更新(recomputeCues),B3.19.2 CC 浮层会消费它。

### 1.2 cuesCache 加 eslint disable 注释

设计 § 2.4 写"cues 缓存"。本段 cuesCache 没消费方(B3.19.2 才用),lint 报
`@typescript-eslint/no-unused-vars` 警告。

加 `// eslint-disable-next-line` + 注释明示"B3.19.2 之前的过渡"。**不删 cuesCache**:
- 数据流路径(parse → cache → 消费)在本段就建立,B3.19.2 不用回头改 import / 声明
- 注释明示意图,后续审计能跟踪

### 1.3 transcript toggle 按钮 label 显字幕行数

设计文档说按钮显 `Transcript [Show ▾]`。**实际**:
- 空 transcript:`Transcript [Show ▾]`
- 有 transcript:`Transcript (5 lines) [Show ▾]`(显有效行数,排除注释/空行)

UX 优化 — 折叠状态用户也能感知有几行字幕,跟"先存数据,再做 UI" 渐进路径一致
(B3.19.2 加 CC 浮层之前用户也能看到字幕"已经有"了)。

### 1.4 transcript 区只在播放态显(对齐 v0.2 修正 + 完成判据 #10)

`appendTranscriptArea(n)` 只在 `buildDirectVideo` / `buildYouTubeEmbed` 末尾调,
**`buildPlaceholder`** 不调(没视频谈不上字幕)。完成判据 #10 验证项。

## 4. 已知约束 & 后续

### 4.1 cuesCache 暂无消费方

本段 cuesCache 仅 parse 后存,无渲染消费 — B3.19.2 CC 浮层会用 `findActiveCue(cuesCache, video.currentTime)`。
eslint 警告由 disable 注释明示意图。

### 4.2 transcript 折叠状态不持久化

每次打开笔记 transcript 区**默认折叠**(避免每个 video block 大块字幕区铺屏)。
若用户希望"上次打开是展开就展开",留 Phase D 加 attrs.transcriptExpanded
(本段不预留,Q2=A 决策)。

### 4.3 youtube 模式 CC 同步不工作

iframe 跨 origin 拿不到 video.currentTime。**B3.19.2 CC 浮层仅 direct 模式启用**;
youtube 字幕 + CC 集成留 Phase D YouTube IFrame API。

### 4.4 textarea 输入时 attr 写回**不**触发 paint

设计偏差 § 1.1 — update() transcript 路径只 recomputeCues,不 paint。**所以**:
- 用户在 textarea 编辑过程中:debounce 写 attr → update → recomputeCues(textarea 不重建,光标不跳)
- 但若**外部**(粘贴 PM doc / 撤销重做)改 transcript → textarea 不会同步显新值
  (因为本路径不重建 textarea)
- **当前用户不会撞上**:transcript 来源单一(textarea 输入)
- B3.19.3 加 fetchTranscript / B3.19.4 加翻译时,需要"外部回填 → textarea 同步"路径
  → 那时再加 update() 路径粒度优化

### 4.5 大 transcript 性能

500ms debounce + 整字符串 dispatch:每次写 attr 整字符串。1 小时视频几千行字幕仍可接受
(现代 PM transaction 几百 KB 字符串没瓶颈)。Phase D 真有性能问题再优化。
