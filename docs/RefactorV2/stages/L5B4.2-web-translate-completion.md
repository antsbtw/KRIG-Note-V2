## 4. 阶段中遇到 / 解决的问题

### 4.1 V1 WebviewElement 类型在 V2 重定义后接口不齐

V2 WebView.tsx 用了 minimal `WebviewElement` interface(不 import 'electron'),
但只声明了 loadURL/getURL/canGoBack 等基础 API。SyncDriver 需要 isLoading + executeJavaScript。

**修法**:WebView.tsx 的 WebviewElement 接口补齐 isLoading + executeJavaScript;
sync-driver 自己也声明 minimal WebviewElement interface(按需最小集)。
两个 interface 通过 structural typing 自然兼容。

### 4.2 跨 slot 通信架构选择

V1 走 main 进程 IPC(left renderer.send → main → right renderer.on),需 slot-manager
知道当前 ws 的 left/right webContents。V2 单 React tree,直接走 module-level 单例。

**决策(Q1=A)**:slot-bus 模块 — V2 简化路径,省 ~50 行 IPC 设施。

### 4.3 SyncDriver 生命周期管理

如果 isTranslateMode 状态切换(用户点翻译按钮 toggle),SyncDriver 必须 attach/detach。
否则关闭翻译后 driver 仍在 poll,浪费 CPU。

**修法**:WebView useEffect 监听 isTranslateMode:
- true:new SyncDriver + bind + subscribe + start
- false / cleanup:driver.destroy() + unsub slot-bus

### 4.4 controller/passive 切换防回环

V1 经验:页面跳转 / 长加载时控制权易错乱。用 `remoteNavUntilRef` 时间窗口
(2 秒内的 did-navigate 不回发 NAVIGATE)区分"对面触发的导航 vs 用户主动导航"。

**实施**:WebView + TranslateWebView 都有 remoteNavUntilRef,2 秒窗口对齐 V1。

### 4.5 翻译注入期间 SyncDriver 的 isBusy guard

V1 经验:翻译注入正在执行 element.js 时,不应 poll 同步事件(冲突 / 抛错)。
SyncDriver 接受 `isBusy` callback,翻译进行中跳过 poll。

**实施**:TranslateWebView 创建 SyncDriver 时传 `() => translateDriver.injecting` 作 isBusy。

## 8. 用户验证阶段踩的坑(全部修复 + 落 memory)

核心实施 commit 后用户实测,4 个 bug 暴露并修复:

### 8.1 占位符替换只替换首个匹配 → 翻译用错语言

**现场**:console "krigLang":"\__KRIG_TARGET_LANG__"(占位符未替换)
**根因**:`String.replace(string, string)` 只替换第一个匹配,inject 文件里
注释 + 变量都写了占位符,只换了注释里那处。
**修法**:`replace(/__KRIG_TARGET_LANG__/g, ...)` 全局替换
**memory**:[feedback_inject_placeholder_replace_global.md](memory)

### 8.2 slot-bus 时序竞态 → 早期消息丢失

**现场**:console "subscribe left ... pending: 3 / flush 0"
**根因**:右栏 setupWebview microtask 发 REQUEST_URL 时左栏 useEffect 还没跑,
listener 数 0 → 早版本静默丢消息,加缓冲后 TTL 1 秒过滤又把消息全过期掉
**修法**:slot-bus 加 pending 缓冲,subscribe 时 flush 全部不过滤 TTL
**memory**:[feedback_module_event_bus_subscribe_race.md](memory)

### 8.3 缺 CSP bypass → Google Translate widget 永不创建

**现场**:console "hasCtor: false / divChildren: 0" 6 秒后仍然
**根因**:disablewebsecurity 不关页面 CSP,Google Translate element.js 跑了
但加载子 chunks 时被 page CSP block
**修法**:对齐 V1 setupCSPBypass — 独立 partition `persist:webview-translate` +
main 进程对此 session 剥离 Content-Security-Policy 头
**memory**:[feedback_webview_csp_bypass_partition.md](memory)

### 8.4 right webview 初始 about:blank 把 left 冲空

**现场**:翻译成功 + 双向同步工作,但左栏白屏 URL 显 "about:blank"
**根因**:right webview 初始 src='about:blank' 加载触发 did-navigate('about:blank'),
sync driver 没守门 → takeControl + sendToOther NAVIGATE 'about:blank' →
左栏 loadURL('about:blank') 把 google news 冲掉
**修法**:三处 about:blank 守门(did-navigate 不发 / NAVIGATE 接收不接受 / URL 持久化跳过)
**memory**:[feedback_webview_about_blank_guard.md](memory)

### 8.5 切翻译语言后页面不刷新 — **L5-B4.2.2 已修复**

**现场**:用户从语言下拉切 zh-CN → ja,reload 触发了,但页面文字仍是英文(未翻译成日语)
**根因**:Google Translate widget 一旦初始化用 cookie 决定目标语言;运行时通过
`select.dispatchEvent('change')` 或写新 cookie 都不可靠刷新翻译;reload 后 widget
重建但 cookie 同步时序敏感,实测仍不稳定。

**修复路径(走过 2 条)**:

- L5-B4.2.1 — Google translate.goog URL 路径替代 widget 注入
  → 切语言可靠了,但其他维度全面降级(reCAPTCHA / 加载慢 / Google 自家服务不工作 /
  左栏被中转 URL 污染)。**已归档不 merge**,分支 `feature/L5B4.2.1-google-url-translate`
  保留作调研记录。

- **L5-B4.2.2(已采用)** — 接受 widget 注入"运行时切语言不可靠"的固有限制,改用
  产品 UX 兜底:
  - targetLang 持久化到 per-ws state(`pluginStates['web'].targetLang`)
  - 切语言 → banner 提示「需重启 app 才生效」+「立即重启」按钮
  - 重启走 `webContents.reloadIgnoringCache()` + 清 translate partition cookie 路径
    (不真重启 main 进程,避免 dev 模式 vite dev server URL 失效白屏)
  - 第一次启动用 `navigator.language` 自动选默认语言
  - 详见 [L5B4.2.2-translate-lang-persist-completion.md](./L5B4.2.2-translate-lang-persist-completion.md)

**总结**:核心架构正确(V1 直迁 + V2 简化 slot-bus),5 坑全部已修(8.5 走 L5-B4.2.2
UX 兜底路线)。**外部 SDK / 跨进程通信 / webview 生命周期**的边界 case 是这次 5 坑的
共性 — 全部落 memory 防后续重复踩。
