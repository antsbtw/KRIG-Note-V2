## 0. 背景

L5-B3.4 link mark merge 后,NoteEditor 已有 12 个 block(text-block / 6 个简单 block /
4 个 list 系列 / unknown 占位)。但**没有 image / math / table 等复杂 block**,导致 V1 → V2
迁移过程中:

- 用户从 markdown / AI 提取 / V1 笔记导入的图都丢
- L5-B4.3.1 md-to-pm 转图时输出 `image` 节点,V2 schema 不识别 → 走 unknown 占位
- L5-B4.3.2/3 AI 闭环测试无法验证图片场景

L5-B3.5 是 NoteEditor schema 完整化 **Phase A 第一步**(下一步 L5-B3.6 math、L5-B3.7 table)。

**关键效果**:image block 加进 schema 后,L5-B4.3.1 的 md-to-pm 输出的 `{ type: 'image', ... }`
**自动从 unknown 占位升级为真实 image 节点**,无需改 md-to-pm —— 印证 L5-B4.3.1 的"反向驱动 schema 补齐"路线。

## 2. NodeSpec 设计

V1 用 `content: 'textBlock'`(image 内含 caption 段落)。V2 schema 没有 `textBlock`,改用 V2 现有 `text-block`:

```ts
const imageNodeSpec: NodeSpec = {
  content: 'text-block',         // 图下方 caption(可空,V2 schema 用 text-block)
  group: 'block',
  draggable: true,
  selectable: true,
  attrs: {
    src:       { default: null },    // image URL / data: URL / media:// URL
    alt:       { default: '' },      // 替代文本(可访问性 + md 来源)
    title:     { default: '' },
    width:     { default: null },    // 像素值,resize 后写入
    height:    { default: null },
    alignment: { default: 'center' }, // 'left' | 'center' | 'right'
    // V1 还有 atomId / sourcePages / thoughtId(KRIG 知识图谱挂钩),V2 暂不实现
    //   留 attrs 默认 null 不破坏后续接入
  },
  parseDOM: [
    {
      tag: 'div.krig-image-block',
      getAttrs(node) {
        const el = node as HTMLElement;
        const img = el.querySelector('img, .krig-image-block__svg-canvas');
        return {
          src: img?.getAttribute('src') || el.getAttribute('data-src'),
          alt: img?.getAttribute('alt') || '',
          title: img?.getAttribute('title') || '',
          width: parseInt(el.getAttribute('data-width') || '0', 10) || null,
          height: parseInt(el.getAttribute('data-height') || '0', 10) || null,
          alignment: el.getAttribute('data-alignment') || 'center',
        };
      },
    },
  ],
  toDOM(node) {
    const wrapper = ['div', {
      class: 'krig-image-block',
      'data-alignment': node.attrs.alignment,
      'data-src': node.attrs.src || '',
      ...(node.attrs.width ? { 'data-width': String(node.attrs.width) } : {}),
      ...(node.attrs.height ? { 'data-height': String(node.attrs.height) } : {}),
    },
      ['img', {
        src: node.attrs.src || '',
        alt: node.attrs.alt || '',
        title: node.attrs.title || '',
      }],
      ['figcaption', { class: 'krig-image-block__caption' }, 0],
    ] as const;
    return wrapper;
  },
};
```

**关键决策**:
- `content: 'text-block'`(单 caption,可空)— 跟 V1 行为一致,允许图下方写说明
- caption 是 inline 编辑(NodeView 用 contentDOM 暴露)
- atomId / sourcePages / thoughtId 留空(将来 KRIG 知识图谱挂钩用)

## 3. NodeView 设计

V1 `imageRenderer` 通过 `createRenderBlockView` 工厂注册;V2 直接写 `NodeViewConstructor`,
跟 task-list / toggle-list 等现有 block 风格一致。

### 3.1 状态机
- **placeholder 态**(无 src):显示 `🖼` icon + Upload 按钮 + Embed link 输入
- **图片态**(有 src):显示图 + alignment toolbar + 左右 resize handles + caption

状态切换通过 `update` 返回 false 触发 PM 重建 NodeView(跟 V1 一致)。

### 3.2 Placeholder 内联 helper
不抽 render-block-base,把 V1 `createPlaceholder` 的核心逻辑内联到 image node-view 里
(~60 行)。如果未来 audio/video 需要相同模式,**再**抽公共模块。

### 3.3 SVG 增强(V1 现有,直迁)
V1 检测 `.svg` / `data:image/svg+xml` 时用 `<div>` + `innerHTML` 直接挂 SVG DOM,
而不是 `<img>` 标签 — 解决 SVG CSS 变量失效 / 字体 fallback / 内部事件等问题。

V2 直迁这套逻辑(loadSvgContent / injectSvgStyles 函数,~80 行)。

### 3.4 Resize handles
左右 2 个 handle,鼠标拖拽改 width attr(写回 PM)。V1 已实现,V2 直迁(~50 行)。

### 3.5 Alignment toolbar(可选)
V1 在选中时显示 `◁ ▣ ▷` 切对齐方式。V2 这次**简化**:只保留底层 `alignment` attr,**toolbar UI
留 L5-B3.5.1 后续做** — 因为 V2 没有 V1 风格的 RenderBlock toolbar 框架,做这个会引入
新抽象(可能跟 NoteEditor 已有 floating-toolbar 重叠)。

L5-B3.5 范围:**alignment 只通过右键菜单切**(右键菜单已在 V2 注册系统里)。
toolbar UI:留 L5-B3.5.1 评估。

## 4. mediaStore 集成

**关键场景**:
- 用户从本地上传图片 → FileReader 读 base64 → mediaPutBase64 → `media://...` URL 写 src
- 用户粘贴 base64 data URL → 同上
- 用户输入外链 → 直接写 src(不下载,跟 V1 一致)
- L5-B4.3.1 md-to-pm 已经做了 base64 → media:// 的转换,产出的 image 节点 src 已是 `media://`,
  image block 直接渲染就行

V2 image 上传逻辑通过 IPC 走 mediaStore:

```ts
// node-view 里 placeholder 的 onUpload
async function onUpload(dataUrl: string) {
  const r = await mediaPutBase64(dataUrl);
  if (r.success && r.mediaUrl) {
    updateAttrs({ src: r.mediaUrl });
  } else {
    // 失败 fallback:直接用 data URL(暂时显示,刷新后丢)
    console.warn('[image] mediaPutBase64 failed:', r.error);
    updateAttrs({ src: dataUrl });
  }
}
```

## 5. 决策点

### Q1:image content 是 'text-block' 还是 'text-block?'(可选)?

V1 用 `content: 'textBlock'`(强制有 caption 节点,可为空段落)。V2 选项:
- **A. `'text-block'`**(强制 1 个 caption)— 跟 V1 一致
- **B. `'text-block?'`**(可选 caption,可空)— 更灵活,markdown 转出来的图常无 caption
- **C. 无 caption,`atom: true`**(图就是图,没 caption)— 最简单,但失去 V1 加说明的功能

**推荐 A** — 跟 V1 一致,空 caption 不写时就是空段落,不影响 doc

### Q2:alignment 工具栏 UI

如 § 3.5 所述,V1 有 RenderBlock toolbar 风格的 alignment 切换 UI:

- **A. 不做,只走右键菜单**(L5-B3.5 范围)— V2 simpler,后续看需求
- **B. 选中图时显示 floating mini toolbar**(图正上方浮动)— 类似 V1
- **C. 右键 + handle menu 双入口**

**推荐 A** — 减少这次工作量;alignment 不是高频操作

### Q3:resize handles 是否纳入这次范围

V1 有左右 resize handles,鼠标拖拽改 width。

- **A. 这次做**(直迁 V1 ~50 行)— 完整体验
- **B. 留 L5-B3.5.1**(本阶段只静态显示,固定 100% 宽)

**推荐 A** — 已经直迁到这一步,带上不增加多少复杂度

### Q4:SVG 增强

V1 SVG 走 `<div>` + `innerHTML` 直接挂 DOM(~80 行,含 fetch / XHR fallback / styles inject)。

- **A. 直迁全套**(SVG 文件支持完整,Claude artifact 经常输出 SVG 图表)
- **B. 简化**(SVG 也用 `<img>`,可能 CSS 变量失效 / 复杂 SVG 渲染不全)
- **C. 不做 SVG 特殊处理**,留后续 sub-stage

**推荐 A** — Claude / ChatGPT artifact SVG 是常见输出,关键场景

### Q5:slash menu 注册

V2 slash menu 在 `views/note/slash-menu-content.ts`(我会查):

- **A. 这次顺便加 image slash 项**(`/image` 命令,选中后插入空 image placeholder block)
- **B. 留后续**

**推荐 A** — 一次性做完,unitary 体验

## 8. 风险

### 8.1 `content: 'text-block'` 限制

V1 image content 是 `textBlock`(V1 schema)。V2 用 `text-block` 但其有 attrs.level(heading
1-3)。如果 caption 误带 level=1,会显示成 H1 标题字号。

**预案**:NodeView createContentDOM 暴露 caption 时,**或**在 toDOM/parseDOM 强制 caption 的
text-block attrs.level=null。如果 PM 在 `text-block` 上没强制 default level=null,我们写
`content: 'text-block'` 可能允许任意 level → 写 NodeView 时 force default。

### 8.2 SVG 加载的安全风险

V1 直接 `innerHTML = svgText`,**SVG 内的 `<script>` 会执行**(同源策略限制)。
风险点:
- 用户从外部粘贴恶意 SVG → 触发 XSS
- AI 提取的 SVG 可能含 `<script>`(罕见但可能)

**预案**:加载 SVG 文本后,**走 DOMPurify** 或手动清洗 `<script>` 节点。V1 没做这个清洗,
是潜在安全漏洞。V2 这次顺便加。

### 8.3 V2 NoteEditor 现有 doc 不兼容

如果 NoteEditor 现有 doc 里没有 `image` 节点,加 image block 不会破坏现有 doc。但如果
现有 doc 有 unknown 节点 attrs.originalType='image',**这些 unknown 不会自动转 image**
(unknown 的 PMNode.fromJSON 还是 unknown)。

**预案**:接受这种 staleness — 旧 unknown 节点保留,新转出来的 markdown 才用 image 节点。
用户如果想升级,可以删掉 unknown 重新粘贴。文档明示。
