## 1. 阶段定位

L5-B4.2 baseline 实现了双屏同步翻译,中文译文 OK,但**切语言**(zh-CN → ja 等)不可靠 —
Google Translate widget 一旦初始化就用 cookie 决定语言,运行时改 cookie / dispatchEvent
都不刷新页面。L5-B4.2 完成报告 § 8.5 标"已知限制",留 L5-B4.2.1 解决。

L5-B4.2.1(URL 路径方案)调研后确认体验全面降级:reCAPTCHA 频繁触发、加载慢、Google
自家服务不工作、左栏被 widget 中转 URL 污染。**已归档不 merge**(分支
`feature/L5B4.2.1-google-url-translate` 保留作调研记录)。

L5-B4.2.2 走第三条路:**接受 widget 注入"运行时切语言不可靠"的固有限制,改用产品 UX
兜底** — 持久化 targetLang 到 per-ws state,切语言时弹 banner 提示"重启 app 才生效",
重启走 `reload webContents + 清 cookie` 路径(不真重启 main 进程,避免 dev 模式白屏)。

切语言对用户低频(大多用户只看一种语言),"重启后生效"在用户工程上完全可接受;
带来的好处是 **L5-B4.2 baseline 其他维度(加载速度、稳定性、链接行为)全保留**。

## 4. 关键设计决策回顾

### 4.1 为什么不真重启 main 进程?

- `app.relaunch + app.quit/exit` 在 dev 模式 (`npm start`) 下白屏:Vite dev server URL
  是编译期注入的全局常量(`MAIN_WINDOW_VITE_DEV_SERVER_URL`),新 instance 拿不到 →
  `loadURL(undefined)` 失败 → 白屏
- `webContents.reloadIgnoringCache()` 等效:workspace 状态从 localStorage 恢复,翻译
  widget 在新 React 树里用新 lang 重 init,用户体验跟"重启" 一致

### 4.2 为什么必须清 cookie?

- 翻译 webview partition 是 `persist:webview-translate` —— **persist:** 前缀让 cookie
  跨 reload 保留
- Google Translate widget 上次用 zh-CN 时写了 `googtrans=/auto/zh-CN` cookie
- 单纯 reload webContents 后 widget 看到旧 cookie → 直接用旧 lang,忽略我们传的新 lang
- 必须 reload 之前清 `goog*` 前缀的 cookie(只清 widget 自家,不动 NID/SID 等登录态)

### 4.3 为什么入口要统一到 WebToolbar 又在右栏顶部加 ▾?

设计文档拍板"统一到 WebToolbar"是 v0.1 决议,实施时遇到对齐问题(左栏有 36px 工具栏,
右栏没有 → 滚动时垂直错位):

- v0.1 方案:右栏顶部留空 placeholder bar 只为对齐 → C 方案
- 用户反馈选 B:右栏顶部加信息条 `自动检测 → <目标语>` + ▾ + ×

最终采用 B —— 两个 ▾ 行为完全等效(都触发 `setWebTargetLang` + 写 per-ws state),
不破坏"统一行为入口"原则,反而**右栏的 ▾ 在用户视野里更自然**(用户在看右栏时,直接在
那里改语言,不用回到左栏)。

### 4.4 webviewTick 时序修复

reload 后 `slotBinding.right` 是 `'web-translate-view'` → `isTranslateMode = true`。
React 渲染顺序:
1. WebView mount,`isTranslateMode = true`,`webviewRef.current = null`(ref 未绑)
2. SyncDriver useEffect 跑 → `wv` 是 null → 直接 return → driver 永远不创建

加 `webviewTick` state,setupWebview 里 ref bind 时 `setWebviewTick(v => v + 1)`,
SyncDriver useEffect deps 加 `webviewTick` → ref 绑了之后 effect 重跑 → driver 创建。

## 5. 用户验证阶段踩的坑

### 5.1 第一次"重启"白屏

用 `app.relaunch + app.exit(0)` → dev 模式下 vite dev server URL 不传给新 instance →
新 instance 加载失败 → 白屏。改成 reload webContents 解决。

### 5.2 重启后语言不切换

reload 但没清 cookie → widget 看到旧 `googtrans` cookie → 不用新 lang。
加 `clearTranslateCookies` 清 `goog*` 前缀 cookie 解决。

### 5.3 重启后同步滚动消失

reload 后 SyncDriver useEffect 在 ref 绑之前跑 → driver 不创建。
加 `webviewTick` bump + deps 重跑解决。

### 5.4 第二次白屏(更严重)

driver 创建后调 `driver.start()` → `injectSyncScript` 调 `executeJavaScript` → webview
还没 attached → **同步 throw**(不是返回 rejected promise)→ React 顶层 crash →
整个 WebView 组件 unmount → 全屏黑。
所有 executeJavaScript 调用包 try/catch 解决。

**这次教训:外部 SDK 的"同步 throw vs 返回 rejected promise"必须实测,假设错了导致
反复白屏。** 已落 memory:`feedback_external_sdk_lifecycle.md`(L5-B4.2 已记 Electron
webview 生命周期门道,本次补充"executeJavaScript 在未就绪时同步 throw"的细节)。
