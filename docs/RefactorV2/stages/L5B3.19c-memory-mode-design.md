## 0. 上下文

### 0.1 V1 Memory Mode 算法

```typescript
function* memoryPlaybackSequence(totalSegments: number): Generator<MemoryStep> {
  if (totalSegments <= 0) return;
  yield { type: 'play', segments: [0] };
  for (let i = 1; i < totalSegments; i++) {
    yield { type: 'repeat', segments: [i - 1, i] };
  }
}
```

序列:`[0]` → `[0,1]` → `[1,2]` → `[2,3]` ... `[n-2,n-1]`

每 step:
- 取 segments[0] 作起始,seekTo + play
- 监听 currentTime,触达 (lastSeg+1)*segDuration - 0.3s 时调 advanceMemoryStep
- advanceMemoryStep → next step;到 generator end → stopMemoryMode

prevMemoryStep:
- 重建 generator 跑到 targetStep - 1,然后 advance
- (V1 实现略冗余,但稳定)

### 0.2 a 段就位的依赖

- `time-tracker.onTimeUpdate(cb)`:订阅 currentTime;返回 unsub
- `time-tracker.onDurationReady(cb)`:订阅 duration 解析(YouTube 异步推送 / video metadata)
- `time-tracker.seekTo(seconds)`:跳转
- `time-tracker.play()`:播放(YouTube postMessage / video.play)
- `time-tracker.getDuration()`:同步取 duration

## 2. 模块边界 + 关键 API

### 2.1 actions/memory-button.ts

```typescript
export interface MemoryControl {
  /** 整体挂点 — 包含 ⏮ + 🧠 + dropdown + ⏭ 一组 wrap div */
  el: HTMLElement;
  /** 是否激活(为 true 时 ⏮/⏭ enabled,🧠 文字含进度)*/
  isActive(): boolean;
  /** 主动停止(node-view 在某些场景调,如 src 切换时)*/
  stop(): void;
  destroy(): void;
}

export function createMemoryButton(deps: {
  /** 取当前 segmentDuration(从 attrs)*/
  getSegmentDuration: () => number;
  /** 取 time-tracker(可能为 null,未 mount 时按钮 disabled)*/
  getTracker: () => TimeTracker | null;
  /** 取上次 memoryLastStep(初次从 attrs 恢复)*/
  getLastStep: () => number;
  /** state 写回 attrs(node-view 协调持久化)*/
  onUpdateAttrs: (patch: { segmentDuration?: number; memoryLastStep?: number }) => void;
}): MemoryControl;
```

**内部状态**:
```typescript
let active = false;
let generator: Generator<MemoryStep> | null = null;
let currentStep: MemoryStep | null = null;
let stepIndex = 0;
let timeUnsub: (() => void) | null = null;
```

**生命周期**:
- 点 🧠(未激活时)→ 显 dropdown(30/60/90/120s)
- 点 dropdown 项 → segmentDuration 写回 + start
  - `start()`:取 duration → 重建 generator → 跑到 lastStep → advance → 订阅 onTimeUpdate(段边界检测)
- 点 🧠(已激活)→ stop
  - `stop()`:active=false / unsubscribe / 写回 lastStep
- 点 ⏮ → prev step(重建 generator 到 prev,advance)
- 点 ⏭ → advance(直接 next)
- 段边界达到 → 自动 advance

**段边界检测**(订阅 onTimeUpdate):
```typescript
const boundary = (currentStep.segments[currentStep.segments.length - 1] + 1) * segmentDuration;
if (currentTime >= boundary - 0.3) advance();
```

0.3 秒 buffer 是 V1 实测的容错值(避免 300ms 轮询恰好错过)。

**🧠 文字进度**:`🧠 ${stepIndex}/${total}` 其中 `total = Math.ceil(duration / segmentDuration)`。
duration 异步可用(YouTube 启动后才推);**未就绪时显 🧠 等待**。订阅 `onDurationReady` 触发首次 label 更新。

## 6. 风险

### 6.1 generator 重建代价
prev / start 时重建 generator 跑到 targetStep。V1 数百段视频实测无感(generator 是同步循环)。**接受**。

### 6.2 0.3s buffer 跟 300ms 轮询冲突
轮询周期 = 0.3s,buffer = 0.3s。理论上 currentTime 触发一次 advance 后,下个轮询 currentTime 又跨过新边界。
但 advance 内 seekTo 会把 currentTime 拉回段起始 — 触发 boundary check 失败(currentTime < boundary)。**自然不连续触发**。

### 6.3 active state 没持久化
仅 segmentDuration + memoryLastStep 持久化。重启后**默认未激活**,用户需重新点 🧠 启动。这跟 V1 一致 — 简化 + 避免"打开笔记自动播视频"困扰。
