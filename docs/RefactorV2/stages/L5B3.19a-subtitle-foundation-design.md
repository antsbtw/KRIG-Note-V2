### 2.2 helpers/time-tracker.ts(新引入,本段核心)

```typescript
export interface TimeTracker {
  /** 订阅 currentTime 变化(每 ~300ms 触发,值是当前播放位置秒数)*/
  onTimeUpdate(cb: (time: number) => void): () => void;
  /** 订阅 duration 解析完成(YouTube 异步推送,video metadata 加载)*/
  onDurationReady(cb: (duration: number) => void): () => void;
  /** 同步取当前时间(0 if not ready)*/
  getCurrentTime(): number;
  /** 同步取总时长(0 if not ready)*/
  getDuration(): number;
  /** 跳转(YouTube postMessage / video.currentTime=)*/
  seekTo(seconds: number): void;
  /** 播放(YouTube postMessage / video.play)*/
  play(): void;
  /** 销毁:卸载 listener / 停轮询 */
  destroy(): void;
}

/** 工厂:videoEl 已有时建 video tracker;否则建 youtube tracker(收 iframe ref)*/
export function createTimeTracker(
  source: { videoEl: HTMLVideoElement } | { iframe: HTMLIFrameElement },
): TimeTracker;
```

**实现要点**:
- **单源 300ms 轮询**:tracker 内部一个 `setInterval` 轮询 `getCurrentTime()`,所有订阅者共享(对齐 V1 单 polling 驱动 CC + vocab + memory 模式)
- **YouTube tracker**:启动时 postMessage `{event:'listening'}`;监听 `infoDelivery` 更 `currentTime` / `duration`;`seekTo`/`play` postMessage 命令
- **Video tracker**:直接读 `videoEl.currentTime`;`play()` 调 `videoEl.play()`;`seekTo` 写 `videoEl.currentTime = seconds`
- **零订阅时停轮询**:所有订阅都 unsubscribe 后 `clearInterval`,新订阅恢复(避免空转)

### 2.9 spec.ts

```typescript
attrs: {
  src: { default: null },
  embedType: { default: null },
  title: { default: 'Video' },
  mimeType: { default: null },
  duration: { default: null },
  atomId: { default: null },
  // L5-B3.19.a NEW:
  activeTab: { default: 'play' },          // 'play' | 'data' | 'transcript' | <future translation lang>

}
```

`transcriptText` 持久化策略(Q-a-2,P1 修正):
- **真相源 = transcriptText 原文字符串**(用户在 textarea 编辑的内容)
- subtitleCues 仅作内存派生值,**不持久化**(每次 mount 时由 transcriptText 调 parseSubtitleCuesFromText 重 parse — 一次性 ~ms 级开销可接受)
- transcriptTab textarea input 时,节流 500ms 写 attrs.transcriptText(不写 cues)
- mount 时,attrs.transcriptText 非空则填回 textarea + 内存 parse 一份 cues 给 overlay/订阅者
- **优点**:保留所有原文信息(注释 / 空行 / 不规范行 / 时间格式偏好);如果 parser 升级(支持新格式),老笔记不需迁移
- **缺点**:长 transcript 每次 mount 多 ~10ms parse 时间(可接受)

## 6. 风险

### 6.1 enablejsapi=1 跟现有 B3.16 行为差异
开 enablejsapi 后 YouTube iframe 会 postMessage 推 infoDelivery。如果 V2 其他地方监听 window.message 误中,可能出 bug。
**应对**:tracker 内部严格校验 `event.source === iframe.contentWindow`(对齐 V1,已验证)。

### 6.2 Tab 切换时 player iframe / video 重 mount?
切到 Data Tab 时 play-tab 隐藏(`display: none`),不 unmount。否则 YouTube 重新加载视频(浪费流量 + 失记忆位置)。
**约定**:四个 Tab panel 的 `display: none` 切换,不 detach DOM。time-tracker 不停(继续 poll currentTime,但订阅都不消费时无副作用)。

### 6.3 transcriptText 单一真相源(P1 修正后)
~~原 v0.1 的 attrs.subtitleCues 缓存 + textarea 反向重建 方案丢失原文细节(注释 / 空行 / 时间格式偏好)~~ — 已废弃。
**最终约定**(P1 修正):
- `attrs.transcriptText` = **唯一持久化真相源**(原文字符串)
- subtitleCues = 内存派生值,**不持久化**(每次 mount 由 transcriptText 重 parse)
- textarea 显示 = `attrs.transcriptText`
- 用户编辑 textarea → 节流 500ms 写 `attrs.transcriptText`(不直接写 cues)→ 内存 parse 一份新 cues 推订阅
- 优点:零数据丢失;parser 升级后老笔记自动受益
- 取舍:每次 mount ~10ms parse 开销(长 transcript 几千行实测 < 50ms,可接受)

### 6.4 Vimeo 占位 vs 总设计 Q4
总设计 Q4=A "不回 Vimeo / generic"。本段加 vimeo/generic 检测但**不渲染** — 是"为 Phase D 留位",符合 Q4 精神(不影响功能)。
