## 1. NodeSpec 设计

V1 同款,attrs 改造:**noteId default null**(V1 是 `{}` 必填,V2 改成可空容错)。

```ts
const noteLinkNodeSpec: NodeSpec = {
  inline: true,
  group: 'inline',
  atom: true,
  attrs: {
    noteId: { default: null },   // V2 note-store 中的 note id(如 'note-3')
    label: { default: '' },      // 显示文本(目标笔记 title 派生,可手动指定)
  },
  parseDOM: [{
    tag: 'span.krig-note-link',
    getAttrs(dom) {
      const el = dom as HTMLElement;
      return {
        noteId: el.getAttribute('data-note-id') || null,
        label: el.textContent?.replace(/^📄\s*/, '') || '',
      };
    },
  }],
  toDOM(node) {
    return [
      'span',
      { class: 'krig-note-link', 'data-note-id': (node.attrs.noteId as string) ?? '' },
      `📄 ${(node.attrs.label as string) || 'Untitled'}`,
    ];
  },
  // atom 节点 — leafText 让复制 / textBetween 还原源码 [[label]]
  leafText: (node) => `[[${(node.attrs.label as string) || ''}]]`,
};
```

**关键点**:
- `inline + atom + group: 'inline'` — 行内 atom,光标不进入
- `leafText`:复制时还原成 `[[label]]`(对齐 mathInline 模式)— V1 没有这个,V2 加上

## 2. NodeView 设计

V1 行为:
- 渲染 `📄 <label>`(点击高亮)
- 异步 noteLoad 验证目标存在 + 派生最新 title 同步到 attrs.label
- 目标不存在 → 红色 "未找到"
- 点击 → noteOpenInEditor(切当前 ws activeNoteId)

V2 改造:
- noteLoad → `noteStore.get(noteId)`(同步函数)
- 派生最新 title → 监听 noteStore subscribe,目标 title 变化时 setNodeAttribute label
- noteOpenInEditor → 用 link-click 协议路由(已有 krig://note/<id> 路径),或者直接调
  `setActiveNote(workspaceId, noteId)`
- contenteditable=false + 点击高亮 / hover

## 5. 决策点

### Q1:noteId 持久化策略

V1 noteId 是 SurrealDB record id(永久)。V2 noteStore 的 id 是 `note-<counter>`(简单递增)。

- **A. 直接用 noteStore.id**(简单,note 删除后 link 失效自动显"未找到")
- **B. 双重 fallback**:noteId + label,noteStore 删除后用 label 模糊匹配(可能误匹配)

**推荐 A** — 跟 V1 行为一致;失效用红色"未找到"明示

### Q2:title 同步策略

V1 NodeView mount 时异步 noteLoad → 自动同步最新 title 到 attrs.label。
V2 noteStore 是 React subscribe 模式,可以做实时同步:

- **A. 仅 mount 时同步一次**(对齐 V1 行为):简单,但用户改了被引用 note 的 title,
  正在打开的 note 内的 link 不刷新(直到再次打开)
- **B. 实时 subscribe 同步**:noteStore 变化 → NodeView update 检查目标 title 是否变化
  → setNodeAttribute label;实时,但每次 store 变化都遍历所有 noteLink

**推荐 A** — 实时 subscribe 工作量大且性能边界要小心;V1 行为已能满足 99% 场景

### Q3:点击 note-link 行为

V1:`viewAPI.noteOpenInEditor(noteId)` → 切当前 NoteView activeNoteId(覆盖左栏当前)。

V2 选项:
- **A. 同 V1**:setActiveNote(workspaceId, noteId)— 当前 ws 切到目标 note(覆盖)
- **B. 新 ws 打开**:noteStore.openInNewWorkspace(noteId)— 不覆盖,保留来源 note
- **C. krig://note/<id> 协议**:走 link-click-integration 已有路由(实际跟 A 等价)

**推荐 C** — V2 已有 krig:// 协议路由,统一入口跟链接行为一致;link-click 也复用

### Q4:slash menu 是否加 `/note-link` 项?

V1 设计 `slashMenu: null`(只 `[[` 触发)。

- **A. 跟 V1 一致**,只 `[[` 触发(slash 菜单不显示)
- **B. 也加 slash `/note-link`**:某些用户可能习惯 slash

**推荐 A** — `[[` 是 markdown / Roam / Obsidian 标准触发,熟悉用户优先用它

### Q5:V2 noteStore 标题派生时机

V2 创建 note 时 `Note.title` 是独立字段(从 NavSide 命名 / 或派生 doc 第一段);
L5-B3.11 改成 doc 首块 isTitle text-block。**搜索面板显示什么 title?**

- 用 `Note.title` 字段(noteStore 内的)— 这字段在 L5-B3.11 后由 deriveTitle 维护,
  来源是 doc.firstChild.textContent
- 显示 "Untitled" 当空(对齐空 title 的 placeholder)

**已确认** — 用 `Note.title` 字段,空时显 "Untitled"

## 8. 风险

### 8.1 `[[` 跟 markdown 输入规则冲突

V2 input-rules 没有 `[[ ` 规则(LinkPanel 用 `Cmd+K`),不冲突。但 markdown 标准的
`[text](url)` link 用 `[`,**当用户输 `[a]` 时不应该误触发** — V1 plugin 严格判断
连续两个 `[`(textBefore.endsWith('[[')),已防止。

### 8.2 noteId 失效

note 被删除 → noteLink 显"未找到"。**风险**:用户复制粘贴 note 到另一个 workspace,
另一个 ws 没这个 note id → 也显未找到。**预案**:V2 单 React tree,所有 ws 共享 noteStore,
note 不会因为切 ws 失效;用户主动删 note 才会失效,符合预期。

### 8.3 popup 跟 slash menu 同时打开冲突

`[[` 在 slash menu 打开时被输入(用户先按 `/` 再输入 `[[`)— 两个面板可能同时显示。

**预案**:note-link-command plugin 触发 popup 时,popupController 自动关其他 popup
(popup-controller 单例只一个 active);slash menu 是 controller 模式,popup show 时
也要主动 hide slash(L5-B3.11 加 popup 联动 slash hide,本阶段顺手做)。
