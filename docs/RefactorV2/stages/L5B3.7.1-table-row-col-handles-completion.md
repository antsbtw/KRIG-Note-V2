## 4. 关键设计

### 4.1 三类 handle 的几何来源(M2 全用 DOM 量,M3 可考虑 ResizeObserver / PluginView)

| handle | 几何来源 | 触发重算 |
|---|---|---|
| 列 dot | 第一行各 cell `getBoundingClientRect()`;dot 横跨 cell 宽 | NodeView `update()` 回调(行/列/colwidth 变都触发)+ 初始 rAF |
| 行 dot | tbody 直接子 tr `getBoundingClientRect()`;dot 纵跨 row 高 | 同上 |
| ⋯ handle | `.selectedCell` DOM 集合 bounding rect 并集;居中浮在上沿 | 监听 view.dom 'mouseup' / 'mousedown' / 'keyup' + rAF 防抖 |

**update() 内重算延后一帧**:colgroup `style.width` 由 columnResizing 写入,
跟 update 不同步;延后一帧让 layout 稳定后再量。

### 4.2 popup context 通信(menu-context 单例)

popupController 不带额外参数,而 TableMenuPanel 要知道"操作哪一行/列/选区"。
解决方案:`menu-context.ts` 模块级单实例,node-view 点击 handle 时:

```ts
setTableMenuContext({ instanceId, tablePos, scope, rowIdx?, colIdx? });
popupController.show('text-editing.popup.table-menu', dotEl);
```

popupController 全局**只允一个 popup**,所以单实例不会撞。TableMenuPanel unmount 时
useEffect cleanup 调 `clearTableMenuContext()` 防残留。

### 4.3 PM 命令分发:setSelection → command(state, dispatch)

prosemirror-tables 库的 `deleteRow` / `addColumnAfter` 等都基于当前 selection 决定
操作哪行/列(`selectedRect(state)`)。而 handle 知道用户点的具体 row/col,
所以策略是:**先 setSelection 到目标 (rowIdx, colIdx) cell 内**,再调命令。

例外:列对齐(整列 align)— 走 CellSelection 路径才能作用整列。
TableMenuPanel `align` 函数构造 `new CellSelection($anchor, $head)`,
anchor = col 顶 cell,head = col 底 cell。

### 4.4 NodeView destroy 清理

view.dom 上挂的 mouseup / keyup / mousedown 监听必须 destroy 清掉,
否则 view 重建(切笔记 / 重启)会泄漏多次绑定。`requestAnimationFrame` 的 id 也 cancel。

## 6. 风险点 / 已知坑

| 风险 | 当前处理 | 后续建议 |
|---|---|---|
| view dispatch 不通过 NodeView 时,⋯ handle 显隐可能滞后(纯键盘选区 — 罕见)| mouseup/keyup/mousedown 兜底 | M3 plugin 化(PluginView dispatch hook)|
| colspan>1 / rowspan>1 时 dot 数 = 第一行可见 cell 数(不是 logical 列数)| M2 简化:用户从 dot 操作"该 cell 所在列",符合直觉 | M3 用 colCount 重算 logical 列 |
