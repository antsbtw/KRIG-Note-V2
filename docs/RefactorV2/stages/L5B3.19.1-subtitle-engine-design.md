### 1.3 出口契约

```ts
// src/drivers/text-editing-driver/blocks/video-block/subtitles.ts
export interface SubtitleCue {
  startTime: number;  // 秒
  text: string;
}

/**
 * 解析 transcript 文本为字幕 cues
 * 支持格式:
 *   [MM:SS] text
 *   [HH:MM:SS] text
 * 不匹配的行静默忽略(用户可在 textarea 写注释)
 */
export function parseSubtitleCuesFromText(text: string): SubtitleCue[];

/**
 * 找当前时间对应的 active cue(线性扫,O(n))
 * 返回 startTime <= currentTime 的最后一条
 */
export function findActiveCue(cues: SubtitleCue[], currentTime: number): SubtitleCue | null;
```

## 2. 模块设计

### 2.1 字幕引擎(纯 utility)

`src/drivers/text-editing-driver/blocks/video-block/subtitles.ts`(NEW ~30 行):

V1 直迁 — `parseSubtitleCuesFromText` + `findActiveCue` + `SubtitleCue` interface。

**纯函数 + 0 副作用 + 无外部依赖**(本段最干净的 utility,Phase D 抽到独立
capability 也无成本)。

### 2.2 videoBlock schema 加 attr

```ts
// blocks/video-block/spec.ts attrs:
transcript: { default: '' },   // [MM:SS] text\n[MM:SS] text\n... 多行字符串
```

parseDOM/toDOM 加 data-transcript(serializer hint,跟 downloadedVideoPath 同模式)。

**为什么持久化为字符串而非 cues 数组?**
- 用户可能直接在 textarea 编辑(改 cue 文本 / 时间)— 字符串源是真理之源,cues 是 derived
- PM JSON 序列化字符串简单;数组要约束 type
- B3.19.4 加翻译时,翻译产物也持久化为同格式字符串(每语言一个 attr,如 `transcriptZh`)
  — 本段不做,但 schema 设计预留这个心智

### 2.3 NodeView transcript textarea(direct + youtube 共用)

`buildDirectVideo(n)` 与 `buildYouTubeEmbed(n)` 内 video/iframe 元素下方都加 transcript 区
(Q3=A 决策一致行为)。**抽公共辅助** `appendTranscriptArea(playerWrap, n)`,在两个分支
末尾调用一次,避免代码重复。

```
+---- video player(<video controls> 或 youtube <iframe>)----+
|                                                            |
|                                                            |
+------------------------------------------------------------+
|  Transcript:  [Show ▾]                                    |   ← 折叠按钮(default 折叠)
+------------------------------------------------------------+
|  <textarea>                                                |   ← 展开后用户编辑 transcript
|  [00:00] Hello world                                       |
|  [00:05] How are you                                       |
+------------------------------------------------------------+
```

**direct vs youtube 差异**:本段都支持编辑 transcript,**唯一**区别在 B3.19.2 加 CC 浮层时:
- direct video:有 `<video>.currentTime`,可挂 `timeupdate` 监听 → CC 实时同步
- youtube iframe:跨 origin 拿不到 currentTime(YouTube IFrame API 留 Phase D),
  CC 浮层 B3.19.2 只在 direct 模式启用;youtube 模式仍可编辑 transcript 但无 CC

本段不涉及 CC 浮层,**两个模式行为一致**(都能编辑 transcript)。

**关键交互**:
- transcript 区 default **折叠**(只显 `Transcript [Show ▾]` 按钮,不占空间)
- 点击 → 展开 textarea(monospace 字体,支持滚动)
- textarea 内输入触发 debounce 500ms → 写入 attrs.transcript(避免每个字符 dispatch)
- attrs.transcript 变化 → cues 缓存重算(internal,不持久化)

**非目标**:
- ❌ CC 浮层(B3.19.2 — 视频上方实时显字幕)
- ❌ 字幕跟 video.currentTime 同步(B3.19.2)
- ❌ Fetch 按钮抓 YouTube transcript(B3.19.3)
- ❌ Translate 按钮(B3.19.4)

本段**只是给用户一个能力:可以手动粘贴 / 编辑 transcript 文本**。能解析,但还没"显示出来"
(那是 B3.19.2)。这种"先存数据,再做 UI" 的渐进路径让用户提前能用(粘贴文本到笔记
作为知识收藏,即使没 CC 浮层)。

## 3. 决策点

### Q1:transcript 文本格式严格度

V1 接受 `[MM:SS]` 或 `[HH:MM:SS]`,不匹配行**静默忽略**。

- **A. 同 V1**:不匹配行忽略(用户可写注释行)
- **B. 严格校验**:不匹配行红色提示

**推荐 A**(对齐 V1 + 友好)。

### Q2:transcript 区默认折叠 vs 展开

- **A. 默认折叠**(显 `Transcript [Show ▾]` 按钮)
- **B. 默认展开**:用户立即看到 transcript 编辑区

**推荐 A**(避免界面过载;用户主动展开)。**折叠状态本身不持久化**(每次开
都默认折叠 — 避免用户每个 video block 都看到大块字幕区)。

### Q3:youtube embed 模式是否支持 transcript

V1 youtube embed 也有 transcript 区(为后续 B3.19.3 自动抓做铺垫)。

- **A. 同 V1**:youtube + direct 都支持 transcript
- **B. 仅 direct**:youtube 留 B3.19.2/3(YouTube IFrame API 整合时)

**推荐 A**(用户也想给 YouTube 视频粘贴 transcript 文本,本段就够用 — 即使没 CC 浮层也能手动看)。
**但**:YouTube iframe 内的 video 元素 V2 拿不到(跨 origin),所以 youtube 模式
本段没有 currentTime 同步 — 没关系,本段不做 CC 浮层。

### Q4:transcript 持久化字段名

- **A. `transcript`**:简单
- **B. `transcriptText`**:更明确(避免跟未来 transcript 数组冲突)

**推荐 A**(简单,跟 V1 命名习惯一致;cues 数组是 derived,不进 attrs)。

### Q5:textarea debounce 间隔

- **A. 500ms**
- **B. 1000ms**(更长,减少 PM dispatch 频率)
- **C. 不 debounce**(每个字符 dispatch — 性能差)

**推荐 A**(500ms 兼顾响应感和频率)。

## 6. 风险

### 6.1 [低] textarea 在 NodeView 内部交互冲突

PM contentEditable=false 区域内嵌 textarea(用户输入要绕过 PM)。stopEvent 已经
拦 .krig-video-block__transcript-area 类目的事件(参考 audio/video block 现有
stopEvent 处理)。

### 6.2 [低] transcript 字段持久化大小

用户粘贴超长 transcript(1 小时视频,几千行)→ PM doc 增大几十 KB。
**预案**:本段不限制(localStorage 容量足够);Phase D 真有性能问题再加压缩或外部存储。

### 6.3 [低] debounce 边界

textarea 输入到 500ms 间隔内切笔记 → 最后一次输入未 dispatch → 数据丢失。
**预案**:NodeView destroy 时强制 flush pending dispatch(代码加几行)。
