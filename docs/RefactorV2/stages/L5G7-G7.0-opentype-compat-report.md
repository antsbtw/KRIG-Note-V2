# G7.0 — opentype.js@1.3.4 系统字体兼容性清单(本机 Mac 实测)

> 2026-06-20 · 执行人实测 · 开工 G7.1 前置(设计 §11,总指挥审计补)
> 验证脚本(用完即弃,跑完已删):`scripts/_throwaway_g7_0_*.mjs`
> 铁律:别猜、看真实数据 — 本清单全是本机真实系统字体跑出来的,非读代码脑补。

---

## 0. 一句话结论

**可以做,且能覆盖 CJK。** opentype.js@1.3.4 对**单字体 .ttf/.otf 兼容率 99.9% / 99.5%**;唯一硬伤是 **`.ttc`(TrueType Collection)直接 parse 100% 失败**(`Unsupported OpenType signature ttcf`)——而 **Mac 所有主力中文字体(苹方/黑体/宋体/华文/Hiragino)全是 .ttc**。

**但 .ttc 可救**:写了一段 **纯 JS、零新依赖** 的 ttc 拆解 shim(从 collection 头切出第 N 个子字体重组成独立 sfnt),实测 **47/49 个 .ttc 拆解后 parse 成功**,**苹方 PingFang SC / 黑体 Heiti SC / Hiragino Sans GB 全部 `getPath('测')` 出真实字形**(苹方 217 path 命令)。

→ **不需要换库 / 补 fontkit。** opentype.js@1.3.4 + ttc shim 足够覆盖首版。

## 3. .ttc 拆解 shim 实测(救 CJK 的关键)

写了纯 JS ttc 拆解(`extractSfntFromTtc(buf, fontIndex)`):读 TTCHeader → 定位第 N 个子字体的 sfnt 表目录 → 因 table record 的 offset 是**整文件绝对偏移**,重建一个独立 sfnt buffer(新头 + 重定位表目录 + 拷各表数据)→ 喂 `opentype.parse`。

**`/System/Library/Fonts/*.ttc`(49 个)拆解结果:**

| 指标 | 数量 |
|---|---|
| 拆解→parse 成功 | **47/49 (95.9%)** |
| 失败 | 2/49(ADTNumeric:递归栈溢出;Apple Color Emoji:位图无 outline,getPath 空——无关画板文字) |

**重点 CJK 字体逐一验证(全部含真实「测」字形 + getPath 出路径):**

| 字体 | numFonts | 抽出的 family | cjk(测) | getPath |
|---|---|---|---|---|
| **PingFang.ttc(苹方,AssetsV2)** | 24 | PingFang SC / TC / HK / MO | ✅ | ✅ 217 cmds |
| **STHeiti Light.ttc(黑体)** | 2 | Heiti TC / Heiti SC | ✅ | ✅ 525 cmds |
| **Hiragino Sans GB.ttc** | 4 | Hiragino Sans GB W3 / W6 | ✅ | ✅ 254 cmds |

→ **苹方简体(macOS 默认中文)完全可嵌可渲染。** 这是墙 3「可移植」对 CJK 用户成立的硬证据。

## 4. 对 G7.1 的直接结论(supported 判定范围)

| 格式 | 首版判定 | 处理 |
|---|---|---|
| `.ttf` 单字体 | **supported** | 直 `opentype.parse` |
| `.otf` 单字体 | **supported** | 直 `opentype.parse` |
| `.ttc` 集合 | **supported(经 shim)** | ttc 拆解 shim 抽子字体 → parse;扫描需展开为 `{family, style, path, fontIndex}` 多条 |
| 个别异常(format-13 cmap / 栈溢出 / 无 outline 位图) | **unsupported,fail loud** | `embed` 时 `console.warn` 明确跳过 + UI 提示,不静默崩(红线) |
| 可变字体 / `.dfont` | 首版 unsupported(本机样本里未见明显占比;.dfont 已基本绝迹) | 同上 fail loud |

**无需引入 fontkit / subset-font 解析层。** opentype.js@1.3.4 + 纯 JS ttc shim 即可。

## 5. ⚠️ 由 G7.0 暴露、设计未覆盖的偏差(待总指挥确认)

设计 §10/G7-4 把 `.ttc` 列为"首版 unsupported,fail loud"。实测表明 .ttc **是 CJK 唯一格式**,标 unsupported = 中文系统字体全军覆没,违背"可移植对中文用户成立"。我据实测主张 **首版即支持 .ttc(经 shim)**,但这带来一个设计模型偏差:

**偏差 D1:扫描 / 嵌入的标识从 `path` 变成 `{path, fontIndex}`。**
- 一个 .ttc 含 N 个子字体(苹方 24 项 / 去重后 4 个 family)。选字体、读二进制、嵌入都要带 `fontIndex`。
- 影响面:G7.1 扫描产物结构、G7.2 `embed(path, fontIndex)`、font-store 落盘需把"抽出的单 sfnt"存为独立文件(而非原 .ttc),fontId 的 SHA256 应基于**抽出的 sfnt buffer**(这样去重粒度对、嵌入体积也只含单子字体而非整个 collection)。
- 好处:嵌入体积天然变小(只嵌 collection 里选中的那一个子字体,不是整包),对 8MB 守卫有利。

**我的处置建议**:按实测推进——首版支持 .ttc via shim,标识用 `{path, fontIndex}`,font-store 存"抽出的单 sfnt"。这是对设计的**增强而非缩减**(覆盖面更广、体积更小),不破任何红线(仍纯主进程读取、仍 opentype 渲染、仍 embed: 前缀)。

→ **若总指挥认可,我据此开 G7.1;若要严格按设计 v0.2 首版只做 .ttf/.otf(CJK 全部 fail loud 留 backlog),也照办。** 不默默偏离,故此记录待确认。

**✅ 已拍板(2026-06-20,用户/总指挥):采纳 D1 —— 首版即支持 .ttc(经 shim),标识 `{path, fontIndex}`,font-store 存抽出的单 sfnt。** 设计 §10/G7-4 关于 .ttc 的 "首版 unsupported" 条以本实测结论为准更新为 supported。据此推进 G7.1。
