## 0. 背景

V1 → V2 迁移盘点(`v1-note-migration-audit.md`)Phase B2:**paste-media**。

V1 实现:`src/plugins/note/plugins/paste-media.ts` 89 行,handlePaste 检测剪贴板图片
→ 智能插入 image block。

**V1 设计要点(直接抄):**
1. 检测 `clipboardData.items` 中第一个 `image/*` 文件
2. **HTML fallback 让步**:剪贴板同时含 `<table>` / `<tr>` / `<thead>` / `<th>` /
   `<td>` / `<h1-6>` 时**让步**给后续 handler(Word / Excel 通常会附带 PNG fallback,
   但真正想要的是结构化 HTML)
3. FileReader → dataUrl → 创建 image node(带 caption textBlock)
4. 智能位置:用 `$from.depth` 拿当前 block,空 textBlock 替换 / 否则插入之后

**V2 改动:**
- V2 image schema 已落地(L5-B3.5),节点名 `image`,caption 子是 `text-block`
- V2 mediaStore 已落地,粘贴的 dataUrl 走 `mediaPutBase64` → `media://` URL,
  比 V1 直接存 dataUrl 更优(刷新不丢 + 不污染 doc)
- V2 typed schema:nodeFromJSON 改成 `schema.nodes.image.create({ src }, captionNode)`
- 不集成 smart-paste / handler dispatcher(留 Phase E)— 仅做基础图片粘贴

## 2. 核心实现

```ts
// src/drivers/text-editing-driver/plugins/build-paste-media-plugin.ts
import { Plugin } from 'prosemirror-state';
import { mediaPutBase64 } from '@storage/media-store';

export function buildPasteMediaPlugin(): Plugin {
  return new Plugin({
    props: {
      handlePaste(view, event) {
        const cd = event.clipboardData;
        const items = cd?.items;
        if (!cd || !items) return false;

        // 1. 找第一个 image/* 文件
        let imageFile: File | null = null;
        for (let i = 0; i < items.length; i++) {
          if (items[i].type.startsWith('image/')) {
            imageFile = items[i].getAsFile();
            if (imageFile) break;
          }
        }
        if (!imageFile) return false;

        // 2. HTML 含结构化标签 → 让步给后续 handler(本阶段后续是 PM 默认粘贴,
        //    但保留这条规则,Phase E smart-paste 接入时无需改本插件)
        const html = cd.getData('text/html') || '';
        if (/<\s*(table|thead|tbody|tr|th|td|h[1-6])\b/i.test(html)) {
          return false;
        }

        // 3. schema 校验
        const { schema } = view.state;
        const imageType = schema.nodes.image;
        const textBlockType = schema.nodes['text-block'];
        if (!imageType || !textBlockType) return false;

        // 4. FileReader → dataUrl → mediaStore → media:// URL
        //    异步路径:dispatch 时用 view.state(可能已变化,setNodeAttribute
        //    安全;但本阶段图简单,先 dataUrl 占位 mount,异步落 mediaStore 后
        //    更新 src — 跟 image NodeView upload 路径一致)
        const reader = new FileReader();
        reader.onload = async () => {
          if (view.isDestroyed) return;
          const dataUrl = reader.result as string;
          // 同步路径:先用 dataUrl 占位插入(用户立即看到图)
          const captionNode = textBlockType.create();
          const imageNode = imageType.create({ src: dataUrl }, captionNode);
          insertImageAtCursor(view, imageNode);

          // 异步路径:落 mediaStore → 拿到 media:// URL → 找到刚插入的 image
          // 节点 setNodeAttribute(对齐 image NodeView upload 行为)
          //
          // 实现策略:用 newImageNode 的 PM pos(insertImageAtCursor 返回)+
          // 异步 update src:'media://...'
          //
          // 简化版:直接调 mediaPutBase64;失败留 dataUrl(刷新后丢,但 session 内 OK)
          //
          // 完整版:mark image node "uploading" + finalize 后改 src
          //
          // 本阶段选简化版(图片可能很大,异步处理出错时降级到 dataUrl 不影响 UX)
        };
        reader.readAsDataURL(imageFile);

        // 5. 阻止 PM 默认粘贴行为(handlePaste true)
        return true;
      },
    },
  });
}
```

### 2.1 智能插入位置(对齐 V1)

```ts
function insertImageAtCursor(view: EditorView, imageNode: Node): void {
  const { state } = view;
  const { $from } = state.selection;

  // 用 $from.depth 兼容嵌套容器(callout / blockquote / table cell)
  const depth = Math.max(1, $from.depth);
  const blockPos = $from.before(depth);
  const blockNode = state.doc.nodeAt(blockPos);

  let tr = state.tr;

  if (
    blockNode &&
    blockNode.type.name === 'text-block' &&
    blockNode.textContent.length === 0 &&
    !blockNode.attrs.isTitle  // L5-B3.11 — title 不可被替换
  ) {
    // 空 text-block(非 title)→ 替换
    tr = tr.replaceWith(blockPos, blockPos + blockNode.nodeSize, imageNode);
  } else {
    // 否则在当前 block 之后插入
    const afterPos = blockPos + (blockNode?.nodeSize || 0);
    tr = tr.insert(afterPos, imageNode);
  }

  view.dispatch(tr);
}
```

V1 → V2 关键差异:
- V1 `textBlock` → V2 `text-block`(短横线)
- V1 没 isTitle 概念 → V2 严格判 `!blockNode.attrs.isTitle`(B3.11 加的约束)

### 2.2 mediaStore 异步落盘(可选优化)

V1 直接存 dataUrl,V2 image NodeView 已经走 mediaPutBase64 → media:// URL 路径。
paste-media 也走同路径,但有两种策略:

**A. 同步插入 dataUrl + 异步替换 src 为 media://**(V2 image upload 已有路径)
**B. 异步等 mediaStore 完成再插入**(用户感知"粘贴卡了一下")

**推荐 A** — UX 即时(用户立即看到图),mediaStore 完成后透明替换。失败回落到
dataUrl(session 内可见,刷新丢)— 跟 image NodeView upload 失败行为一致。

实现细节:
```ts
reader.onload = async () => {
  // 同步插入 dataUrl 占位
  const dataUrl = reader.result as string;
  const captionNode = textBlockType.create();
  const tempImg = imageType.create({ src: dataUrl }, captionNode);
  const insertedPos = insertImageAtCursor(view, tempImg);  // 返回 PM pos
  if (insertedPos == null) return;

  // 异步:落 mediaStore → 替换 src
  const r = await mediaPutBase64(dataUrl, imageFile.type, imageFile.name);
  if (view.isDestroyed) return;
  if (r.success && r.mediaUrl) {
    const tr = view.state.tr.setNodeAttribute(insertedPos, 'src', r.mediaUrl);
    tr.setMeta('addToHistory', false);  // mediaStore 替换不记入 history(用户认为是同一次粘贴)
    view.dispatch(tr);
  }
  // 失败:留 dataUrl(image NodeView 同行为)
};
```

`insertImageAtCursor` 改返回插入位置(给异步 setNodeAttribute 用)。

### 2.3 PM pos 漂移问题

异步 setNodeAttribute 时,用户可能已经在文档其他地方继续编辑,`insertedPos` 可能漂移。

**预案**:在 `insertImageAtCursor` 内部 dispatch 后,记下 `tr.mapping`;但 mapping 在
后续编辑下还是会失效。

**简化**:不映射 — 用户在异步窗口期(几百 ms - 几 s)如果改了文档,setNodeAttribute
会落到错位置 / 报错(view.state.tr.setNodeAttribute 的 pos 失效会抛)。**try/catch 兜底
忽略**,留 dataUrl,session 内可见(跟 image upload 行为一致)。

更稳的方案:用 PM 节点 ID(自动生成 attr)定位 → 找到节点位置 → setNodeAttribute。
但这要给 image schema 加 id attr,V1 没做,**本阶段先不做**,留 sub-stage 优化。

## 3. 决策点

### Q1:HTML 让步规则保留 vs 砍

V1 在 `paste-media` 里检测 `<table|h1-6|...>` 让步给后续 handler。V2 当前**没**
smart-paste handler dispatcher,让步意味着 PM 默认粘贴(rich text → mark + textblock)。

- **A. 保留 V1 规则**:V2 当前 PM 默认粘贴对 Word/Excel HTML 是退化体验
  (没有 table 节点结构),但**等 Phase E smart-paste 接入时无需改本插件**
- **B. 砍掉规则**:V2 没 smart-paste,粘贴 Word 表格的图反而比破碎 HTML 好

**推荐 A** — 规则前向兼容,本阶段用户体验差异可控(用户主要场景是粘贴截图 / 网图,
不是从 Excel 粘贴含表格的 PNG fallback);未来 Phase E 直接接入即可。

### Q2:同步 dataUrl + 异步 mediaStore vs 仅 dataUrl

- **A. V2 优势路径**(异步 mediaStore):粘贴的图最终落 media://,刷新不丢,
  跟 image NodeView upload 一致;失败降级 dataUrl
- **B. 仅 dataUrl**(完全对齐 V1):简单,但刷新后丢,跟 image NodeView 行为不一致

**推荐 A** — V2 已有 mediaStore 基建,paste-media 顺势用,跟 image upload 路径
统一;复杂度可控(主要是 PM pos 漂移 try/catch 兜底)。

### Q3:caption 节点空白 vs 派生

V1 创建空 `textBlock` caption;V2 image NodeView 同行为。

- **A. 同 V1 创建空 caption**:简单
- **B. 派生**:粘贴时若有 alt 信息(从 HTML 找 `<img alt>`)填入 caption

**推荐 A** — 粘贴的图通常没 alt(剪贴板 image/* 是裸文件),B 复杂度收益不大;
后续 alt 派生可以做到 image NodeView upload(不限于 paste)。

### Q4:挂载位置(plugins 链)

editor-view-builder.ts 的 plugins 顺序决定 handlePaste 调用顺序。

paste-media 应该**早于** PM 标准 keymap(否则 PM 已经处理粘贴,handlePaste 不再调)。
建议放在 `buildLinkClickPlugin / buildNoteLinkCommandPlugin` 之后,`buildMarkKeymap`
之前。具体位置不重要(handlePaste 是 prop 不是 keymap,执行顺序按 plugin 注册顺序
返回 true 即拦截)。

**推荐**:在 `buildNoteLinkCommandPlugin` 之后插入(对齐"L5-B3 顺序加")。

### Q5:粘贴大图 / 多图

V1 只取第一个 image/* 文件,多图粘贴丢失后面的。

- **A. 同 V1**:只取第一个(简单,大部分用户场景就是单图截屏)
- **B. 全部插入**:遍历 items,每个 image/* 都创建 image node 顺序插入

**推荐 A** — 多图粘贴是边界场景,V1 已验证够用;B 留 sub-stage(若用户反馈)。

## 6. 风险

### 6.1 PM pos 漂移(异步 mediaStore 替换 src)

异步窗口期(典型 200-1000ms)内用户继续编辑 → setNodeAttribute pos 失效。
**预案**:try/catch 兜底,失败留 dataUrl(session 内可见,刷新丢)。

### 6.2 mediaStore IPC 失败

`window.electronAPI?.mediaPutBase64` 不可用 / 失败 — `mediaPutBase64` 已返
`{ success: false, error }`,paste-media 留 dataUrl(降级行为对齐 image NodeView upload)。

### 6.3 大图阻塞 FileReader

FileReader.readAsDataURL 对 5MB+ 大图可能卡 100-500ms。
**预案**:本阶段不优化(用户场景多是截屏 < 2MB);Phase E 可改 ArrayBuffer + Blob URL。
