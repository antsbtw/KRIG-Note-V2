## 1. 实际改动 vs 设计

### 1.1 完全照设计

- **memoryPlaybackSequence generator** 直迁 V1
- **段边界检测 0.3s buffer**(Qc-2=A)
- **duration 未就绪等 onDurationReady**(Qc-4=A):start 时显 `🧠 ⏳`,duration 推送后续启动
- **start 跳到 attrs.memoryLastStep / stop 写回 stepIndex**:对齐 V1 持久化策略
- **active 不持久化**:对齐 V1(重启不自动播视频)
- **Qc-6=A actionBar 顺序对齐 V1**:一次到位

### 1.2 微调

- **memoryBtn deps 用 closure-based getter**(getSegmentDuration / getTracker / getLastStep)而非快照参数 — 让 attrs 变化(如 segDur 切换)立即生效,不需重建 button
- **onUpdateAttrs 内部走 dispatch addToHistory=false**:避免 segmentDuration / memoryLastStep 进 undo 栈污染
- **destroyFramework 中 memoryBtn.destroy 提前到 tracker.destroy 之前**:确保 stop() 写回 lastStep 时 tracker 仍在(否则 stepIndex 状态丢失)

### 1.3 update() 不主动同步 segDur / memoryLastStep

外部驱动(撤销 / 协作)若改 attrs.segmentDuration / memoryLastStep,memoryBtn 内部状态不重新同步。理由:
- segDur 切换:用户一般主动从 dropdown 改;撤销 segDur 改动是少见行为
- memoryLastStep:active 状态本身不持久化,start() 时才读 — 已隐含同步
- 如真出问题(用户撤销但 memory 还在播)→ stop + 重新 start 即可,体验损失极小

接受现状,不引入 update 复杂度。

## 3. 风险监测

### 3.1 0.3s buffer + 300ms 轮询冲突
设计 § 6.2 已分析:advance 内 seekTo 把 currentTime 拉回段起始 → 触发 boundary check 失败(currentTime < boundary) → 不连续触发。**逻辑自洽**,实现已落地。

### 3.2 YouTube autoplay policy
Memory Mode 调 `tracker.play()` 在 YouTube iframe 上可能被静音 / 阻拦(Chrome 86+ 严格)。V1 已踩,需用户先手点过 iframe 一次。**接受现状**(Qc-5=A);UI 上 memBtn 看起来在播但 iframe 实际可能没动。后续如真扰用户加 toast 提示。
