## 0. 背景

L5-B4.3 目标:把 V1 ai-note-bridge 闭环(用户在 AI 网页右键"提取到笔记")迁到 V2,
让 note 能消化 AI 对话内容(尤其 Claude artifact)。直迁路线已拍板(用户判断 V1 测试积累值得)。

L5-B4.3.1 是**第一波前置基建**,落 3 个独立模块,**没有可见用户功能**(只是把砖头搬来):

1. **`mediaStore`** — V2 没有 media 存储,Claude artifact 含 base64 图,必须先有
2. **`md-to-pm`** — V2 没有 Markdown → ProseMirror 转换,AI 提取产物是 Markdown,必须先转
3. **`ai-service-types`** — Claude / ChatGPT / Gemini 服务识别(URL → service profile)

3 个模块独立,合并一个 sub-stage 是为了 L5-B4.3.2/3 上来就能直接用,不会反复"修上一个的依赖"。

## 2. 模块设计

### 2.1 mediaStore(V2 版)

**职责**:存储二进制媒体(图 / 音 / 视频),发 `media://` 协议 URL 给 webview / note 引用。

**文件结构**:

```
src/storage/media-store.ts                  # 类型 + IPC 调用封装(renderer 用)
src/platform/main/media/media-store-impl.ts # 文件系统实现(main 用)
src/platform/main/media/media-handlers.ts   # IPC handler 注册
```

**API(类型层)**:

```ts
// src/storage/media-store.ts
export interface MediaPutResult {
  success: boolean;
  /** media:// 协议 URL,可直接放进 PM image src 或 webview src */
  mediaUrl?: string;
  /** 内部 ID(去重 hash),后续可查 */
  mediaId?: string;
  error?: string;
}

/** 把 base64(或 data URL)写进 media store,返回 media:// URL */
export async function mediaPutBase64(
  input: string,
  explicitMime?: string,
  hintedFilename?: string,
): Promise<MediaPutResult>;

/** 从外部 URL 下载,写进 media store(给 web-bridge 拉远端图用)*/
export async function mediaDownload(
  url: string,
  type: 'audio' | 'image' | 'video',
): Promise<MediaPutResult>;
```

**实现层(main 进程)**:

- 存储路径:`{userData}/krig-data/media/{images|audio|video|files}/`
- 索引:`{userData}/krig-data/media/media-index.json`(JSON,跨重启持久)
- 去重:SHA256 hash 前 16 字符作 mediaId
- 协议:`media://images/img-<hash>.<ext>` → `protocol.handle('media', ...)` 解析到本地文件
- 大小限制:image 20MB / audio 50MB / video 200MB / files 20MB(沿用 V1 默认)
- ext 选择:文件名 hint > MIME 查表 > 'bin'

**协议注册时机**:`app.whenReady` 之后,`createMainWindow` 之前(否则 webview 加载 `media://` 会 404)。

**IPC 通道**:
- `MEDIA_PUT_BASE64`(invoke,handle):`(input, mime?, filename?) → MediaPutResult`
- `MEDIA_DOWNLOAD`(invoke,handle):`(url, type) → MediaPutResult`

### 2.2 md-to-pm

**职责**:Markdown 字符串 → ProseMirror JSON(直接套到 NoteEditor 的 schema)。

**文件**:`src/storage/md-to-pm.ts`(直迁 V1 363 行)

**API**:

```ts
// PMNode 结构跟 ProseMirror JSON 对齐(NoteEditor 能直接消费)
interface PMNode {
  type: string;
  attrs?: Record<string, unknown>;
  content?: PMNode[];
  marks?: { type: string; attrs?: Record<string, unknown> }[];
  text?: string;
}

/** Markdown → PMNode[](带 base64 图的话会走 mediaPutBase64 写入 media store)*/
export async function markdownToProseMirror(md: string): Promise<PMNode[]>;
```

**支持的 Markdown 元素**(V1 已实现,V2 直迁):
- block:heading(h1-h6)/ paragraph / code block / blockquote / bullet list / ordered list / task list / horizontal rule / image / math block (`$$...$$`)
- inline marks:bold / italic / code / link / inline math (`$...$`)

**改造点**:V1 的 `mediaSurrealStore.putBase64()` 调用 → 改成 V2 `mediaPutBase64()`(同语义,封装在 storage/media-store.ts)。

**未支持但 AI 提取常见的元素**(留 L5-B4.3.2/3 评估补丁):
- 表格(`| col | col |`)— V1 也没实现
- 嵌入 HTML — V1 直接转纯文本

### 2.3 ai-service-types

**职责**:定义 Claude / ChatGPT / Gemini 三家的服务 profile(URL 模式 / DOM selector / API endpoint),并按 URL 检测当前是哪家。

**文件**:`src/shared/types/ai-service-types.ts`(直迁 V1 ~120 行)

**API**:

```ts
export type AIServiceId = 'claude' | 'chatgpt' | 'gemini';

export interface AIServiceProfile {
```

## 3. 决策点

### Q1:mediaStore 的 IPC 异常处理

V1 mediaSurrealStore.putBase64 失败时返回 `{ success: false, error }`,不 throw。V2 继承同款契约。
但 `mediaDownload` 涉及网络请求,可能超时:

- **A. 客户端不超时,服务端 net.fetch 自身超时**(V1 默认):简单
- **B. 客户端 30s 超时,超时返回 { success: false, error: 'timeout' }**

**推荐 A** — 跟 V1 对齐;后续如有需要再加 timeout 参数

### Q2:V2 schema 缺失节点(image / mathBlock / mathInline / fileBlock / externalRef / table 等)如何处理

**v0.1 草案曾推荐"降级为 text-block / link mark"** — 用户拒绝,理由:降级会**永久损失语义**,
且这些缺失节点正是后续 note 完整化要补的,降级会绕过"必须实现这些节点"的压力。

**v0.2 修订方案**:
- md-to-pm 输出**目标 V2 schema 节点名**(image / mathBlock / mathInline / fileBlock /
  externalRef / table / tableRow / tableHeader / tableCell — 全部驼峰命名,跟 bulletList /
  taskItem 等一致)
- V2 schema 暂未实现这些节点 → md-to-pm 输出 `{ type: 'unknown', attrs: { originalType,
  missing: true, raw, error? } }` 占位
- doc 能装(unknown 节点不破坏 PM doc),NoteEditor 渲染层显示"暂未支持: <originalType>"
- L5-B4.3.3 闭环测试时 unknown 节点暴露 schema 缺口,反向驱动后续 sub-stage 补齐
- schema 补齐后 md-to-pm **不需要改**,直接输出真实节点(注释里有 PM_NODE_REGISTRY 标记 ✅/❌
  作进度参考)

**为什么 unknown 节点而不是直接输出 image / mathBlock?**:
- 如果 md-to-pm 直接输出 `{ type: 'image', ... }`,V2 schema 不识别,**整个 PMNode.fromJSON
  会拒绝**,markdownToProseMirror 调用方拿不到任何内容(整篇丢)
- unknown 节点是 V2 schema 显式注册的"占位类型"(L5-B4.3.x 加),保证 doc 能装

**Q2 = unknown 节点占位,不降级**

### Q3:ai-service-types 是否一次到位 3 家

V1 定义里 Claude / ChatGPT / Gemini 的 profile 数据是整套的(selectors / apiPatterns 都有)。
V2 是直迁还是先做 Claude:

- **A. 直迁全部 3 家**(类型完整,后续 sub-stages 直接用)
- **B. 只做 Claude,其他 2 家 stub**(代码少,但留 TODO)

**推荐 A** — 类型不可能"半做",直迁才是直迁

### Q4:`media://` 协议是否复用 webview-translate partition

translate partition 已经注册了 CSP bypass,如果 media:// 协议挂在那个 session 上,翻译 webview 也能加载 media://;但其他 partition(普通 webview / renderer)不行。

V1 走的是 `protocol.handle('media', ...)` 全局注册(所有 session 共享),V2 同款。

- **A. 跟 V1 一致,全局注册**:任何 webview / renderer 都能加载 media://
- **B. 只给 default session**:其他 partition 单独注册

**推荐 A** — 直迁

### Q5:V2 Note 存的 Markdown 还是 ProseMirror JSON

NoteEditor 用 ProseMirror,所以 Note.doc 是 PM JSON(`DriverSerialized`)。md-to-pm 转出 `PMNode[]` 后,需要包成 NoteEditor 期望的 `DriverSerialized` 格式才能塞进 `noteStore.update(id, { doc })`。

- 这部分**包装代码 L5-B4.3.3 处理**(知道往哪个 note 写时再包),L5-B4.3.1 不做
- L5-B4.3.1 只产出 `PMNode[]`(中间格式)

✅ 已确认

## 6. 风险

### 6.1 mediaStore 协议注册时机

V1 在 `app.whenReady` 后注册 media:// 协议。如果 V2 register 时机晚于第一个 webview 创建,
webview 加载 media:// 会 ERR_FILE_NOT_FOUND(协议未注册)。

**预案**:在 `src/platform/main/index.ts` 的 `app.whenReady().then(...)` 内,**第一行**调
`mediaStore.registerProtocol()`,确保早于 createMainWindow。

### 6.2 V2 storage 目录初次出现的导入路径稳定性

`src/storage/` 在 vite alias 已预留为 `@storage/*`。L5-B4.3.1 是首次使用此 alias,如果
有路径解析问题(比如 alias 跟 tsconfig 不一致),会编译失败。

**预案**:实施时先 verify `tsconfig.json` 的 `paths` 是否含 `@storage/*`(不含的话补上),
typecheck 通过再继续。

### 6.3 V1 md-to-pm 用的 ProseMirror node type 跟 V2 NoteEditor schema 是否一致

V1 NoteEditor schema 跟 V2 NoteEditor schema 在 L5-B3.x 阶段都做过演进,可能 node type 名
有差异(比如 V1 用 `bullet_list`,V2 用 `bulletList`)。

**预案**:L5-B4.3.1 只产出 PMNode[] 中间格式,**不包装成 DriverSerialized**(留 L5-B4.3.3
判定 schema 兼容性)。如果迁过来的 PMNode 类型名跟 V2 schema 不一致,L5-B4.3.3 写适配层。
