## 2. 关键架构决策（红线级，后续阶段不得违背）

### A. 媒体上传走「路线 B：webview 喂文件给 X 上传控件」,不走官方 API ⭐

- **路线 A(否决)**:X 官方 `media/upload`(INIT/APPEND/FINALIZE)+ API key + OAuth。
  → 等于程序直接发布,**撞反自动化红线**,且 API 收费/限额。**不走。**
- **路线 B(采纳)**:在 X 真实发推框里,把截好的图 / note 里的图**喂给 X 自己的文件
  上传控件**(`<input type=file>` / 拖放区,selector 待 spike,如 `[data-testid="fileInput"]`),
  让 **X 网页前端自己完成 INIT/APPEND/FINALIZE**。我们只负责"喂文件",上传与发布仍由
  X / 用户完成。
- **守红线**:依然「填内容(含媒体),用户点发布」。

### B. 写方向最高红线(贯穿所有阶段):永远「填充内容,用户点发布」

- 任何阶段**绝不程序点发布按钮 / 不批量自动发 / 不定时发**。
- 若未来出现"攒一批一键发""定时发"等诉求 → 必须改成「排队 + 每条都等用户在 X 手动确认」,
  不得真后台自动发。新功能撞这条 → 停下来跟总指挥确认。
- 印证:X 2026 自动化规则,全自动操作会被封号。

### C. fail-loud 降级(贯穿)

- 注入 / 上传 / 截图任一步不可靠 → 退「复制到剪贴板 / 提示用户手动」+ 弹窗明示降级,
  **绝不静默假装成功**。关联记忆 [[feedback-fail-loud-no-fallback]]。

### D. 仿 X 预览:复用 X 自己渲染,不凭空重写 CSS

- **不做**:在 note 里用我们自己的 CSS 重新实现一遍 X 的推文样式(要追 X 改版,维护地狱)。
- **做**:内容(含媒体)注入 X 框后,X 自己会渲染出预览态 —— 截取/镜像**X 真实渲染的
  预览**给用户看。这样"仿真"=X 自己渲染的,最准且零追版成本。
- **依赖**:这条最自然的实现依赖「注入链路 + 媒体上传已就位」,故预览排在媒体之后(§3)。
- **待定细节**:预览做「实时」(改 note→预览跟变,成本高)还是「快照式」(点一下生成)。
  **倾向先做快照式。** 开工前再定。

### E. selector / DOM 交互全部集中进 profile 配置

- 所有 X selector 进 `X_PROFILE.selectors`(`src/shared/types/x-service-types.ts`),
  失效 fail-loud 降级。加新写动作 = 加 selector + 加编排函数,不碰注入引擎
  (`web-service-base/webview-input.ts` 的公共原语 AI/X 共用)。

## 3. 分期实施路线（总指挥定序）

> 阶段 1、2 已完成。下表"状态"列 ✅=已交付 / 🔜=下一步 / 📋=已规划。

| 阶段 | 内容 | 状态 |
|---|---|---|
| **阶段 1** | 提取推文 → note tweetBlock(= 用户构思的"回复方式1",本质是阅读/收藏,不是回复) | ✅ |
| **阶段 2** | 发文字推 + 回复(现场写)+ 字数/降级/剪贴板兜底 | ✅(selector 待实机 spike) |

| **阶段 2.5-b** | **文字推的图片/视频媒体上传**(路线 B,§2.A):note 里的图/视频 → 喂 X 上传控件 → 附到推文。最多 4 图 + 视频。 | 🔜 |
| **阶段 3** | **article 格式发送**:块序列注入器(逐块映射 note→X article 富文本),note 有/article 没有的格式 → 截图兜底。**纯图片发送**(用户决定2)= 截图能力的副产品,整段截图当媒体附件。 | 📋 设计已有(设计文档 §5 阶段3) |
| **阶段 4** | **仿 X 预览**(§2.D,复用 X 渲染);依赖媒体上传就位。 | 📋 |
| **阶段 5** | **回复素材库**(用户决定4):一种「X 回复素材」收集 + 挑选 UI。技术轻,可灵活插队。 | 📋 |

### 用户构思的"回复 3 方式" → 归位说明
- **方式1「提取推回 note 做链接记录」** = 阶段 1 已实现(阅读/收藏,非回复)。
- **方式2「note 中现场写回复」** = 阶段 2 已实现。
- **方式3「从 note 预备好的素材里挑一条回复」** = 阶段 5「回复素材库」(新功能)。

## 4. spike 方法论（固化,后续所有涉及 X DOM 的阶段照此走）⭐

总指挥拍板:「参考已有开源库,发现问题再自己 spike」。精确化为:

```
1. 先翻 1-2 个活跃开源库的 selector + 交互序列,拿"已知能用的起点"
   (参考源见 §5;X 提取本身就是 V1 参考社区选择器来的,同脉络)
2. 在我们自己的 X webview 里验证这些 selector 还在不在、交互通不通
3. 一致 → 直接用;不一致/失效 → 只对差异部分实机 spike 补
4. 只取「定位 + 喂内容」的部分,坚决丢弃开源库里「自动点发布/批量」的部分(红线 §2.B)
5. selector 全部集中进 profile 配置,失效 fail-loud 降级(§2.E)
```
