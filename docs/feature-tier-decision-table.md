## 0. 怎么用这张表

1. **现在（测试期）**：所有功能**全免费**，不分档。后端按账号发**一年期免费授权**，到期由后端权威记录（见设计文档 §测试期策略）。这张表先列全、想清楚，**一年后**才启用分档。
2. **填写方式**：在「档位」列填你的决定（`free` / `pro` / `团队` / `不收费` / `内部`）；「定价理由」列写为什么。
3. **判断维度**（建议）：
   - **后端是否有真实成本/价值**（如 AI 调用算力、云服务）→ 适合做付费墙，且后端能真正锁住。
   - **纯本地功能**（导入/备份/编辑）→ 客户端锁不住（机器主人能绕过），放付费墙意义有限且可能伤体验，但仍可作为「软性付费引导」。
   - **使用频率/刚需程度** → 越刚需越该 free（降低流失），越增值越可 pro。
   - **竞品对标** → 同类产品这功能收不收费。
4. 「锁定强度」列已替你标好：`[S]`=后端能真锁，`[L]`=本地软锁（仅 UX 引导，技术上挡不死）。这影响「这个付费墙值不值得设」。

## 13. 我的建议草案（讨论起点，可全部推翻）

如果一年后只做 `free` / `pro` 两档，我的初步建议是：

- **free**：核心编辑、笔记/文件夹、表格分栏、导航、网页浏览/书签/下载/剪藏、电子书阅读、导入导出备份、X 查看、Thought、基础高级块（公式/Mermaid）。
  - 理由：这些要么是刚需（流失敏感），要么纯本地锁不住，放 free 既留住用户又不浪费对抗成本。
- **pro**：AI 全家桶（提问/提取/同步）、语言学习（翻译/TTS/词典，若经后端）、Canvas/图谱、yt-dlp 视频下载、X 发布、PDF 平台提取。
  - 理由：这些要么后端有真实成本能真锁（AI、翻译、PDF 提取），要么是明显增值的重功能（Canvas、视频）。

**核心逻辑**：付费墙优先压在 `[S]`（后端能真锁、有成本）的功能上；`[L]` 功能若设付费墙，要清楚它是「软引导」而非「真限制」。

> 这只是起点。你按自己对市场/用户的判断改即可——改这张表不用动后端，调 `FEATURE_MIN_TIER` 也不用发版（若做成后端下发则连改表都热更）。
