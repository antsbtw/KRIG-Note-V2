## 1. 一句话定位

**KRIG-Note 是基于知识图谱的知识表达媒介。**

底层是关系图谱（机器可推理），表层是多种视觉表达（人可理解），两者通过 **Substance Pattern** 连接。

---

## 2. 核心区分：知识图谱 vs 知识表示

这是 KRIG-Note 整个设计哲学的根。

### 2.1 知识图谱（Knowledge Graph）

| 维度 | 定义 |
|------|------|
| **服务对象** | 机器 |
| **目的** | 知识推理 / 关联计算 / 一致性检查 |
| **表达形式** | 原子化的关系网（subject-predicate-object atom） |
| **评判标准** | 机器能否高效遍历、推理、查询 |
| **客观性** | 客观 — 同一份图谱对所有机器是一样的 |

代表技术：OWL、RDF、Neo4j、SPARQL。

### 2.2 知识表示（Knowledge Representation）

| 维度 | 定义 |
|------|------|
| **服务对象** | 人 |
| **目的** | 知识传递 / 学习 / 理解 |
| **表达形式** | 图形化、结构化、模式化的视觉呈现 |
| **评判标准** | 人能否"一眼看懂"、"愿意继续看"、"能学到东西" |
| **主观性** | 主观 — 不同的人需要不同的表达方式 |

代表技术：思维导图、流程图、组织架构图、时间线、矩阵热图。

### 2.3 KRIG-Note 的位置

```
        知识图谱（机器层 · 客观）
             ↓
   ┌─────────────────────────┐
   │  KRIG-Note 在这里：      │
   │  把图谱"翻译"成多种      │
   │  人能理解的视觉表达       │
   └─────────────────────────┘
             ↓
        知识表示（人层 · 主观）
```

**KRIG-Note 站在知识图谱的肩膀上做知识表示。**

它**不是**一个图谱可视化工具（那只是渲染层）；
它**是**一个让"客观的图谱"能产出"多种主观的表达"的媒介。

### 2.4 核心循环：人机共同优化的知识闭环

KRIG-Note 不只是"机器 → 人"的单向翻译，是 **人机协作优化** 的双向闭环：

```
                  ── 推理 ──
              ╱─────────────╲
             ╱               ╲
            │                 │
   机器 ←──┤  图谱      KR 图  ├──→ 人
            │                 │
             ╲               ╱
              ╲─────────────╱
                  ── 优化 ──
```

**椭圆两极**：左 = **图谱**（客观 / 机器友好），右 = **KR 图**（主观 / 人友好）。

**椭圆两弧**：
- **上弧 · 推理**：机器在图谱侧的活动 — 关系推理、一致性检查、模式识别
- **下弧 · 优化**：人在 KR 图侧的活动 — 看、改、重新组织

**循环方向**：机器从左侧推动图谱演化 → 流向 KR 图（被翻译） → 人在右侧优化 KR 图 → 反向更新图谱 → 机器再次推理。

**循环不停转，知识不断打磨。** 每一轮，图谱和 KR 图都被人和机器共同打磨得更准确。

#### 闭环的具体场景

- **A. 拖动反向标注关系**：用户在力导视图发现 A、B 被拉得近 → 直觉觉得有关系 → 拖动靠近 → 系统提示"加一条 A relates-to B？" → 人确认 → 图谱新增。
- **B. 视图发现数据错误**：树视图把 Shell 画成 Window 的兄弟 → 人识别"contains 方向反了" → 反向修正 atom。
- **C. 模式识别反向加抽象**：矩阵视图看到 5 个 View 行为相似 → 人确认是同类 → 反向加 Substance Pattern → 下次自动按模式呈现。
- **D. 人机互为老师**：机器建议合并相似节点 → 人拒绝 → 机器学到这种相似不构成合并理由。或：人重排节点 → 机器观察 → 推断布局偏好。

#### 为什么这个闭环重要

| 维度 | 机器擅长 | 人擅长 |
|------|---------|--------|
| 完整性 | 强 | 弱 |
| 一致性检查 | 强 | 弱 |
| 规模处理 | 强 | 弱 |
| 上下文判断 | 弱 | 强 |
| 抽象 / 模式识别 | 中 | 强 |
| 价值取舍 | 弱 | 强 |

**单向工具（机器→人）只用了一半。双向闭环让两者互补。**

KRIG-Note 真正可能颠覆的地方在这里：**不是更好的图谱工具，是让人机能通过图谱"对话"的媒介**。

## 3. 关键洞察：同一图谱 → 多种表达

### 3.1 为什么"一图多表"是必然的

> **同一个知识结构，不同的用户一定会产生不同的表达方法。**

这不是 UI 偏好的问题，是**认知模型差异**：

```
同一份 KRIG 软件架构图谱：

  软件架构师 看到 → 「分层架构图」  →  树视图
  产品经理   看到 → 「用户流程图」  →  时间线视图
  性能工程师 看到 → 「调用关系图」  →  矩阵视图
  新人       看到 → 「导览图」      →  简化的力导视图
```

**强迫所有人看同一种图，是工具的失败。**

### 3.2 推论：图谱稳定，视图廉价

这个定位带来一个反直觉的、但非常重要的设计原则：

> **图谱内容应该是稳定的资产；视图应该是廉价的、可丢弃的、可大量并存的消耗品。**

| | 图谱（atom 层） | 视图（pattern / view-mode 层） |
|---|---|---|
| 性质 | 资产 | 消耗品 |
| 改动门槛 | 高（影响所有衍生视图） | 低（随时可改） |
| 用户态度 | 慎重 | 轻松 |
| 数量 | 唯一 | 可大量并存 |

这跟 Notion / Obsidian 完全相反 —— 它们是"页面是资产，视图很贵"。
**KRIG-Note 是"关系是资产，视图很廉价"。**

### 3.3 推论：人能创造视图，不只是消费

知识传递是双向循环：

```
人接收知识 →  形成理解 → 重塑表示 → 分享给别人
            （内化）      （外化）       （传递）
```

KRIG-Note 必须支持完整的循环：
- 用户**接收**别人做的视图（被动消费）
- 用户**重塑**视图、创造自己的表达（主动外化）
- 用户**分享**自己创造的表达模式给社区（传递）

**用户能自定义 Substance Pattern / View Mode，是知识传递的关键环节，不是"个性化设置"。**

## 4. 三层架构（与 KRIG-Three-Layer-Architecture.md 衔接）

愿景层面的三层（自上而下）：

```
┌────────────────────────────────────────────────┐
│ Layer 3 · 知识表示（主观 · 多样 · 人友好）       │
│   - Substance Pattern（结构模式）               │
│   - View Mode（视图模式：force / tree / matrix）│
│   - 最终渲染 = Pattern × View Mode              │
└────────────────────────────────────────────────┘
                       ↑
┌────────────────────────────────────────────────┐
│ Layer 2 · 知识内容填充（桥梁层）                │
│   - 把"原始信息"转成"原子化知识"                 │
│   - 来源 A：AI 抽取笔记 / 对话（调 LLM API）     │
│   - 来源 B：公共图谱按需查询（Wikidata 等）      │
│   - 来源 C：用户手动（已部分实现）               │
│   - 严格人工审核（人是仲裁者）                   │
│   - 详见 docs/graph/KRIG-Graph-Layer2-Vision.md │
└────────────────────────────────────────────────┘
                       ↑
┌────────────────────────────────────────────────┐
│ Layer 1 · 知识图谱容器（客观 · 机器友好）        │
│   - graph_geometry / graph_intension_atom       │
│   - 容器已实现 ✓（v1.4），内容由 Layer 2 增量填充 │
└────────────────────────────────────────────────┘
```

**Layer 1 是机器的领地，Layer 3 是人的领地，Layer 2 是把外部世界搬进图谱的桥梁。**

KRIG-Note 的核心价值不在 Layer 1（OWL / Neo4j 已经做得很好），
而在 **Layer 2 + Layer 3** 的组合：让用户能**轻松填入知识**（L2）+ **自由表达知识**（L3）。

> **重要**：Layer 2 不自建 NLP / 推理，调外部 API（LLM / Wikidata）。KRIG 是编排者，不是引擎。详见 `docs/graph/KRIG-Graph-Layer2-Vision.md`。

## 5. 设计原则（从愿景推导）

未来所有设计决定要回到这 5 条检查：

### 5.1 图谱面向机器，视图面向人

任何决定问一句：**这是给机器用的，还是给人用的？**
- 给机器（推理/查询）→ 放在 Layer 1（atom）
- 给人（理解/传递）→ 放在 Layer 3（pattern / view-mode）

### 5.2 关系是资产，视图是消耗品

任何决定问一句：**改这个东西有多贵？**
- 改 atom（关系/属性）→ 应该贵 / 慎重
- 改 view（pattern / view-mode）→ 应该便宜 / 轻松

### 5.3 一图多表

任何视图层的决定问一句：**这个能不能并存多份？**
- 如果"系统强制只能有一种" → 红灯，违反愿景
- 如果"用户可创建任意多种" → 绿灯

### 5.4 用户能创造视图模式，不只能选

任何视图框架问一句：**用户能扩展吗？**
- 系统硬编码所有视图模式 → 红灯
- 系统提供基础 + 用户可加 → 绿灯

### 5.5 媒介定位 — KRIG-Note 是"图谱表达媒介"

任何功能取舍问一句：**这个是辅助"表达"，还是离题了？**
- 让图谱→视觉的翻译更优雅 → 在做对的事
- 加任意的图谱编辑功能但与"表达"无关 → 可能离题

### 5.6 视图是双向接口

视图不只是"展示"图谱，还是**修改**图谱的入口。任何视图设计要问：
- 这个视图能让用户**看**吗？（必须）
- 这个视图能让用户**改**吗？（应该 — 视图里发现的问题应该能就地修正）
- 改动是否**反写回 atom 层**？（必须 — 否则视图和图谱会脱钩）

视图修改图谱的入口可以多样（拖动、右键菜单、专门的编辑模式），但**修改必须最终落在 atom**，不能停留在"只是看起来变了"。

## 6. 与既有工具的根本不同

| 工具 | 媒介 | KRIG-Note 的差别 |
|------|------|-----------------|
| **Word** | 文字 | KRIG 的媒介是图谱，文字只是 atom 之一 |
| **Excel** | 表格 | 表格只是众多视图之一 |
| **Notion** | 数据库 + 页面 | Notion 视图依赖数据库 schema；KRIG 视图是结构模式的派生 |
| **Obsidian** | Markdown + 双向链接 | Obsidian 节点孤立，无"模式"；KRIG 把模式作为一等公民 |
| **Logseq** | 大纲 + 块引用 | 同 Obsidian |
| **Notion Database Template** | 字段固定 | KRIG Pattern 是**结构固定**（节点+关系+布局），比模板深一层 |
| **Neo4j / OWL Tools** | 图谱本身 | 这些是给机器用的；KRIG 是给人用的 |

**KRIG-Note 的位置：在"知识管理工具"和"知识图谱工具"之间，实际上是一个新品类 — 知识表达媒介（Knowledge Expression Medium）。**

## 8. 不在愿景内的事（明确排除）

为避免愿景泛化，明确**不做**的事：

- ❌ **替代 Obsidian / Notion 的全部功能** — KRIG 不是"全能笔记工具"，是知识表达媒介
- ❌ **完整的图谱推理引擎** — 推理是机器的事，留给 Neo4j / OWL Tools；KRIG 必要时可以集成，但不自建
- ❌ **复杂的协作 / 实时同步** — 至少 v1.x 单人本地优先
- ❌ **强制某种"正确的"视图** — 违反"一图多表"原则
- ❌ **AI 单方面生成 / 替代人的知识判断** — AI 可作为"机器"的延伸参与闭环（建议关系、识别模式、推断偏好），但**人始终是闭环的最终仲裁者**。AI 写入图谱的任何东西必须经人确认 / 可撤销。
