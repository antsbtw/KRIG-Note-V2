# 存储层评估（2026-04-09）

## 范围
- SurrealDB 客户端与生命周期：`src/main/storage/client.ts`
- Schema 初始化：`src/main/storage/schema.ts`
- 数据存取：`src/main/storage/note-store.ts`、`src/main/storage/folder-store.ts`、`src/main/storage/activity-store.ts`
- Session 持久化（文件）：`src/main/storage/session-store.ts`
- Markdown/Atom 转换：`src/main/storage/md-to-atoms.ts`、`src/main/storage/md-to-pm.ts`
- 参考原则：分层设计、模块自包含、存储与交换格式、可替换性、命名/可描述性。

## 发现与代码证据（按严重度排序）
1) **Session 双路径不一致、数据库 `session` 表未使用**  
   - Session 实际写到本地 JSON 文件（`session-store.ts`），但 schema 定义了 `session` 表且未被使用。  
   - 影响：存在逻辑分叉风险。  

2) **全局凭据/端口硬编码，缺少配置与安全控制**  
   - 位置：`src/main/storage/client.ts` **第 14-18 行**，硬编码了 `DEFAULT_PORT = 8532` 及用户名/密码 `root/root`。未启用 TLS。  
   - 影响：安全极易泄漏，如果机器默认端口占用将导致启动失败，与“配置化”原则严重不符。  

3) **数据完整性保障不足（SCHEMALESS + 递归删除）**  
   - 所有表 SCHEMALESS，note/folder 约束仅靠查询；`folder-store.delete` 递归删除文件夹并直接删除其笔记，无软删除或事务保护。  

4) **ID 生成与并发/去重策略薄弱**  
   - `generateId()` 依赖 `Date.now()`+随机串，未防止并发冲突；无 Surreal 内唯一约束。  

5) **客户端健壮性不足（缺少降级与提示）**  
   - 位置：`src/main/storage/client.ts` **第 125-142 行 `waitForReady` 方法**。  
   - 痛点：启动若端口占用或二进制缺失当前仅通过死循环重试然后抛出异常，无备选端口策略；init 失败后 `isReady` 永远 false，**毫无对 UI 层的状态反馈**，造成静默失败体验极糟。

## 改进建议
- **引入应用配置中心**：使用类似 `.env` 或动态配设为 DB 端口和账密提供注入能力，启动时采用类似 `portfinder` 的自动空闲探测机制解决冲突。
- **完善异常界面接管**：在 DB 启动失败或异常时，需派发 IPC 给前端，以全屏 Error 遮罩或 Dialog 明示存储初始化失效原因，阻止死局。 
- 明确 Session 存储策略：要么使用 Surreal `session` 表，要么删除表定义，统一一处。 
- 为 note/folder 增加唯一约束与外键/软删除标记；删除操作引入事务或回收站。 
- 抽象 ID 生成（如 nanoid/UUID）并在 DB 层添加 UNIQUE 索引。
