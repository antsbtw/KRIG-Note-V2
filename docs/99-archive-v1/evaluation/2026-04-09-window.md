# KRIG-Note 窗口框架层评估（2026-04-09）

## 评估范围
- 主窗口与槽位：`src/main/window/shell.ts`
- 布局计算与分割条：`src/main/slot/layout.ts`、`src/main/slot/divider.ts`
- Preload：`src/main/preload/shell.ts`、`src/main/preload/divider.ts`
- 参考原则：`principles.md`、`design-philosophy.md`

## 发现与代码证据（按严重度排序）
1) **窗口层硬编码加载逻辑，未遵循注册反转规范（违反“框架与插件分离”）**  
   - 位置：`src/main/window/shell.ts` **第69-191行** `createViewForWorkMode` 函数内。  
   - 证据：反复使用 `if (viewType === 'note') ... else if (viewType === 'ebook')` 判断和分发具体的 HTML 和资源路径。  
   - 影响：新增功能模块必须去改核心通讯框架，违背插件封闭性；应当交由各插件注册其装载点，由框架读取注册表自动反射。

2) **业务逻辑直接注入框架生命周期（严重越权）**  
   - 位置：`shell.ts` **第125-178行**。  
   - 证据：硬编码 `variant === 'extraction'` 的判断，去监听 session 下载流并显式调用了 `importExtractionData(data)` 与 `setPendingNoteId` 等重度的业务逻辑操作。  
   - 影响：将文件流下载处理与数据转换跟 Window 生命周期严重绑死。违背单一职责和控制反转原则。

3) **NavSide 宽度为进程级全局状态，未随 Workspace 隔离**  
   - 位置：`layout.ts` 使用模块级 `navSideWidth`（全局），导致不同工作区共用 NavSide宽度。

4) **视觉与交互样式内联硬编码**  
   - 位置：`shell.ts` 中定义的 `TOGGLE_HTML` 和 `NAV_RESIZE_HTML` 包含了数百个字符的 CSS 内联配置；`divider.ts` 同理。  
   - 影响：让基于配置驱动的统一换肤变为不可能。  

## 改进建议（重构路径）
- **核心：拆解 `createViewForWorkMode` 分发地狱**。改由主进程阶段载入 `workModeRegistry` 中预加载定义的页面URL，框架代码仅抽象包裹一层统一的 `WebContentsView` 容器。
- **业务下载逻辑下沉**：解绑 Extraction 拦截，把文件下载提取工作作为专属协议的 RPC Handler 丢入到 Plugin 层的主进程服务里，框架不再承担这部分数据清洗逻辑。
- 将 `navSideWidth` 纳入 WorkspaceState，确保每个 Workspace 持有独立的尺寸记忆。
- 提供共享主题/样式源，废除或替换 TOGGLE/Divider/Resize 的内联样式字符串设计。
