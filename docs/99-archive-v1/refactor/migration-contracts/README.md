# 迁移功能契约（Migration Functional Contracts）

每个插件迁移**之前**必须建立一份功能契约文档：`<插件>.md`。

> **契约的作用**：
> 1. **Builder 必读**：动手前知道"哪些细节绝对不能碰"
> 2. **Auditor 必查**：审计时拿这个对账
> 3. **回归发现工具**：迁移后跑一遍 § C 验收清单，丢了什么立刻知道

---

## 标准格式

每份契约**必须包含三个章节**（顺序固定）：

```markdown
# <插件名> 功能契约（迁移基准）

> **建档日期**：YYYY-MM-DD
> **基线 commit**：<迁移开始时的 main 头部 SHA>
> **覆盖范围**：plugins/<plugin>/ 目录所有功能

---

## § A. 已验证的功能点

> 来源：git log feat(<plugin>) + memory feedback/fix 条目 + 用户补充
> 每条是一个"明确可指向某段代码 + 用户已确认 work 的特性"

- A1. <特性名>（来源：commit XXXX 或 memory: <name>）
  - **关键代码标识**：grep 关键词 / 文件路径 / 函数名
  - **验证场景**：用户做什么操作能看到这个特性
- A2. ...

---

## § B. 已知陷阱清单

> 来源：memory 中所有 feedback_* / fix_* 条目，迁移时必须绕过的雷
> 每条是一段"曾经踩过坑、必须保留的防御代码或写法"

- B1. <陷阱名>（memory: <name>）
  - **防御代码标识**：grep 关键词（如 `setSize.*true` / `Number.isFinite` / `closeRightSlot`）
  - **失守后果**：如果这段代码丢失会发生什么（用户视角）
- B2. ...

---

## § C. 验收清单

> 迁移完成时**必须手测过**的场景，每条一句话 + 一个勾选框
> 这些场景应覆盖 § A 的所有特性 + § B 中"用户能感知"的陷阱

- [ ] C1. <场景描述：在 X 状态下做 Y 操作，期望看到 Z>
- [ ] C2. ...

---

## § D. 不在本契约覆盖范围（声明）

> 列出已知的**当前已坏 / 已被废弃 / 不在本次迁移目标**的功能点
> 避免 Auditor 把"先于迁移就坏的 bug"误判为迁移引入

- D1. ...
```

---

## 起草流程

1. **Builder 起草**：动手前读完该插件相关的所有 memory + git log + 现有代码，列出 § A / § B / § C 的初稿
2. **用户 review**：补充隐式契约（用户记得但 memory 没写的细节）
3. **锁定基线 commit**：契约头部记录"基线 commit SHA"——这是审计的对账基准
4. **写入仓库**：合并到 main，保护起来
5. **迁移期不修改**：除非发现遗漏（独立 PR 增补），否则契约在迁移期间是不可变的

## 与 AUDITOR-PROMPT 的关系

[AUDITOR-PROMPT.md](../AUDITOR-PROMPT.md) § 三 B 项要求 Auditor 对契约 § A 和 § B 逐条 grep 对账。**契约越具体，审计越严格**。如果契约只写"画板能用"这种粗粒度描述，审计会失效。

每条契约项必须有"可 grep 的标识"——具体函数名、文件路径、关键参数。审计员靠这些标识在 diff 里找代码痕迹。
