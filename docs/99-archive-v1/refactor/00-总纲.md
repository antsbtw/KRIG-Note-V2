## 0. 为什么要做这次重构

通过 2026-05-02 的全栈评估（L0~L5 + DataModel 共 8 份）发现：

- **L0 越权下发布局特权 API**（`openCompanion` / `closeRightSlot`）给 L5 插件
- **四个插件（Note / EBook / Web / Graph）全部利用此漏洞微操右栏**
- **L3 `WorkspaceState` 塞满业务字段**（`activeNoteId` / `activeBookId` 等）
- **五大交互组件**（ContextMenu / Toolbar / Slash / Handle / FloatingToolbar）**各搞一套**，无注册制、样式不一致
- **能力（编辑、AI 对话、画板交互等）散落在各视图重复实现**，复用靠 copy-paste

继续叠功能将使违规面指数级扩大。**必须先把"宪法"立起来。**

## 1. 两条不可妥协的原则

### 1.1 分层原则
1. **上层不插手下层的业务**：不硬编码具体插件名，不预设业务字段
2. **下层不能干预上层的业务**：L5 不直接调 L4/L3 的特权 API，只能上抛 Intent

### 1.2 注册原则
所有"可扩展点"必须走注册制。包括但不限于：
- 插件加载（替代 `app.ts` 硬编码 import）
- WorkMode（已实现 ✅）
- NavSide 面板（已实现 ✅）
- **五大 UI 交互组件**（待实现 — 本次重构核心）
- **能力（Capability）**（本次重构新增的核心抽象）

### 1.3 抽象原则（本次重构新增）

> **真正的复用单位是"能力"。视图（View）只做声明，所有实现走 Capability。**

视图（View）是**用户感知的可视化层柱子**（对应三层架构顶层），多个视图通过**显式安装能力（Capability）**复用横向通用功能。

**规则 A（核心）**：所有有状态/有生命周期/有 UI 的外部依赖（npm 包）必须经 Capability 封装。**视图与插件层禁止直接 import 此类外部依赖**。
> 例：ProseMirror、Three.js、ELK、pdf.js、epubjs、WebContentsView、AI SDK 等——一律封在 `src/capabilities/<x>/` 内部，视图通过 `install: ['capability.<x>']` 使用。

**规则 B（例外）**：纯函数工具（无状态/无生命周期/无 UI/调用即返回）允许任何文件直接 import。白名单维护在 `tools/lint/pure-utility-allowlist.ts`，新增需独立 PR 评审。
> 初版白名单：`dayjs / date-fns / lodash / lodash-es / clsx / classnames / nanoid / uuid / zod / react / zustand / jotai / @types/*`
> **注意**：`electron` 的 renderer API（如 `ipcRenderer`）**不在白名单**——必须经 `@shared/ipc-client` 包装层。

**规则 C（颗粒度）**：Capability 颗粒度按"未来可扩展"原则设计——即便当前只一个视图消费，只要符合"有状态封装"特征，也建立独立 Capability，为未来跨视图复用预留接口。
> 不存在"单视图独占可直接 import"的口子。零例外是宪法的健康状态。

**为什么这样做**：
- **换底层库零成本**：未来 ProseMirror → Lexical / Slate、Three.js → Babylon.js 等切换，只改对应 capability 内部，所有视图无感
- **eslint 简单可执行**：一条规则 + 一份白名单 = 零容忍违规，CI 自动拦截
- **目录极致清爽**：`plugins/` 下零 npm 直接 import，视图只剩纯声明

**过渡期处置**：现有代码已存在大量直接 import 外部库的违规（web 插件、ebook 插件等）。波次 1 上 eslint 时设为 **warn**（不阻塞 CI），避免一次性大爆炸；**波次 3** 该插件迁移时升为 **error**，强制清干净。

## 2. 推进节奏：六波分波，自上而下定契约 + 自下而上做替换

> **核心策略**：让"违规旧 API"和"合规新 API"共存一段时间，逐插件迁移完再删旧。每一步可回滚、可验证。

| 波次 | 工作 | 产出 | 风险 |
|------|------|------|------|
| **波次 1** | 契约定型（纯类型/文档/eslint） | `shared/intents.ts`、`shared/ui-primitives.ts`（含 Capability/ViewDefinition 类型）、CLAUDE.md 硬规则、eslint 拦截 | 0（不动运行代码） |
| **波次 2** | 建中间层（IntentDispatcher + 五大 Registry + CommandRegistry + 首批 Capability 模块骨架） | `main/workspace/intent-dispatcher.ts`、`renderer/ui-primitives/*`、`src/capabilities/*` | 低（旧 API 仍可用） |
| **波次 3** | L5 插件按违规严重度倒序迁移：把视图改写成 ViewDefinition + install Capability | Graph → EBook → Note → Web，每个插件一条分支 | 中（每插件单测） |
| **波次 4** | L0/L4 收口（删特权 API、plugin loader） | `app.ts` 瘦身、`shell.openRightSlot` 内部化 | 低（此时已无人调用） |
| **波次 5** | L3 WorkspaceState 瘦身（业务字段 → `pluginStates`） | `WorkspaceState` 重构、删除专用 IPC | 低（不考虑旧数据兼容，已备份） |
| **波次 6** | L2 Design Token（独立、可插队） | CSS variables 替代 inline styles | 0（视觉等价改动） |

### 2.1 波次 3 的迁移顺序

综合"违规严重度 + 功能复杂度 + 流程成熟度"三维度调整后的顺序：

| 子波次 | 任务 | 理由 |
|--------|------|------|
| **3.1** | **Graph Step A**（行为保持迁移） | 最熟悉，先打通"视图改写为 ViewDefinition + install"的流程 |
| **3.2** | **EBook**（完整 A+B） | 小样本，把"如何抽 Capability"流程跑通 |
| **3.3** | **Graph Step B**（Capability 抽离） | 流程成熟后回头啃 Three.js 大块 |
| **3.4** | **Note**（完整 A+B） | 最大头，等流程稳定再啃硬骨头 |
| **3.5** | **Web/AI**（完整 A+B） | 违规最多但模式已成熟 |

### 2.2 每个迁移分两阶段独立 PR：行为保持 + 结构优化

**避免"重构 + 重写"混做导致丢功能**，每个插件的迁移**强制拆为两个独立 PR**：

#### Step A：行为保持迁移（Behavior-preserving）
- **目标**：把视图改写成 `ViewDefinition` 形态、声明 `install: [...]`，**视图层不再直接 import 外部 npm 包**
- **代码内部行为零改动**：现有的 IME 处理、抖动修复、retina 适配、防御代码等**原封不动搬**
- **只允许**：移动文件、加包装层、改 import 路径、新建 ViewDefinition 文件
- **完成判据**：`docs/refactor/migration-contracts/<插件>.md` § C 验收清单 100% 手测通过
- **PR 命名**：`refactor/L5-<插件>-step-a`

#### Step B：结构优化迁移（Structural cleanup）
- **目标**：把现有插件目录里的外部依赖代码搬入 `src/capabilities/<x>/`
- **此阶段才允许**：边界重画、函数拆分、capability 内部重组
- **每搬一块跑一遍验收清单**
- **PR 命名**：`refactor/L5-<插件>-step-b`

**为什么必须分**：A 和 B 同 PR 做，一旦发现行为变化，git bisect 无法定位是文件移动还是逻辑改写引入。分开后回滚成本极低。

### 2.3 迁移期"看到 X 不动 X" 原则（叠加 memory `feedback_branch_module_boundary`）

> 重构 graph 插件时，看到 `plugins/note/` 里有任何看似不合理的代码，**忽略它**。看到 `plugins/graph/` 里非本次迁移目标的代码，**忽略它**。
>
> 任何"顺手清理"都是对功能契约的潜在威胁。**只有专门的清理 PR 才允许动这些**，并且要走完独立的契约登记流程。

### 2.4 功能契约登记（每个插件迁移前必做）

在动手任何插件迁移之前，先建一份 `docs/refactor/migration-contracts/<插件>.md`，登记三类内容：

- **§ A 已验证的功能点**：来自 git log 的 `feat(...)` 列表 + memory 中相关 feedback/fix 条目
- **§ B 已知陷阱清单**：来自 memory 的所有 `feedback_*` 条目，迁移时必须绕过的雷
- **§ C 验收清单**：迁移完成时**必须手测过**的场景（一句一勾选）

详见 [migration-contracts/README.md](migration-contracts/README.md)。

### 2.5 每次迁移走"独立审计"：双会话隔离机制

**不允许 Builder（执行者）自审**——同一个 LLM 既写又审等于让被告兼任陪审员。

每个迁移 PR 完成后，**由用户启动独立的 Claude 会话作为 Auditor**，按 [AUDITOR-PROMPT.md](AUDITOR-PROMPT.md) 执行审计。Auditor 不写代码（Plan Mode 启动）、不读 memory（避免被实现技巧误导）、只对照总纲 + 功能契约 + git diff。

审计输出结构化报告：通过 / 不通过 / 待 Builder 证明。Auditor 给客观信息，最终是否 merge 由用户拍板。

## 3. 防"对话失忆"的四道防线

LLM 协作天然有"会话遗忘"问题。**不靠记忆，靠机制**——把规则物化成代码会读到的文件。

### 防线 1：CLAUDE.md 写死红线（每次会话必读）
追加"重构期硬规则"段落，列出：
- L5 禁止 import 的特权 API 名单
- L3 禁止新增业务字段
- 五大交互必须走 Registry
- **Atom 禁止挂 view-meta 字段**

### 防线 2：ESLint `no-restricted-imports` 自动拦截（CI 卡死）
违规代码 commit 不过去。**这是最硬的一道**，不依赖人也不依赖 LLM 记忆。

### 防线 3：每条 feature 分支挂 `.refactor-card.md`
任务卡格式：契约引用、本分支只做什么、禁止顺手做什么、完成判据（grep + eslint + 手测）。会话失忆时只需说"看 .refactor-card.md 继续"。

### 防线 4：auto-memory 记重构期决议
持续将关键决议写入 `~/.claude/.../memory/`，跨会话生效。

### 4.4 跨插件引用禁令
`plugins/<X>/**` **禁止 import** `plugins/<Y>/**`。彻底杜绝跨插件耦合（对应 memory `project_navside_arch_debt`）。由 eslint 强制。

跨视图共享逻辑必须通过：
- 共享资源走 `src/capabilities/`（能力）
- 共享类型走 `src/shared/`（契约）
- 跨视图协作走语义层（Atom）+ Intent 事件

## 5. View-scoped Registry：以 Capability 为复用单位

### 5.1 核心命题

> **不是运行时千变万化，而是视图时声明不同。**
> **不是按视图重复实现，而是按能力横向复用。**

否决"VS Code/Obsidian 式 `when` 表达式动态菜单"路线（不可静态分析、难测试、风格不一致）。

采用 **View-scoped + Capability-based** 模型：
> 每个用户视图（对应三层架构顶层柱子）是一个独立的注册单元，启动时**声明它安装哪些能力**。能力（Capability）携带 schema、UI 注册项、命令实现、实例工厂等完整内容，被任意视图按需安装。

### 5.2 概念三元组

```
Plugin   ─┐
          ├── View ─── install ──→ Capability (跨插件共享)
View    ──┘             │
                         └── 视图独有 contributions（不与他人共享的部分）

可视化层（三层架构顶层）：
  ┌──────────┐  ┌──────────┐  ┌──────────────┐  ┌──────────────┐
  │ note.    │  │ note.    │  │ graph.canvas │  │ graph.family │
  │ editor   │  │ thought  │  │              │  │ -tree        │  ← 平等的视图柱子
  └────┬─────┘  └────┬─────┘  └──────┬───────┘  └──────┬───────┘
       │             │               │                  │
       └─ install ───┴── install ────┴─── install ─────┘
                          │
                          ↓
       ┌────────────────────────────────────────────┐
       │  Capabilities (跨视图复用的能力单元)        │
       │  text-editing / web-rendering / canvas-... │
       └────────────────────────────────────────────┘
```

- **Plugin**：note / ebook / web / graph（粗粒度，对应 plugins/* 目录）
- **View**：note.editor / note.thought / graph.canvas / graph.family-tree / web.chatgpt …（用户感知的视图柱子，**互相平级**）
- **Capability**：text-editing / web-rendering / ai-conversation …（跨视图横向复用）

**关键认知**：View 之间无父子继承。共享通过显式 install Capability 表达。

### 5.3 注册时机：分两段

```
启动期（plugin.register()）
  └─ 注册 ViewDefinition：viewId、install: [...]、视图独有 contributions
      （框架记录元数据，不立即挂载交互项）

激活期（view 实例化时）
  └─ 框架根据 ViewDefinition.install 列表，把所有 capability 的注册项 + 视图独有项
     合并挂载到该视图实例的 active scope
  └─ view 销毁时整批卸载
```

**关键**：菜单项不是每次右键时动态算出来的，而是 view 激活时**整批挂载**好的。右键时 ContextMenuRegistry 只查询当前 active scope，不执行用户函数。

### 5.4 数据契约（草图，正式以 `shared/ui-primitives.ts` 为准）

```ts
// 一个用户视图的定义
export interface ViewDefinition {
  viewId: string;                  // e.g. 'note.editor' / 'graph.family-tree'
  install?: CapabilityId[];        // 安装的能力列表
  // 视图独有项（不与其他视图共享）
  contextMenu?: ContextMenuItem[];
  toolbar?: ToolbarItem[];
  slash?: SlashItem[];
  handle?: HandleItem[];
  floatingToolbar?: FloatingToolbarItem[];
}

// 一项能力的定义
export interface Capability {
  id: string;                      // e.g. 'capability.text-editing'

  // 1. UI 注册项（同 ViewDefinition 五大类）
  contextMenu?: ContextMenuItem[];
  toolbar?: ToolbarItem[];
  slash?: SlashItem[];
  handle?: HandleItem[];
  floatingToolbar?: FloatingToolbarItem[];
  keybindings?: KeyBinding[];

  // 2. Schema 贡献（如 ProseMirror 的 block/mark 定义）
  schema?: SchemaContribution;

  // 3. 数据转换（atom ↔ 该能力的内部表征）
  converters?: { toAtom: (x: unknown) => Atom[]; fromAtom: (a: Atom[]) => unknown };

  // 4. 实例工厂（视图调用获得可挂载的实例）
  createInstance?: (host: HostElement, options: CapabilityOptions) => CapabilityInstance;

  // 5. 命令实现（被菜单项的 command: string 引用）
  commands?: Record<string, CommandHandler>;
}

// 菜单项（其他四种类似，字段略）
export interface ContextMenuItem {
  id: string;
  label: string;
  group?: string;
  order?: number;
  icon?: string;
  command: string;                 // 字符串引用 CommandRegistry，不是函数
  enabledWhen?: 'always' | 'has-selection' | 'is-editable';  // 有限枚举
}
```

**示例：视图侧的声明**

### 5.5 强约束（健壮性保证）

1. **ViewDefinition 必须是顶层 const 导出**，不能是函数返回的对象（保证静态可分析）
2. **command 字段必须是字符串**，函数实现走 `CommandRegistry.register(id, handler)`
3. **view id 必须命名空间化**：`<plugin>.<view>`（如 `note.editor` / `graph.family-tree`）
4. **capability id 必须命名空间化**：`capability.<name>`（如 `capability.text-editing`）
5. **Capability 之间不能互相 install**（禁套娃，避免依赖图）。需要组合时，View 自己 install 多个 capability
6. **Capability 颗粒度**：至少有两个互不相干的视图会使用，否则属于视图私有功能
7. **Atom 永远不挂 view-meta 字段**（见 § 6.1 视图状态持久化的归属）
8. **eslint 强制**：`plugins/<X>/**` 禁止 import `plugins/<Y>/**`
9. **CI 校验**：每个 ViewDefinition 的 command id 必须在 CommandRegistry 中存在；否则 CI 卡住

### 5.6 多视图共享方法的标准解法

**问题**：note.editor 和 note.thought 共享文本编辑能力；web.chatgpt 和 web.claude 共享 AI 对话能力。如何不重复声明？

**解法**：拆 Capability，View 通过 `install` 显式组合。

**Web/AI 共享示例（双重组合）**：
```ts
// 基础能力：所有 web 视图都用
'capability.web-rendering' — WebContentsView 渲染、链接跳转、新窗口打开

// 扩展能力：仅 AI 站点用
'capability.ai-conversation' — 切模型、提取最新回复、继续对话

// 视图自由组合
{ viewId: 'web.generic', install: ['web-rendering'] }
{ viewId: 'web.chatgpt', install: ['web-rendering', 'ai-conversation'] }
{ viewId: 'web.claude',  install: ['web-rendering', 'ai-conversation'] }
{ viewId: 'web.gemini',  install: ['web-rendering', 'ai-conversation'] }   // 新增 AI 站点零改动
```

加新 AI 站点（gemini）时，只新增一个 ViewDefinition 声明，不改任何已有代码。

### 5.7 五大交互的统一与差异

- **统一**：全部走 View-scoped 模型；都用 `command: string`；生命周期跟随 view
- **差异**：触发逻辑各自独立（右键 / hover / `/` 输入 / selection-change）；渲染实现各自独立

> 先各自独立 Registry，**不强行合并为超级 Registry**。等 3~4 个落地后再看是否抽公共基类。

### 5.8 视图是声明，实现都在 Capability 里

**KRIG 弃用 "engine"（引擎）作为架构概念**。任何"看起来像引擎"的东西——ProseMirror、Three.js、ELK、pdf.js、WebContentsView——本质都是**外部依赖**。它们的归宿只有一个：被 Capability 封装。

用户视图（ViewDefinition）只负责**声明**——声明自己安装哪些能力、自己有哪些独有交互项。所有"实现"——无论是富文本编辑、画板渲染、自动布局、PDF 解析、网页渲染——**全部以 Capability 形式存在于 `src/capabilities/`**，被视图通过 `install` 显式组合。

**没有"插件私有引擎"**：不存在 `plugins/<X>/engine/`、`plugins/<X>/runtime/` 这类绕开 capability 的目录（由 § 4.1 目录约束 + eslint 强制）。

**外部依赖到 Capability 的对应关系**：

| 外部依赖（npm 包） | 归属 Capability | 主要消费视图 |
|------|------|------|
| `prosemirror-*` | `capability.text-editing` 内部 | note.editor / note.thought / graph.* 节点 label |
| `three` | `capability.canvas-interaction` 内部 | graph.canvas / graph.family-tree |
| `elkjs` | `capability.elk-layout` 内部 | graph.* |
| `pdfjs-dist` | `capability.pdf-rendering` 内部 | ebook.pdf |
| `epubjs` | `capability.epub-rendering` 内部 | ebook.epub |
| `electron WebContentsView` | `capability.web-rendering` 内部（main 进程封装） | web.* |
| `@anthropic-ai/sdk` 等 AI SDK | `capability.ai-conversation` 内部 | web.chatgpt / web.claude |
| Atom schema、Library（Shape/Substance） | 语义层契约 / 共享资源 | 所有视图共享 |

**为什么这样做**：未来切换底层库（ProseMirror → Lexical / Slate、Three.js → Babylon.js、pdf.js → 其他 PDF 库）时，**只改对应 capability 内部，所有 install 它的视图一行不改**。这是"换底层零成本"承诺的物理保证。一旦视图直接 import 外部依赖，承诺即破。所以零例外。

**关于 graph.canvas 的现状**：当前 `src/plugins/graph/canvas/` 既是 graph.canvas 视图又混了 Three.js 渲染逻辑——这是工程现状，不是目标态。波次 3 迁移 graph 时按目标态拆分：视图声明留在 `plugins/graph/views/canvas/`，所有 Three.js 逻辑搬入 `src/capabilities/canvas-interaction/`。

**关于 Note 视图的现状**：波次 3 迁移 Note 时**只迁移交互注册 + 视图声明 install 列表**，不强制立即把 NoteEditor 内部代码完全搬入 `capability.text-editing/`——这是大手术。波次 3 只要求视图层做到"通过 install 声明能力"+"不直接 import prosemirror-*"，capability 内部代码逐步整理可以由独立分支推进。

## 6. 配套数据模型四层（与 UI 分层正交）

来自 [DataModel-Architecture-Design](../evaluation/2026-05-02-DataModel-Architecture-Design.md)：

| 层 | 受众 | 内容 |
|---|---|---|
| **语义层 Semantic** | 知识图谱、AI、SurrealDB | Entity、Triple、FromReference、Intent |
| **表征层 Representation** | 数据库、同步、导入导出 | KRIG-Atom、Topology、Resource、Metadata |
| **渲染层 Visualization** | React、ProseMirror、ECharts、Three.js | Framework State、Transient UI State、Layout Tree |
| **互操作层 Interoperability** | IPC、API、Converter | Converter 契约、Message Protocol、Sync Protocol |

强约束：
- 数据库 Schema 只能用 Semantic + Representation
- L5 React 组件只能用 Visualization；保存时必须经 Converter 转 Representation
- IPC 只允许传 Representation 或 Semantic 数据；**禁止传 DOM / 函数**

### 6.1 视图状态持久化的归属（重要新规）

> **Atom 永远不携带任何视图特定字段**（不加 `meta.view`、不加 `meta.canvas`、不加任何 view-meta）。
>
> 这条原则来自三层架构 § 2.3 "Atom 独立性原则"。违反则破坏"开放扩展、封闭修改"承诺，新视图加入需修改 Atom schema。

视图需要持久化的状态分两类，去处不同：

| 类型 | 例子 | 持久化去处 |
|------|------|-----------|
| **视图独有属性** | Graph 节点位置/颜色、family-tree 辈分、timeline 日期 | **视图自己的索引表**（如 `graph_node.position`、`family_tree_node.elderhood`、`timeline_node.date`） |
| **视图临时状态** | 光标位置、滚动位置、缩放级别、选中节点 | **Workspace 的 `pluginStates` 字典**（波次 5 已规划） |
| **视图内容** | 节点 label 的文字、笔记正文 | **Atom[]**（语义层），但视图持有自己的副本（v1.2 阶段；远期愿景见三层架构 § 2.4 投影模型） |

**反例**（违反）：
```ts
// ✗ Atom 知道哪些视图存在
Atom: { type, content, meta: { canvas: { x, y }, timeline: { date } } }
```

**正例**：
```ts
// ✓ Atom 干净
Atom: { type, content }

// ✓ 视图特性挂在视图自己的索引表上
graph_node: { atomRef, position, color, ... }
timeline_node: { atomRef, date, ... }
```

**与 memory 中两条历史记录的关系**：
- `feedback_thought_is_noteview_variant` —— "Thought = NoteEditor 薄包装" 是**实现细节**，架构定位上 thought 和 editor 是**平级 View**（本总纲 § 5.2）。当前实现是工程经济选择，未来不必绑定。
- `project_canvas_as_note_view_variant` —— "视图特性挂 atom.meta.canvas" 已**过时**，按本节修订为"挂在视图自己的索引表"。

## 7. 角色与提示词体系（三角架构）

> **核心命题**：单个 LLM 既写代码又审自己等于"被告兼任陪审员"。重构能成必须靠**角色独立 + 信息物化**。

### 7.1 三个独立角色

```
┌─────────────────────────────┐
│  用户（最终裁决者）          │  保留所有破坏性操作授权 + 重大方向决定
└──────────────┬──────────────┘
               │ ↑
       拍板    │ │ 升级
               ↓ │
┌─────────────────────────────┐
│  Commander（总指挥）         │  调度、综合判断、起草配套设施、向用户汇报
│  = Claude（持续会话）        │  禁止：直接写迁移代码 / 直接审计 / 擅自破坏性 git
└──────────────┬──────────────┘
               │
        派发任务卡 + 启动指令
               │
       ┌───────┴───────┐
       ↓               ↓
┌────────────┐  ┌────────────┐
│  Builder   │  │  Auditor   │
│ = Claude   │  │ = Claude   │
│  独立会话  │  │ Plan Mode  │
│            │  │  独立会话  │
│ 严格按 card│  │ 对照规则+  │
│ 执行迁移   │  │ 契约对账   │
│ 写自检表   │  │ 输出报告   │
│            │  │            │
│ 禁止读     │  │ 禁止读     │
│ memory     │  │ memory     │
│ 禁止越界   │  │ 禁止写代码 │
└────────────┘  └────────────┘
```

### 7.2 角色互锁矩阵

| 维度 | 用户 | Commander | Builder | Auditor |
|------|------|-----------|---------|---------|
| 写迁移代码 | — | ❌ | ✅ | ❌ |
| 写配套文档（card / 契约 / 启动指令） | ❌ | ✅ | ❌ | ❌ |
| 审计代码 | — | ❌ | ❌ | ✅ |
| 综合两份报告做调度判断 | ✅ | ✅ | ❌ | ❌ |
| 拍板 merge / 回滚 | ✅ | ❌ | ❌ | ❌ |
| 读 memory | ✅ | ✅ | ❌ | ❌ |
| 做破坏性 git 操作 | ✅ | ❌ | ❌（仅 commit） | ❌ |
| 决定"通过/不通过" | ✅ | ❌ | ❌（仅陈述事实） | ✅ |

### 7.3 三份提示词

| 角色 | 提示词文件 | 何时启用 |
|------|----------|---------|
| Commander | [docs/refactor/COMMANDER-PROMPT.md](COMMANDER-PROMPT.md) | 每次新会话开始时 Commander（Claude）自读重新校准 |
| Builder | [docs/refactor/BUILDER-PROMPT.md](BUILDER-PROMPT.md) | 用户启动新迁移子波次时，新开 Claude 会话粘贴 |
| Auditor | [docs/refactor/AUDITOR-PROMPT.md](AUDITOR-PROMPT.md) | 用户启动审计时，新开 Plan Mode Claude 会话粘贴 |

### 7.4 信息载体（避免聊天框传递信息丢失）

三角之间通过**文件**而非聊天传递信息：

| 信息流 | 载体 | 写入方 | 读取方 |
|--------|------|-------|-------|
| Commander → Builder 派活 | `docs/refactor/cards/refactor-<分支名>.md`（永久保留） | Commander | Builder |
| Builder 完成报告 | `tmp/builder-report.md`（每轮覆盖） | Builder | Commander |
| Commander → Auditor 派审 | 聊天启动指令（含分支名 + 契约引用 + 关注点） | Commander | Auditor |
| Auditor 审计报告 | `tmp/auditor-report.md`（每轮覆盖） | Auditor | Commander |
| 用户决策 | 聊天 | 用户 | Commander 转达 |

> **`tmp/` 不入 git**（视为会话工作区）。`docs/refactor/cards/` 入 git（作为重构历史档案）。

### 7.5 一个迁移子波次的标准节拍

```
[1] Commander 起草 refactor-card → 用户 GO
[2] 用户新开 Builder 会话 → Builder 输出"启动确认"含歧义清单 → Commander 澄清
[3] Builder 执行 → commit → 写 tmp/builder-report.md
[4] Commander 读 builder-report → 综合后向用户汇报"Builder 阶段总结"
[5] Commander 起草 Auditor 启动指令 → 用户新开 Plan Mode 会话
[6] Auditor 审计 → 写 tmp/auditor-report.md
[7] Commander 读 auditor-report → 综合两份报告 → 向用户输出"决策建议"（merge / 修 / 回滚）
[8] 用户拍板 → Commander 列 git 命令 → 用户执行
[9] 进入下一子波次的 [1]
```

完整节拍详见 [COMMANDER-PROMPT.md](COMMANDER-PROMPT.md) § 三。

### 7.6 防"乐观汇报"硬规则

LLM 天然倾向"包装好消息"，必须用结构反向：

1. Commander 每次汇报先列 ⚠️/❌ 项，再列 ✅ 项
2. Builder 自检表的"F. 我没做但 card 要求的事"段必须诚实填写，不能默写为完成
3. Auditor 疑议从严：任何无法 100% 确认的标 ⚠️"待 Builder 证明"，不放行
4. 三角之间出现冲突 → Commander 必须摆给用户看，不"应该没事"消化

### 7.7 角色失效的判定

任意一条违反 = 该角色失效，需要重启会话：

- Commander 直接 Edit/Write 业务代码 → 失效
- Commander 替用户做破坏性 git 操作 → 失效
- Builder 范围越界（改了 card 之外的代码）→ 失效，回滚提交重来
- Builder 自审通过（替代 Auditor）→ 失效
- Auditor 写代码或修改文件 → 失效
- 任何角色读了 memory（除 Commander）→ 失效

## 8. 待最终拍板的几个细节（不影响波次 1 启动）

1. **CommandRegistry 是否波次 2 一并建？**
   倾向：是。否则菜单 `command: string` 是空壳。已写入波次 2 产出。

2. **View 激活粒度**
   倾向：按 view instance 激活（一个画布一份菜单）。理由：一个工作区可能同时有多个不同 view 的实例。

3. **是否支持 workspace 级 global contribution**
   倾向：是。需在 ViewDefinition 之外加一类 `GlobalContribution { scope: 'global' }`。NavSide 右键、"在新窗口打开"等天然属于 global。

4. **Capability 列表的合并/拆分**
   见 § 5.9 末段，落地时按需决定。

5. **family-tree 当前无代码**，目录中标 "(计划中)"，作为目标态展示。
